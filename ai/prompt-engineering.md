# プロンプトエンジニアリング

## 何か

LLMから意図した出力を引き出すための入力設計技術。
「うまく書けば良い答えが出る」という経験則から、再現性のある技法として体系化されている。

## System Prompt vs User Prompt

| | 役割 | 例 |
|---|---|---|
| System Prompt | AIの役割・制約・ルールを定義 | 「あなたはTypeScript専門家です。〜」 |
| User Prompt | 具体的なタスク指示 | 「この関数をリファクタリングして」 |

System Promptは毎ターン読まれる。**AIの「性格」と「制約」はここに書く。**

```
System: あなたはセキュリティレビューの専門家です。
コードを見たとき、OWASP Top 10 の観点で問題を指摘してください。
問題がなければ「問題なし」とだけ答えてください。余計な説明は不要です。

User: [コードを貼る]
```

## 主要な技法

### Zero-shot

例を与えず、直接タスクを指示する。シンプルなタスクに有効。

```
「以下のJSONをTypeScriptの型定義に変換してください」
```

### Few-shot

例を数件見せてから本番タスクを渡す。出力フォーマットや判断基準を揃えるのに有効。

```
以下の形式でコミットメッセージを作成してください：

例1:
変更: ユーザー登録時のメール検証を追加
メッセージ: メール形式のバリデーションをUserエンティティに追加

例2:
変更: 注文一覧のN+1クエリを修正
メッセージ: 注文一覧のクエリをJOINに変更してN+1を解消

本番:
変更: パスワードリセット機能を実装
メッセージ: 
```

### Chain-of-Thought（CoT）

「ステップバイステップで考えてください」と指示することで、複雑な推論の精度が上がる。

```
// 単純な指示
「このアルゴリズムのバグを直してください」

// CoT版
「このアルゴリズムのバグを直してください。
まず処理の流れを追い、どこで期待と違う動作が起きているか説明してから、修正してください。」
```

### Step-back Prompting

具体的な問題の前に、一般的な原則を考えさせる。

```
「この認証コードをレビューする前に、安全なセッション管理の原則を列挙してください。
そのうえで、以下のコードを確認してください。」
```

### Role Prompting

AIに特定の役割を与える。その分野の専門的な観点が出やすくなる。

```
「あなたは10年のキャリアを持つセキュリティエンジニアです。
悪意ある第三者の視点で、このAPIエンドポイントを攻撃できる箇所を探してください。」
```

## 出力フォーマットの制御

### JSON で受け取る

```
以下の情報をJSONで返してください：
{
  "severity": "high" | "medium" | "low",
  "issues": [{ "line": number, "description": string }],
  "suggestion": string
}

[コード]
```

### 構造を指定する

```
以下の構造で回答してください：
## 問題点
（箇条書き）

## 修正方法
（コードブロック付き）

## リスク
（修正しない場合の影響）
```

## 長いコンテキストへの対応

LLMは長い入力の**中間部分を忘れやすい**（Lost in the Middle問題）。

**対策**: 最も重要な情報は先頭か末尾に置く。

```
// Bad: 重要な制約が中間にある
[大量のコード]
[重要: このプロジェクトではthrowを使わずResult型を使う]
[さらに大量のコード]
「リファクタリングしてください」

// Good: 制約を最後に再掲する
[コード]
「リファクタリングしてください。
必須: throwを使わず Result<T, E> 型を使うこと。」
```

## プロンプトの品質を上げるチェックリスト

- [ ] タスクが1つに絞られているか（複数タスクは分割する）
- [ ] 期待する出力フォーマットを明示しているか
- [ ] 「やること」だけでなく「やってはいけないこと」も書いているか
- [ ] 曖昧な言葉（「良い感じに」「適切に」）を使っていないか
- [ ] 例を1つ以上添えているか（Few-shot）

## よくある失敗パターン

| 失敗 | 原因 | 対策 |
|---|---|---|
| 「わかりました」で終わる | タスクが不明確 | 「〜してください」と動詞で終わらせる |
| 余計な説明が長い | 出力フォーマット未指定 | 「〜だけ答えてください」と制約する |
| 毎回違う形式で返ってくる | フォーマット指定なし | JSON or Markdownテンプレートを指定 |
| 途中で別の話題に逸れる | System Promptが弱い | 「この話題だけ扱う」と明示する |
