# バックエンド言語とWebフレームワーク

バックエンド開発で使われる主要言語と、それぞれに対応するWebフレームワークの概観。バージョン固有の機能よりも「なぜその技術が選ばれるか」「どんな文脈で使われるか」を整理する。

## Ruby / Rails

**Ruby** は「開発者の幸福」を設計思想に掲げた言語。簡潔に書けることを重視する。

**Ruby on Rails** は "Convention over Configuration（設定より規約）" を掲げるフルスタックフレームワーク。ルーティング・ORM・テンプレートが最初から揃っており、素早く Web アプリを立ち上げられる。GitHub・Shopify・Airbnb など多くのスタートアップが初期に採用した実績がある。

```
向いている用途: MVP・スタートアップの初期プロダクト・社内ツール
採用される背景: 少人数でも素早く動くものを作りたい
```

## PHP / Laravel・CakePHP

**PHP** は Web 開発のために生まれた言語。WordPress が世界中の Web サイトで使われていることもあり、Web ホスティングの大半でそのまま動く普及率の高さが特徴。

**Laravel** は Rails に影響を受けたモダンな PHP フレームワーク。認証・キュー・メール送信などが標準装備。現在の PHP エコシステムの中心。

**CakePHP** は日本で比較的普及している PHP フレームワーク。Rails と同様の規約ベースの設計。日本の受託開発・自社サービスで多く使われてきた。

```
向いている用途: Web サイト・EC サイト・企業向け Web アプリ
採用される背景: PHP エンジニアの採用しやすさ・既存資産の活用
```

## Python / Django・FastAPI

**Python** は読みやすさと汎用性が特徴。Web 開発のほか、データ分析・機械学習・スクリプトと幅広く使われる。AI/ML エンジニアが多い組織では Python が共通言語になりやすい。

**Django** は "Batteries included（電池同梱）" を掲げるフルスタックフレームワーク。管理画面・ORM・認証が標準装備。大規模なコンテンツサイトや社内ツールに向く。

**FastAPI** はモダンで高速な API 向けフレームワーク。型アノテーションから自動的に API ドキュメントが生成される。マイクロサービスや AI 機能の API エンドポイントとして近年急速に普及。

```
向いている用途: API サーバー・データパイプライン・AI 連携サービス
採用される背景: ML チームとバックエンドチームが同じ言語を使いたい
```

## Java / Spring Boot

**Java** はエンタープライズ開発の主流言語。型安全・パフォーマンス・豊富なライブラリが強み。銀行・保険・製造業などの大企業システムで長く使われてきた。

**Spring Boot** は Java エコシステムの中心的なフレームワーク。設定の自動化・依存性注入・豊富なモジュールが揃い、大規模なシステムを構造的に開発できる。マイクロサービス構成との親和性も高い。

```
向いている用途: 金融・エンタープライズ・大規模サービス
採用される背景: 既存の Java 資産・エンジニアの採用・信頼性の実績
```

## Kotlin / Spring・Ktor

**Kotlin** は JVM 上で動く Java の後継言語。Java より簡潔に書けながら Java との互換性を保つ。Android 開発の公式言語でもある。

バックエンドでは Spring Boot を Kotlin で書くスタイルが増えている。**Ktor** は JetBrains が開発した Kotlin ネイティブの軽量フレームワーク。

```
向いている用途: Java からの移行・Android + バックエンドを同言語で統一したい
採用される背景: Java 組織が言語をモダナイズしたい
```

## Go / Gin・Echo・Chi

**Go**（Golang） はシンプルさ・高いパフォーマンス・並行処理の扱いやすさが特徴。Google が開発した。コンパイルが速く、シングルバイナリで配布できる。インフラ系ツール（Docker・Kubernetes・Terraform）も Go で書かれている。

**Gin・Echo・Chi** はいずれも軽量なフレームワーク。Go の標準ライブラリが充実しているため、フレームワーク自体は薄い設計が好まれる。

```
向いている用途: マイクロサービス・高トラフィック API・インフラツール
採用される背景: パフォーマンスとシンプルさの両立・インフラ文脈との親和性
```

## TypeScript（Node.js）/ Express・NestJS・Hono

**Node.js** は JavaScript をサーバーサイドで動かすランタイム。フロントエンドと同じ言語でバックエンドも書けるため、フルスタック開発や小規模チームと相性が良い。

**Express** は最もシンプルな Node.js フレームワーク。薄い設計のため自由度が高いが、大規模になると構造化が必要になる。

**NestJS** は Angular に影響を受けた構造化フレームワーク。デコレーターとモジュール設計により、大規模な TypeScript バックエンドを整理して書ける。

**Hono** は軽量・高速を重視した新世代フレームワーク。Cloudflare Workers など Edge 環境にも対応。

```
向いている用途: BFF・API サーバー・リアルタイム通信
採用される背景: フロントエンドチームがバックエンドも担う・Edge デプロイ
```

## C# / ASP.NET Core

**C#** は Microsoft が開発した言語。**.NET** エコシステムと組み合わせて Windows 環境・エンタープライズシステムで広く使われる。ゲームエンジン Unity のスクリプト言語でもある。

**ASP.NET Core** はクロスプラットフォーム対応の Web フレームワーク。パフォーマンスが高く、Microsoft Azure との統合が強い。

```
向いている用途: Microsoft スタックの企業・ゲームバックエンド
採用される背景: 既存の Windows/.NET 資産・Azure 環境
```

## Rust / Axum・Actix

**Rust** はメモリ安全性とパフォーマンスを言語レベルで保証する。学習コストは高いが、実行時のバグが少なく、C/C++ 並みの速度が出る。

Web 開発での採用はまだ少ないが、**Axum**（Tokio チーム製）・**Actix** など高性能フレームワークが整備されている。パフォーマンスが最重要なシステムや、WebAssembly が絡む文脈で選ばれることが増えている。

```
向いている用途: 高性能 API・システムプログラミング・WebAssembly
採用される背景: パフォーマンスとメモリ安全性を同時に求める
```

## 選択の判断軸

特定の言語・フレームワークが「良い・悪い」ではなく、状況によって最適解が変わる。

| 判断軸 | 選択への影響 |
|---|---|
| チームの既存スキル | 最も重要。知らない言語の採用は大きなリスク |
| 採用市場 | エンジニアを採用しやすい言語か |
| 既存システムとの統合 | 言語・プロトコルが揃っているか |
| スケール要件 | トラフィック・チーム規模に対応できるか |
| 開発速度 vs 性能 | Rails/Django は速く作れる、Go/Rust は高性能 |
