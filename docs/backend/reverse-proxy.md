# リバースプロキシ

## なぜ存在するか

クライアントがバックエンドサーバーに直接アクセスする構成では、サーバーの内部構造が外部に露出し、スケール・セキュリティ・運用の柔軟性が下がる。
リバースプロキシはクライアントとバックエンドの間に置く中継層で、**バックエンドを隠蔽しつつ横断的な処理を一箇所に集約する**。

```
クライアント
    ↓
┌─────────────────┐
│  リバースプロキシ  │  ← SSL終端・ルーティング・キャッシュ・ロードバランシング
└─────────────────┘
    ↓       ↓       ↓
サーバーA サーバーB サーバーC
```

クライアントからはリバースプロキシだけが見え、バックエンドの台数・場所・構成は隠れる。

## 主な役割

### ロードバランシング

複数のバックエンドサーバーにリクエストを分散する。

```
ラウンドロビン: A → B → C → A → B → C ...
最小コネクション: 接続数が最も少ないサーバーへ
IPハッシュ: 同一クライアントを同一サーバーへ（セッション維持）
```

### SSL終端

HTTPS の暗号化・復号をリバースプロキシで処理し、バックエンドとの通信は HTTP にする。
証明書管理をバックエンドから切り離せる。

```
クライアント ──HTTPS──→ リバースプロキシ ──HTTP──→ バックエンド
```

### キャッシュ

同一レスポンスをキャッシュして、バックエンドへのリクエストを削減する。

### リクエスト・レスポンスの変換

ヘッダーの付け替え・URLの書き換え・圧縮（gzip）などをバックエンドの手前で処理する。

### 保護

バックエンドのIPを隠す・レートリミット・DDoS対策などのセキュリティ処理を集中させる。

## フォワードプロキシとの違い

| | フォワードプロキシ | リバースプロキシ |
|---|---|---|
| 代理する対象 | クライアント側 | サーバー側 |
| 設置位置 | クライアントの前 | サーバーの前 |
| クライアントからの見え方 | クライアントが意識して使う | 透過的（サーバーと区別がつかない） |
| 用途 | アクセス制御・匿名化・フィルタリング | ロードバランシング・SSL終端・キャッシュ |

## API Gateway との違い

リバースプロキシとAPI Gatewayは重なる部分が多いが、役割の焦点が異なる。

| | リバースプロキシ | API Gateway |
|---|---|---|
| 焦点 | ネットワーク層のトラフィック制御 | APIレベルの管理（認証・レートリミット・ルーティング） |
| 粒度 | 汎用（HTTP全般） | API（REST・gRPC など） |
| 代表 | Nginx・HAProxy・Caddy | Kong・AWS API Gateway |

実際には API Gateway がリバースプロキシの機能を内包していることも多く、両者を明確に分けず同一ソフトウェアで担うケースもある。

## いつ使うか

- 複数のバックエンドサーバーにトラフィックを分散したい（ロードバランシング）
- SSL証明書の管理をバックエンドから切り離したい
- バックエンドの内部構成を外部に隠したい
- 静的ファイルのキャッシュや圧縮をアプリ外で処理したい
