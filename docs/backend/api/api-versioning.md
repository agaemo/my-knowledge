# APIバージョニング

APIに変更を加えるとき、既存のクライアントを壊さずに新旧バージョンを共存させるための設計方針。

## なぜ存在するか

APIは一度公開すると、多くのクライアントが依存する。破壊的変更（フィールドの削除・型の変更・動作の変更）を加えるとき、全クライアントを一斉に移行させることは現実的でない。バージョニングはAPIの新旧バージョンを並行して運用し、クライアントの段階的な移行を可能にする。

## 主なバージョニング方法

### URLパスバージョニング

```
GET /v1/users
GET /v2/users
```

最も広く使われる方法。URLを見ればバージョンが明確で、ブラウザのアドレスバーやログからも確認できる。

**メリット:** シンプルでわかりやすい。キャッシュしやすい。  
**デメリット:** URLの設計としてはRESTの原則（リソースのURIは不変）に厳密には反する。

### クエリパラメータ

```
GET /users?version=1
GET /users?version=2
```

**メリット:** ベースURLが変わらない。  
**デメリット:** キャッシュ制御が複雑になる。バージョンが見落とされやすい。

### リクエストヘッダー

```
GET /users
Accept: application/vnd.myapi.v1+json

GET /users
Accept: application/vnd.myapi.v2+json
```

**メリット:** URLがクリーンに保たれる。RESTの原則に沿っている。  
**デメリット:** クライアントの実装が複雑になる。ブラウザでの確認が不便。

### カスタムヘッダー

```
GET /users
X-API-Version: 2
```

**メリット:** URLがシンプル。  
**デメリット:** 標準ではないため、ドキュメント化が必須。

## 比較

| 方法 | 明示性 | キャッシュ | RESTの原則 | 採用例 |
|------|--------|-----------|-----------|--------|
| URLパス | ◎ | ◎ | △ | Stripe・GitHub・Twitter |
| クエリパラメータ | ○ | △ | ○ | Amazon |
| Acceptヘッダー | △ | ○ | ◎ | GitHub（一部） |
| カスタムヘッダー | △ | ○ | ○ | 独自実装 |

実用上は**URLパスバージョニング**が最もわかりやすく、広く採用されている。

## バージョンの廃止（Deprecation）

```
HTTP/1.1 200 OK
Deprecation: true
Sunset: Sat, 01 Jan 2026 00:00:00 GMT
Link: <https://api.example.com/v2/users>; rel="successor-version"
```

古いバージョンを廃止するときは、十分な移行期間とともにレスポンスヘッダーで通知する。

## 破壊的変更の扱い

バージョンを上げるべき変更（破壊的変更）：

- フィールドの削除・リネーム
- 必須フィールドの追加
- レスポンスの構造変更
- 認証方式の変更

バージョンを上げなくてよい変更（後方互換）：

- 新しいオプションフィールドの追加
- 新しいエンドポイントの追加
- バグ修正（既存の動作が仕様通りになる場合）

## いつ使うか

- 外部に公開するAPI（サードパーティが依存する）
- モバイルアプリのバックエンドAPI（アプリのアップデートを強制できない）
- 社内APIでもチームをまたぐ場合
- 小規模・内部のみのAPIではオーバーエンジニアリングになる場合もある
