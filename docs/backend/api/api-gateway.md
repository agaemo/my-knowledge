# API Gateway

## なぜ存在するか

複数のバックエンドサービスがある構成では、各サービスが個別にレートリミット・認証・ロギングを実装すると重複と不整合が生じる。
API Gateway はこれらの**横断的関心事をエッジで一元管理**するアーキテクチャパターン。

```
クライアント
    │
    ▼
┌──────────────────────────────┐
│         API Gateway          │
│  認証・レートリミット・ルーティング │
└──────────────────────────────┘
    │           │           │
    ▼           ▼           ▼
ユーザーAPI  注文API    商品API
```

## 主な責務

### レートリミット・スロットリング

アクセス数を制御して、特定クライアントによるサービス占有を防ぐ。

```
# ウィンドウ方式の違い
固定ウィンドウ: 1分間に100リクエストまで（00:00〜01:00、01:00〜02:00）
スライディングウィンドウ: 直近60秒で100リクエストまで（常に動いている窓）
トークンバケット: バケットにトークンが補充され、リクエストごとに消費する
```

```
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1700000060
```

レートリミット超過時は `429 Too Many Requests` を返し、`Retry-After` で再試行タイミングをクライアントに伝える。

### ルーティング・リクエスト変換

URLパスやヘッダーに応じてバックエンドサービスへ転送する。

```
/api/v1/users/**  → ユーザーサービス（:3001）
/api/v1/orders/** → 注文サービス（:3002）
/api/v2/users/**  → ユーザーサービスv2（:3003）
```

### 認証・認可

各サービスで個別実装せず、Gatewayでトークン検証を行う。

```
クライアント → Gateway（JWTを検証）→ バックエンド（検証済みヘッダーを受け取る）
```

### その他の横断的関心事

- **ロギング・トレーシング**: 全リクエストの入出力を記録
- **SSL終端**: Gateway で HTTPS を処理し、内部通信は HTTP にする
- **キャッシュ**: 同一レスポンスのキャッシュでバックエンドの負荷を下げる
- **サーキットブレーカー**: バックエンド障害時に即座にエラーを返し連鎖障害を防ぐ

## いつ使うか

**適している状況**:
- マイクロサービス構成で、複数のサービスを外部に公開する
- レートリミットや認証を統一したい
- クライアントがバックエンドの内部構成を知る必要をなくしたい（抽象化）

**適さない状況**:
- モノリスで内部処理がシンプルな場合（オーバーエンジニアリングになる）
- Gateway がSPOFになるリスクを取れない場合（冗長化が前提）

## BFF（Backend for Frontend）との違い

| | API Gateway | BFF |
|---|---|---|
| 目的 | 横断的関心事の集約・トラフィック制御 | クライアント別の最適なAPIを提供 |
| 粒度 | 汎用 | クライアント種別（Web・モバイル）ごとに存在 |
| ロジック | ルーティング・レートリミットのみ | クライアントに合わせたデータ変換・集約 |

両者を組み合わせることも多い（Gateway → BFF → サービス）。
