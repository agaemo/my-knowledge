# Webhook

## なぜ存在するか

HTTPはリクエスト-レスポンス型のため、クライアントが問い合わせなければサーバーからデータを受け取れない。外部サービスとの連携で「何か起きたら教えてほしい」という要件を、問い合わせなしで実現するために生まれたパターン。

## イベント通知パターンの比較

「決済が完了したらすぐ知りたい」という要件を例に、3つのアプローチを比較する。

| | ポーリング | ロングポーリング | Webhook |
|---|---|---|---|
| 誰がリクエストするか | クライアント（定期） | クライアント（保留） | サーバー（イベント時） |
| 仕組み | 一定間隔で「まだ？」と聞き続ける | サーバーがイベントまで応答を保留する | イベント発生時にサーバーが相手のURLにPOSTする |
| 通知の遅延 | ポーリング間隔分 | ほぼなし | ほぼなし |
| 無駄なリクエスト | 多い（変化がなくても毎回送る） | 少ない | なし |
| 受け取り側の要件 | なし | なし | HTTPを受信できるエンドポイントが必要 |
| 主な用途 | シンプルな更新確認 | ブラウザのリアルタイム通知 | サービス間の非同期イベント通知 |

### ロングポーリングとは

通常のポーリングは「一定間隔でリクエストを送り、即座にレスポンスを受け取る」サイクルを繰り返す。イベントがなければ空振りになる。

ロングポーリングはサーバー側が「イベントが来るまでレスポンスを保留」する。クライアントはレスポンスが返ってきたら即座に次のリクエストを送る。

```
クライアント → リクエスト → サーバー（イベント待ち…）
クライアント ← レスポンス ← サーバー（イベント発生 or タイムアウト）
クライアント → 即座に次のリクエスト
```

ブラウザ向けのリアルタイム通知として使われる。WebSocket/SSEが使えない環境向けのフォールバックとして機能することが多い。ブラウザ向けの他のアプローチ（SSE・WebSocket）については[リアルタイム通信プロトコル](/networking/realtime-protocols)を参照。

### Webhookとは

ロングポーリングはあくまでクライアントが主導するプル型。Webhookは逆に、サーバー側がイベント発生時に相手のURLへHTTP POSTを送るプッシュ型。「HTTPコールバック」とも呼ばれる。

```
GitHub  → (pushイベント発生) → POST https://ci.example.com/webhook
Stripe  → (決済完了)         → POST https://api.example.com/webhooks/stripe
Slack   → (メッセージ受信)   → POST https://bot.example.com/events
```

受け取る側がHTTPを受信できるエンドポイントを持っている必要がある。ブラウザには使えない（ブラウザはHTTPリクエストを受信できないため）。

## いつ使うか

外部サービスのイベントを自分のシステムで受け取る場面。

- 決済完了を受けて注文処理を進める（Stripe）
- コードをpushしたらCIを起動する（GitHub）
- フォームが送信されたら処理を始める（Typeform・Zapier）
- メッセージを受け取ってBotが返答する（Slack・Discord）

## 設計上の考慮

### 即時200レスポンス

受け取ったら速やかに200を返す。処理に時間がかかるとタイムアウトと判定されて再送が発生する。重い処理はキューに積んで非同期で行う。

```
受信 → 200 OK を即返す → キューに積む → 非同期で処理
```

### 冪等性

ネットワーク障害やタイムアウトで同じイベントが複数回届くことがある。イベントIDで重複チェックし、同じイベントを2回処理しても結果が変わらないように設計する。

→ [冪等性](/backend/idempotency)

### 署名検証

送信元が正規のサービスかどうかを確認する。多くのサービスはペイロードをHMAC-SHA256で署名し、ヘッダーに付与する。

```
X-Hub-Signature-256: sha256=<HMAC値>
```

受け取り側で同じ秘密鍵を使って署名を検証し、一致しなければ拒否する。

### 再送への備え

送信側は失敗時に指数バックオフで再送する（例：1分後→5分後→30分後）。冪等に処理することを前提として設計する。
