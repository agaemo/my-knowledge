# キャッシュ戦略

頻繁にアクセスされるデータを高速なストレージに一時保存し、レスポンスを高速化・バックエンドの負荷を下げる手法。

## なぜ存在するか

DB へのクエリ・外部 API の呼び出し・重い計算は時間がかかる。同じ結果を毎回生成するのは無駄が多い。

```
キャッシュなし:
  リクエスト → DB クエリ（100ms）→ レスポンス
  100リクエスト → DB に100回クエリ

キャッシュあり:
  1回目: DB クエリ（100ms）→ 結果をキャッシュに保存
  2〜100回目: キャッシュから返す（1ms）→ DB への負荷がほぼゼロ
```

## キャッシュの種類

| 種類 | 例 | 特徴 |
|---|---|---|
| **インメモリキャッシュ** | Redis・Memcached | 最も高速。サーバー再起動で消える |
| **CDN キャッシュ** | Cloudflare・CloudFront | 静的コンテンツ・HTML をエッジで配信 |
| **ブラウザキャッシュ** | Cache-Control ヘッダー | クライアント側に保存。HTTP で制御 |
| **DB クエリキャッシュ** | DB 内部のキャッシュ | DB が自動的に保持 |
| **アプリ内メモリ** | プロセス内の Map など | シンプルだが複数インスタンスで共有できない |

## 主なキャッシュパターン

### Cache-Aside（キャッシュアサイド）

最も一般的なパターン。アプリが明示的にキャッシュを操作する。

```
読み取り:
  1. キャッシュを確認 → ヒットしたらそのまま返す
  2. キャッシュミスなら DB に問い合わせ
  3. 結果をキャッシュに保存して返す

書き込み:
  1. DB を更新
  2. キャッシュを削除（次の読み取りで再構築される）
```

```python
def get_user(user_id):
    cached = redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)          # キャッシュヒット

    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    redis.setex(f"user:{user_id}", 300, json.dumps(user))  # 300秒でexpire
    return user
```

**向いている場面:** 読み取りが多い・データが頻繁に変わらないケース。

### Write-Through（ライトスルー）

DB への書き込みと同時にキャッシュも更新する。

```
書き込み: DB 更新 → キャッシュ更新（同時）
読み取り: キャッシュから返す（常に最新）
```

**メリット:** キャッシュと DB の整合性が高い。  
**デメリット:** 書き込みのたびにキャッシュも更新するためオーバーヘッドがある。

### Write-Behind（ライトビハインド / Write-Back）

キャッシュに書き込んで即座にレスポンスし、DB への反映は非同期で行う。

```
書き込み: キャッシュ更新 → 即レスポンス → 非同期で DB に反映
```

**メリット:** 書き込みが高速。  
**デメリット:** キャッシュが消えると DB への反映前のデータが失われるリスクがある。

### Read-Through（リードスルー）

キャッシュミス時の DB 問い合わせをキャッシュ層が自動で行う。Cache-Aside との違いはアプリが意識しないこと。

## TTL（Time To Live）設計

キャッシュにはデータが古くなる問題（陳腐化）がある。TTL で有効期限を設定する。

```
短い TTL（秒〜分）: リアルタイム性が重要なデータ（在庫数・価格）
長い TTL（時間〜日）: ほぼ変わらないデータ（商品名・設定値）
TTL なし: 静的なデータ（コード・画像）
```

## キャッシュの課題

### キャッシュ無効化（Cache Invalidation）

> "There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton

データが更新されたときに古いキャッシュを確実に消すのが難しい。

**対策:** 更新時にキャッシュキーを削除する・TTL で定期的に失効させる・バージョンをキーに含める（`user:123:v2`）。

### キャッシュスタンピード

TTL 切れが大量に同時に発生すると、DB に一斉にリクエストが集中する。

**対策:** TTL にランダムなジッターを加える・ロック機構で1つのリクエストだけ DB に問い合わせさせる（Probabilistic Early Recomputation）。

### コールドスタート

サーバー再起動直後はキャッシュが空のため、全リクエストが DB に届く。

**対策:** ウォームアップ（起動時に主要データをキャッシュに積む）・段階的なトラフィック移行。

## いつ使うか

- 同じデータへの読み取りが繰り返される
- DB のクエリが遅く、スケールアップより先に試すべき手段として
- 外部 API の呼び出し回数に制限がある（レートリミット）

キャッシュは複雑さを増すため、**まず計測してボトルネックを確認してから導入する**。
