# エッジコンピューティング

処理をユーザーに地理的に近い場所（エッジ）で実行するアーキテクチャ。従来の中央集権型サーバーに対して、世界中に分散した拠点でコードを動かすことでレイテンシを削減する。

## なぜ存在するか

Webアプリが中央のサーバーリージョンにのみ存在する場合、地理的に遠いユーザーへのレスポンスは物理的な通信距離の影響を受ける。エッジコンピューティングは処理をユーザーの近くに移すことで、世界中どこからでも低レイテンシなレスポンスを実現する。

## 仕組み

```
従来型:
ユーザー（東京）→ サーバー（バージニア） RTT: 約150ms

エッジ型:
ユーザー（東京）→ エッジノード（東京） RTT: 約5ms
```

CDNの静的ファイル配信と異なり、エッジでは動的なコード（関数）を実行できる。

## エッジで実行できること

- 認証・認可（JWTの検証など）
- A/Bテストのルーティング
- リクエストの書き換え・リダイレクト
- パーソナライズされたレスポンス
- レートリミット
- 地域ごとのコンテンツ出し分け

## エッジの制約

エッジ環境は通常のNode.js実行環境とは異なり、いくつかの制約がある。

| 制約 | 内容 |
|------|------|
| ランタイム | Web標準API（fetch・Request・Response）のみ。Node.js APIは使えないことが多い |
| 実行時間 | 数十〜数百ミリ秒（長時間処理には不向き） |
| ストレージ | ファイルシステムへのアクセス不可 |
| コールドスタート | ほぼゼロだが、環境によって差がある |

## エッジとサーバーレスの違い

| 観点 | エッジ | サーバーレス（Lambda等） |
|------|--------|------------------------|
| 実行場所 | ユーザーに近い分散拠点 | 特定リージョン |
| レイテンシ | 非常に低い | リージョン依存 |
| ランタイム制約 | 強い（Web標準API） | 緩い（Node.js フル） |
| 向いている処理 | 軽量・高速な応答 | 重い処理・DB接続 |

## 主なエッジプラットフォーム

- **Cloudflare Workers** — 300以上のエッジ拠点。KV・R2・D1などのストレージも提供
- **Vercel Edge Functions** — Next.jsと統合されたエッジ実行環境
- **Fastly Compute** — Wasmベースのエッジ実行環境

## いつ使うか

- グローバルなユーザーに低レイテンシなレスポンスを届けたい
- 認証・リダイレクトなど軽量な処理をサーバー到達前にこなしたい
- 地域・ユーザーごとのコンテンツ出し分けが必要
- CDNキャッシュだけでは対応できない動的なレスポンスが必要
