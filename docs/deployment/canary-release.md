# カナリアリリース

## 何か

新バージョンを全ユーザーに一気に展開せず、最初は一部のユーザー（カナリア）だけに流し、
問題がなければ段階的に全体へ広げるリリース戦略。

炭鉱でガス検知に使われた「カナリア」が由来。新バージョンを先行的にさらす対象のことをカナリアと呼ぶ。

## なぜ存在するか

一括リリースは障害発生時の影響範囲が全ユーザーになる。
カナリアリリースはリスクを段階的に引き受けることで、問題を小さく早期に検出できる。

- 本番環境でしか再現しない問題を少数ユーザーへの影響で抑えられる
- メトリクスを観察しながら安全に拡大できる
- 問題発生時のロールバックが迅速（全体展開前なら影響が限定的）

## 仕組み

```
        ┌─────────────────────────────────────────┐
        │             Load Balancer               │
        └───────────────┬─────────────────────────┘
                        │
          ┌─────────────┴──────────────┐
          │ 5%                         │ 95%
          ▼                            ▼
   [新バージョン v2]            [旧バージョン v1]
   （カナリア）                  （安定版）
          │                            │
          ▼                            ▼
     メトリクス監視               引き続き稼働
  （エラー率・レイテンシ）
```

段階的に拡大: 5% → 20% → 50% → 100%

## ロールアウト判断基準

何を観測して次のステージへ進むかを事前に定義する。

```yaml
# リリース判断基準の例
canary_criteria:
  error_rate: < 0.1%          # エラー率が閾値以下
  p99_latency: < 500ms        # レイテンシが許容範囲内
  observation_window: 30min   # 上記を30分維持したら次のステージへ
  auto_rollback: true         # 閾値超過で自動ロールバック
```

## 実装例

### Nginx による重み付きルーティング

```nginx
upstream backend {
  server v1.internal weight=95;  # 旧バージョン
  server v2.internal weight=5;   # カナリア（新バージョン）
}

server {
  location / {
    proxy_pass http://backend;
  }
}
```

### Kubernetes（Argo Rollouts）

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: { duration: 30m }   # メトリクス確認
        - setWeight: 20
        - pause: { duration: 30m }
        - setWeight: 50
        - pause: { duration: 30m }
        - setWeight: 100             # 全量展開
      analysis:
        templates:
          - templateName: error-rate-check
```

## カナリアの対象ユーザーを選ぶ方法

| 方法 | 内容 | 向いているケース |
|---|---|---|
| ランダム % | リクエストの X% をランダムに振り分け | 最もシンプル |
| ユーザーID | 特定ユーザーIDを固定でカナリアへ | セッション一貫性が必要なとき |
| 社内ユーザー | 従業員だけ先行 | UXへの影響を社内で先に確認したいとき |
| 地域・DC | 特定リージョンだけ | インフラ変更・DB移行のテスト |

## ブルーグリーンとの違い

| | カナリア | ブルーグリーン |
|---|---|---|
| 切り替え方 | 段階的（%を徐々に増やす） | 一括（瞬時に全トラフィックを切替） |
| 本番同時稼働 | 新旧が長期間共存する | 新旧が短期間共存し即切替 |
| コスト | 追加リソースは最小限 | 本番と同規模の環境を2セット用意 |
| 向いている場面 | 長期的なリスク分散 | 即時ロールバックが必要な場面 |

## いつ使うか

- ユーザー規模が大きく、障害の影響範囲を最小化したい
- 新機能のパフォーマンス影響を本番データで測りたい
- A/Bテストと組み合わせてビジネスメトリクスを検証したい

## いつ使わないか

- 新旧バージョンが同時稼働できない（DBスキーマ変更など互換性がない）
- トラフィックが少なくカナリアのサンプルサイズが統計的に意味を持たない
- シンプルな即時切り替えで十分なとき → ブルーグリーンを検討
