# GitOps

Git リポジトリをインフラ・アプリの**唯一の信頼できる情報源（Single Source of Truth）**として扱い、Git への変更を起点に自動デプロイを行う運用モデル。  
Weaveworks が 2017年に提唱。Kubernetes の普及とともに広まった。

## なぜ存在するか

従来の CI/CD では、パイプラインが「プッシュ型」でクラスタに変更を適用する。

```
従来のプッシュ型 CI/CD:
  コードを push → CI が build・test → CI がクラスタに deploy（kubectl apply など）
  問題:
    - CI が本番クラスタへの書き込み権限を持つ（攻撃面が広い）
    - 誰かが手動で kubectl apply した変更が残る（ドリフト）
    - 現在のクラスタの状態を Git を見ても正確にわからない
```

GitOps は「Git の状態 = クラスタの状態」を維持することでこれを解決する。

## 仕組み

**Pull ベースのデプロイ**が核心。クラスタ内のエージェントが Git を監視し、差分を検知したら自動で適用する。

```
GitOps のフロー:
  1. 開発者が Git に Kubernetes マニフェストを push（PR レビュー経由）
  2. クラスタ内のエージェント（ArgoCD など）が Git の変更を検知
  3. エージェントがクラスタの状態を Git の状態に合わせる
  4. Git の状態 = クラスタの状態 が常に維持される
```

## メリット

**ドリフトの自動修正**  
誰かが手動で `kubectl` を実行して変更しても、エージェントが Git の状態に戻す。「本番が何かおかしい」という状態を防ぐ。

**監査ログが Git 履歴として残る**  
「いつ・誰が・何を変えたか」が PR・コミット履歴として残る。

**ロールバックが git revert**  
デプロイの巻き戻しが `git revert` で完結する。

**CI に本番権限を持たせない**  
Pull ベースのため、クラスタ外の CI ツールが本番に直接書き込む権限が不要。攻撃面が狭まる。

## 主なツール

**ArgoCD**  
Kubernetes 向けの GitOps ツール。UI が充実。宣言的に Git リポジトリと Kubernetes クラスタを紐づけ、差分を可視化・自動同期する。

**Flux**  
CNCF のプロジェクト。CLI ベースで Kubernetes への統合が深い。

## 通常の CI/CD との役割分担

GitOps は CI/CD の「CD（Continuous Delivery）」部分を置き換える概念。CI は引き続き必要。

```
CI（継続的インテグレーション）:
  コード変更 → ビルド・テスト → コンテナイメージを生成・push

CD（GitOps）:
  マニフェストを更新（新しいイメージタグ）→ PR → マージ → ArgoCD が自動適用
```

## いつ使うか

- Kubernetes でアプリを運用しているとき
- インフラのドリフト（手動変更の蓄積）が問題になってきたとき
- 本番環境への変更を全て Git の履歴として追跡したいとき
- CI ツールに本番クラスタへの直接権限を持たせたくないとき

Kubernetes を使っていない環境には直接は適用できない。その場合は Terraform などの IaC ツールを Git で管理する「Infrastructure as Code + PR ベースのワークフロー」が近い概念になる。
