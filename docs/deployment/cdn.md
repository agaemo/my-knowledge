# CDN（Content Delivery Network）

## なぜ存在するか

Webコンテンツを単一のオリジンサーバーだけで配信すると、地理的に遠いユーザーは物理的な通信距離の影響でレイテンシが高くなる。また、アクセス集中時にオリジンに負荷が集中する。

CDNは世界中に分散した**エッジサーバー**（PoP: Point of Presence）にコンテンツをキャッシュし、ユーザーに最も近い拠点から配信することでこれらを解決する。

```
オリジンサーバー（東京）だけの場合:
ユーザー（ニューヨーク）→ 東京サーバー  RTT: 約150ms

CDN使用時:
ユーザー（ニューヨーク）→ NY拠点のエッジ  RTT: 約5ms
                              ↑ キャッシュあれば即返却
                              キャッシュなければオリジンへ
```

## 仕組み

### キャッシュとオリジンフェッチ

```
1. ユーザーがファイルをリクエスト
2. 最寄りのエッジサーバーを確認
   - キャッシュHIT  → エッジから即返却
   - キャッシュMISS → オリジンサーバーへ取得し、エッジにキャッシュして返却
3. 以降の同リクエストはエッジから返却
```

### キャッシュ制御

オリジンが返す `Cache-Control` ヘッダーでCDNのキャッシュ動作を制御する。

```
Cache-Control: public, max-age=31536000, immutable
# → 1年間CDNにキャッシュ。ハッシュ付きファイル名（app.abc123.js）に使う

Cache-Control: public, max-age=3600
# → 1時間キャッシュ

Cache-Control: no-store
# → キャッシュしない（個人情報を含むレスポンスなど）
```

## 主な用途

### 静的アセットの配信

画像・JS・CSS・フォントなどは変更頻度が低く、CDNキャッシュと最も相性が良い。
ファイル名にハッシュを付けることで、更新時は新しいURLになりキャッシュを無効化できる。

```
app.abc123.js  → 長期キャッシュ（1年）
app.def456.js  → 内容が変わったら新しいハッシュ
```

### 動画・大容量ファイルの配信

帯域幅を要するコンテンツをオリジンに集中させずエッジから配信する。

### DDoS緩和・WAF

エッジでリクエストを吸収し、悪意あるトラフィックをオリジンに到達させない。

## エッジコンピューティングとの違い

| | CDN | エッジコンピューティング |
|---|---|---|
| 主な役割 | 静的コンテンツのキャッシュ・配信 | エッジでのコード実行（動的処理） |
| コンテンツ | 静的ファイル中心 | 動的レスポンス・ロジック実行 |
| 例 | CloudFront・Fastly・Akamai | Cloudflare Workers・Vercel Edge Functions |

現代のCDNはキャッシュ配信に加えてエッジコンピューティング機能も提供しており、両者の境界は曖昧になっている。

## 主なプロバイダー

| プロバイダー | 特徴 |
|---|---|
| Cloudflare | 世界最大規模のPoP。Workers によるエッジ実行・DDoS対策が強力 |
| AWS CloudFront | AWSサービスとの統合が容易。S3・ALBとの組み合わせが定番 |
| Fastly | リアルタイムキャッシュパージが速い。メディア・ECでの採用が多い |
| Akamai | 最古参。エンタープライズ・大規模配信に強い |

## いつ使うか

- 静的アセット（画像・JS・CSS）の配信を高速化したい
- グローバルユーザーへのレイテンシを下げたい
- オリジンサーバーへの負荷を削減したい
- DDoS対策・WAFをインフラレベルで入れたい
