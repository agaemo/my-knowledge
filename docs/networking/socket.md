# ソケット（Socket）

## なぜ存在するか

プロセス間でデータを送受信するための通信エンドポイント。「どこに・どうやって」データを届けるかを抽象化した仕組みで、ネットワーク越しの通信もローカル内の通信も同じインターフェースで扱える。

## 2種類のソケット

### TCPソケット（IPソケット）

IPアドレス＋ポート番号で通信相手を特定する。ネットワーク越しの通信に使う。

```
127.0.0.1:9000
0.0.0.0:80
```

### Unixドメインソケット

ファイルシステム上のパス（`.sock` ファイル）で通信相手を特定する。**同一マシン上のプロセス間通信専用**。

```
/var/run/php-fpm.sock
/run/gunicorn.sock
```

## .sockファイルの正体 — データは流れていない

`.sock` ファイルはデータを格納しているファイルではない。**「ここに接続口がある」という目印**（アドレス） として機能するだけで、データはカーネルのメモリ上を流れる。

### 仕組みを段階的に

**1. サーバープロセスが「接続口」を作る**

PHP-FPMなどのアプリケーションサーバーが起動するとき、OSに「このパスにソケットを作る」と宣言する。OSはファイルシステム上に `.sock` という特殊ファイルを作成し、その場所とカーネル内の受信バッファを紐づける。

```
PHP-FPM起動
 → カーネルに「/var/run/php-fpm.sock で待ち受ける」と登録
 → ファイルシステムに /var/run/php-fpm.sock が現れる（中身は空）
 → カーネル内に受信キューが用意される
```

**2. クライアントプロセスが接続しにいく**

nginxが `/var/run/php-fpm.sock` を見たとき、「ただのファイル」としては読まずに、「ここに接続する」という操作を行う。OSはそのパスに紐づいたカーネルバッファを探し、両プロセスをつなぐ。

```
nginx → 「/var/run/php-fpm.sock に接続したい」とOSに依頼
OS   → パスからカーネル内のバッファを特定
     → nginx と PHP-FPM の間にメモリ上の通信路を確立
```

**3. データはカーネルのメモリを経由する**

接続後のデータのやり取りは、カーネルが管理するメモリバッファを通して行われる。`.sock` ファイルはもう関係なく、「出会いの場所」として機能し終えている。

```
nginx → [カーネルメモリバッファ] → PHP-FPM
                ↑
        .sockファイルはここには関与しない
```

### ファイルと何が違うか

普通のファイルはディスクにデータを書き込み・読み出しする。`.sock` ファイルはディスクにデータを書かない。カーネルが管理する「特殊ファイル」で、`ls -la` で確認すると種別が `s`（socket）と表示される。

```bash
$ ls -la /var/run/php-fpm.sock
srwxrwxr-x 1 www-data www-data 0 ... /var/run/php-fpm.sock
# ↑ 先頭の "s" が socket を意味する。サイズも 0 バイト
```

### TCPソケットとの対比

IPアドレス＋ポートもデータが流れる場所ではない。`192.168.1.1:80` はあくまで「どこに接続するか」を示すアドレスで、データはやはりカーネルのネットワークバッファを通って流れる。Unixドメインソケットはこのアドレスとして「IPアドレス＋ポート」の代わりに「ファイルパス」を使っているだけ。

| | TCPソケット | Unixドメインソケット |
|-|------------|-------------------|
| アドレスの表現 | `192.168.1.1:9000` | `/var/run/app.sock` |
| データが流れる場所 | カーネルのネットワークバッファ | カーネルのメモリバッファ |
| ネットワークスタック経由 | あり（TCP/IP処理が発生） | なし（カーネル内で完結） |

## TCPソケット vs Unixドメインソケット（選び方）

| | TCPソケット | Unixドメインソケット |
|-|------------|-------------------|
| 通信範囲 | 同一・別マシン問わず | 同一マシン内のみ |
| オーバーヘッド | TCPスタックを経由（若干遅い） | カーネル内で完結（速い） |
| 設定の複雑さ | シンプル | パーミッション設定が必要 |
| ポート競合 | ありうる | なし |

## nginxでの登場シーン

nginxはリバースプロキシとして、バックエンドのアプリケーションサーバー（PHP-FPM・Gunicorn・uWSGIなど）にリクエストを転送する。この転送先の指定にソケットが使われる。

### TCPソケットで渡す場合

```nginx
location ~ \.php$ {
    fastcgi_pass 127.0.0.1:9000;
}
```

```nginx
location / {
    proxy_pass http://127.0.0.1:8000;
}
```

### Unixドメインソケットで渡す場合

```nginx
location ~ \.php$ {
    fastcgi_pass unix:/var/run/php-fpm.sock;
}
```

```nginx
location / {
    proxy_pass http://unix:/run/gunicorn.sock;
}
```

### どちらを選ぶか

- **同一サーバー内**ならUnixドメインソケットが一般的（高速・ポート不要）
- **バックエンドが別サーバー・コンテナ**にいる場合はTCPソケット一択
- Docker Compose でnginxとアプリが別コンテナなら → TCPソケット（`app:8000` のように）
- 同一コンテナ内や同一VMなら → Unixドメインソケットも選択肢

## TCPソケットテーブル

カーネルはすべてのTCPソケットの状態を**コネクションテーブル**（TCPソケットテーブル）として管理している。各エントリはローカルアドレス・リモートアドレス・ポート・状態（LISTEN / ESTABLISHED / TIME_WAIT など）・プロセスIDを持つ。

```
Proto  Local Address        Foreign Address      State        PID/Program
tcp    0.0.0.0:22           0.0.0.0:*            LISTEN       1234/sshd
tcp    192.168.1.10:22      203.0.113.5:54321    ESTABLISHED  5678/sshd
tcp    0.0.0.0:80           0.0.0.0:*            LISTEN       9012/nginx
```

`ss -tulnp`（または旧来の `netstat -tulnp`）で確認できる。

### 主なTCP状態

| 状態 | 意味 |
|------|------|
| LISTEN | 接続待ち受け中 |
| ESTABLISHED | 接続確立済み・通信中 |
| TIME_WAIT | 接続終了後の待機（ポート再利用を防ぐ） |
| CLOSE_WAIT | 相手からFINを受け取り、アプリがまだ閉じていない |

### ペネトレーションテストでの意味

ターゲットのTCPソケットテーブルを読むと、どのサービスがどのポートでリッスンしているかが分かる。ポートスキャン（外部から推測）と組み合わせ、実際にプロセスが存在するかを内部から確認する。

## パーミッションの罠

`.sock` ファイルのオーナー・パーミッションで接続可否が決まる。nginxワーカープロセスのユーザー（多くは`www-data`や`nginx`）が読み書きできないと `502 Bad Gateway` になる。

```bash
# php-fpm側でsocketのグループ・パーミッションを合わせる例
listen.owner = www-data
listen.group = www-data
listen.mode = 0660
```
