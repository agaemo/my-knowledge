# HTTP

## なぜ存在するか

HTTP（HyperText Transfer Protocol）はWebブラウザとサーバーがデータをやり取りするためのプロトコル。HTMLやJSON・画像など、あらゆるWebコンテンツの転送に使われる。1991年に誕生し、インターネットの基盤となっている。

## HTTPの基本

**リクエスト・レスポンス型**
クライアント（ブラウザ・アプリ）がリクエストを送り、サーバーがレスポンスを返す。サーバーから能動的に送信することはできない（→ [リアルタイム通信](./realtime-protocols.md)参照）。

**ステートレス**
各リクエストは独立しており、サーバーは前のリクエストを覚えていない。ログイン状態などを維持するためにCookieやセッション・JWTを使う。

### リクエストの構造

```
GET /api/users HTTP/1.1
Host: example.com
Authorization: Bearer xxxxx
Content-Type: application/json
```

| 要素 | 説明 |
|------|------|
| メソッド | `GET` `POST` `PUT` `PATCH` `DELETE` など |
| パス | リソースのURI |
| HTTPバージョン | `HTTP/1.1` `HTTP/2` など |
| ヘッダー | メタ情報（認証・Content-Type・キャッシュ制御など） |
| ボディ | リクエスト本体（POSTやPUTで使う） |

### レスポンスの構造

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600

{"id": 1, "name": "Alice"}
```

### 主なHTTPメソッド

| メソッド | 用途 | 冪等性 |
|---------|------|--------|
| GET | リソースの取得 | あり |
| POST | リソースの作成・処理の実行 | なし |
| PUT | リソースの完全置換 | あり |
| PATCH | リソースの部分更新 | なし（設計次第） |
| DELETE | リソースの削除 | あり |
| HEAD | レスポンスヘッダーのみ取得（ボディなし） | あり |
| OPTIONS | 許可されているメソッドの確認（CORSプリフライト） | あり |

### 主なステータスコード

| コード | 意味 |
|--------|------|
| 200 OK | 成功 |
| 201 Created | リソース作成成功 |
| 204 No Content | 成功（レスポンスボディなし） |
| 301 Moved Permanently | 恒久リダイレクト |
| 302 Found | 一時リダイレクト |
| 400 Bad Request | リクエストの形式が不正 |
| 401 Unauthorized | 認証が必要 |
| 403 Forbidden | 認証済みだがアクセス権限なし |
| 404 Not Found | リソースが見つからない |
| 409 Conflict | リソースの競合（重複など） |
| 422 Unprocessable Entity | バリデーションエラー |
| 429 Too Many Requests | レート制限超過 |
| 500 Internal Server Error | サーバー側エラー |
| 503 Service Unavailable | サーバーが一時的に利用不可 |

## バージョンの変遷

各バージョンは前世代の構造的な限界（線頭ブロッキング・冗長ヘッダー）を段階的に解決してきた。

### 各バージョンの比較

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|-|----------|--------|--------|
| プロトコル | テキスト | バイナリ | バイナリ（QUIC） |
| 多重化 | なし（1接続1リクエスト） | あり（1接続で並列） | あり |
| ヘッダー圧縮 | なし | HPACK | QPACK |
| HoLブロッキング | TCPレベルで発生 | TCPレベルで発生 | なし（UDPベース） |
| 接続確立 | TCP + TLS | TCP + TLS | QUIC（0-RTT可） |
| サーバープッシュ | なし | あり（廃止傾向） | あり（廃止傾向） |

### HTTP/1.1 の問題点

#### 線頭ブロッキング（Head-of-Line Blocking）

1つのコネクションで同時に扱えるリクエストは1つだけ。前のリクエストのレスポンスが返ってくるまで次のリクエストは送れない。

```
接続1: [リクエストA →][←────────── レスポンスA ──────────][リクエストB →]...
                                                                 ↑ Aが終わるまで待たされる
```

ブラウザはこれを回避するために同じドメインへ複数のTCP接続を並列に張る（通常6本）。しかしTCP接続の確立（三方向ハンドシェイク + TLS）にはコストがかかり、根本解決にはならない。

**Pipelining**（HTTP/1.1 の機能）はリクエストを先送りできるが、レスポンスは送った順に返さなければならないため、遅いレスポンスが後続を止めるという問題は解消されなかった。実質的に普及しなかった。

#### ヘッダーの冗長性

毎リクエストで `Cookie` `User-Agent` `Accept-Encoding` など同じヘッダーをテキストで送り続ける。ページ読み込み1回あたり数十〜数百リクエストが発生するSPAでは無視できないオーバーヘッドになる。

### HTTP/2 が解決したこと

#### 多重化（Multiplexing）

1本のTCP接続上で複数のリクエスト・レスポンスを並列に扱う。通信をバイナリの**フレーム**に分割し、**ストリーム**（論理的な通信チャネル）に割り当てて混在させる。

```
1本のTCP接続
├─ ストリーム1: [リクエストA →]     [←レスポンスA]
├─ ストリーム2:   [リクエストB →][←レスポンスB]
└─ ストリーム3:      [リクエストC →]          [←レスポンスC]
```

アプリケーション層（HTTP層）のHoLは解消した。6本のTCP接続を張る必要もなくなった。

#### ヘッダー圧縮（HPACK）

送信済みのヘッダーをテーブルとして保持し、差分のみ・または参照番号で送る。

#### TCP層のHoLは残る

TCPはパケットを順序通りに届けることを保証する（→ [TCP/UDP](./tcp-udp.md)）。パケットが1つ欠落すると、後続のパケットが届いていてもバッファで保留され、欠落したパケットの再送を待つ。

```
TCP受信バッファ
パケット3 ✓ → アプリへ渡す
パケット4 ✓ → アプリへ渡す
パケット5 ✗ → 欠落。再送要求...
パケット6 ✓ → バッファで待機（パケット5が来るまで渡せない）
パケット7 ✓ → バッファで待機
```

ストリーム2のデータ（パケット6・7）も、ストリーム1で欠落したパケット5を待たされる。HTTP/2がどれだけ多重化しても、TCP層では1本のバイトストリームでしかないため、この問題は避けられない。

### HTTP/3 が解決したこと

#### QUIC（QUICKではなくQUIC）

HTTP/3はTCPではなく**QUIC**（RFC 9000）をトランスポートに使う。QUICはUDPの上に構築されたプロトコルで、ストリームの多重化・暗号化・再送を独自に実装している。

```
HTTP/2（TCPの上）          HTTP/3（QUICの上）
─────────────────          ─────────────────
ストリーム1 ──┐             ストリーム1 → パケット欠落 → ストリーム1だけ待つ
ストリーム2 ──┤ TCP         ストリーム2 → 影響なし、継続
ストリーム3 ──┘ 1本のバイト列  ストリーム3 → 影響なし、継続
      ↓
1パケット欠落で全停止
```

ストリームをTCP層ではなくQUIC層で管理するため、あるストリームのパケット欠落が他のストリームに波及しない。

#### 0-RTT接続

過去に接続したことのあるサーバーへは、ハンドシェイクなしで最初のパケットからデータを送り始められる（0-RTT）。初回接続でも TLS ハンドシェイクと TCP ハンドシェイクを同時に行う（1-RTT）。

HTTP/2 は TCP ハンドシェイク（1RTT）+ TLS ハンドシェイク（1〜2RTT）が必要なため、通信開始までの遅延がより大きい。

## いつ使うか

- **既存サービスのパフォーマンス改善**：まずHTTP/2対応を確認する。多くのCDN・nginxはデフォルト対応済み
- **モバイル・不安定回線**：HTTP/3（QUIC）が効果的。Cloudflare・Google・AkamaiはHTTP/3を有効にしている
- **gRPC**：HTTP/2が前提（ストリーミングRPCに多重化を利用）
- **WebSocket**：HTTP/1.1または HTTP/2のストリームでアップグレード
