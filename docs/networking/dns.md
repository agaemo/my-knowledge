# DNS（Domain Name System）

## なぜ存在するか

IPアドレスは人間が覚えにくいため、`example.com` のような名前をIPアドレスに変換する仕組みが必要。デプロイのたびにドメイン設定が絡むため、仕組みを理解していないと設定ミスやTTL起因の問題で詰まる。

## 名前解決の流れ

```
ブラウザ → ローカルキャッシュ → OSリゾルバ
         → フルサービスリゾルバ（ISPなど）
         → ルートネームサーバー（.）
         → TLDネームサーバー（.com, .jpなど）
         → 権威ネームサーバー（example.comの管理者）
         → IPアドレス返却
```

**フルサービスリゾルバ**（再帰リゾルバ）：クライアントの代わりに問い合わせを繰り返し、最終的なIPを返す。ISP（Internet Service Provider：インターネット接続事業者）が提供するものや、Google（8.8.8.8）、Cloudflare（1.1.1.1）など。

**権威ネームサーバー**：そのドメインのレコードを実際に保持しているサーバー。Route 53・Cloudflare DNS・さくらのネームサーバーなど。

## 主要なDNSレコード

| レコード | 役割 | 例 |
|---------|------|-----|
| **A** | ドメイン → IPv4アドレス | `example.com → 93.184.216.34` |
| **AAAA** | ドメイン → IPv6アドレス | `example.com → 2606:2800:...` |
| **CNAME** | ドメイン → 別ドメイン（エイリアス） | `www.example.com → example.com` |
| **MX** | メール受信サーバーの指定 | `example.com → mail.example.com` |
| **TXT** | テキスト情報（ドメイン所有権確認・SPFなど） | SPF, DKIM, Google検証 |
| **NS** | 権威ネームサーバーの指定 | `example.com → ns1.cloudflare.com` |
| **SOA** | ゾーンの管理情報 | プライマリNS・TTL設定など |

## AレコードとCNAMEの使い分け

**Aレコード**：IPアドレスを直接指定する。ルートドメイン（`example.com`）に使う。

**CNAME**：別ドメインへの参照。IPが変わっても追従する。サブドメイン（`www.example.com`）や CDN・外部サービス（`app.example.com → xxx.vercel.app`）への紐付けに使う。

ルートドメインにCNAMEは設定できない（RFCの制約）。CloudflareのCNAME Flatteningなど独自実装で回避するサービスもある。

## TTL（Time To Live）

キャッシュの有効期間（秒）。TTL=3600なら、変更しても最大1時間は古いIPが参照され続ける。

**デプロイ前の戦略**
- 移行の数時間〜1日前にTTLを短く（300秒など）しておく
- 切り替え後にTTLを元に戻す

## いつ詰まるか

- **CNAME→A の多段参照**：CNAMEが別のCNAMEを指し、さらに先がAレコード。深くなるほど解決に時間がかかる
- **TTLが長いまま切り替え**：古いIPに一部ユーザーが向き続ける
- **NSレコードの変更伝播**：ネームサーバーの変更は世界中のキャッシュが切れるまで最大48時間かかることがある
