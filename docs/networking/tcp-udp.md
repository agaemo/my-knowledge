# TCP / UDP

OSI参照モデルのL4（トランスポート層）に位置するプロトコル。IPがどこへ届けるかを決めるのに対し、TCPとUDPは「どのように届けるか」を決める。

## ポート番号

IPアドレスがマシンを特定するのに対し、**ポート番号**はそのマシン上のどのプロセスに届けるかを特定する。0〜65535（16ビット）の整数。

```
192.168.1.10:80  →  IPアドレス:ポート番号
```

| 範囲 | 名称 | 用途 |
|---|---|---|
| 0〜1023 | ウェルノウンポート | OSが管理。バインドにroot権限が必要 |
| 1024〜49151 | 登録済みポート | アプリケーションが使用する慣例的な番号 |
| 49152〜65535 | エフェメラルポート | OSがクライアント側に一時的に割り当てる |

主なウェルノウンポート：

| ポート | プロトコル |
|---|---|
| 22 | SSH |
| 25 | SMTP |
| 53 | DNS |
| 80 | HTTP |
| 443 | HTTPS |
| 3306 | MySQL |
| 5432 | PostgreSQL |

クライアントがサーバーに接続するとき、OSはエフェメラルポートを自動的に割り当て、レスポンスの返送先として使う。`netstat` や `ss` コマンドで確認できる。

## TCP（Transmission Control Protocol）

**信頼性のある、順序保証のある通信**を提供する。

### 三方向ハンドシェイク

データを送る前に、双方が「通信を始める準備ができた」ことを確認し合う。

```
クライアント          サーバー
    |── SYN ──────────→|   「接続したい（シーケンス番号 x を送る）」
    |← SYN-ACK ────────|   「OK、受け取った（x+1）。こちらのシーケンス番号は y」
    |── ACK ──────────→|   「了解（y+1）」
    |                  |
    ↓  データ転送開始   ↓
```

シーケンス番号はパケットの順序管理と重複排除に使う。

### TCPが保証すること

- **到達保証**：ACKが返ってこなければ自動的に再送する
- **順序保証**：シーケンス番号によりバラバラに届いても正しい順に並べ直す
- **フロー制御**：受信側がウィンドウサイズで「この量まで受け取れる」と通知し、送りすぎを防ぐ
- **輻輳制御**：ネットワークの混雑を検知して送信量を自動調整する（スロースタート・AIMD）

### コネクション終了（四方向）

```
終了したい側         相手側
    |── FIN ──────→|   「送信を終わる」
    |←─ ACK ───────|   「了解」
    |←─ FIN ───────|   「こちらも終わる」
    |── ACK ──────→|   「了解」
    |                  |
（TIME_WAIT状態へ）
```

終了を開始した側は `TIME_WAIT` という状態に入り、約 2×MSL（Maximum Segment Lifetime、通常60〜120秒）待機する。最後のACKが届かなかった場合に相手がFINを再送してくることがあるため、その受け皿として残る。

大量の短命なコネクションを張るサーバーでは TIME_WAIT が大量発生し、エフェメラルポートが枯渇することがある。

### 使いどころ

データが確実に・正確に届く必要がある通信全般。

- HTTP/HTTPS（1.0・1.1・2）
- SSH・SFTP
- SMTP・IMAP
- データベース接続（MySQL・PostgreSQL）

## UDP（User Datagram Protocol）

**コネクションレス・配送保証なし**の軽量プロトコル。ヘッダーが8バイトしかなく（TCPは最低20バイト）、ハンドシェイクも再送もなし。

### UDPが選ばれる理由

信頼性を捨てることで得られるメリットがある。

**低レイテンシ**：ハンドシェイクが不要なため、最初のパケットがすぐ届く。

**ヘッドオブラインブロッキングがない**：TCPでは1パケットが欠落すると、後続のパケットが全て待たされる。UDPにはこの制約がない。

**古いデータが無意味な場合**：動画ストリーミングや音声通話では、再送で遅れて届いた古いフレームは使い物にならない。捨てた方がいい。

### 使いどころ

| 用途 | 理由 |
|---|---|
| DNS | 1往復の問い合わせ。速度最優先 |
| 動画ストリーミング | 多少欠落しても視聴に影響小 |
| VoIP・オンラインゲーム | リアルタイム性最優先。遅延の方が致命的 |
| HTTP/3（QUIC） | UDPの上に独自の信頼性機構を実装 |

### QUIC と HTTP/3

HTTP/3 はTCPではなく**QUIC**というプロトコルを使う。QUICはUDPの上に構築されており、ストリームの多重化・独自の暗号化・独自の再送を実装している。

TCPだとヘッドオブラインブロッキング（HoL）により1つのパケット欠落が全ストリームを止めてしまう。QUICはストリーム単位で管理するため、1つのストリームの問題が他のストリームをブロックしない。

HoLブロッキングがHTTPの各バージョンでどう扱われてきたかは [HTTP](./http.md#線頭ブロッキングhead-of-line-blocking) を参照。

## TCP vs UDP まとめ

| | TCP | UDP |
|---|---|---|
| コネクション | あり（ハンドシェイク必要） | なし |
| 到達保証 | あり（再送あり） | なし |
| 順序保証 | あり | なし |
| 速度 | 遅い（オーバーヘッドあり） | 速い |
| ヘッダーサイズ | 最低20バイト | 8バイト |
| 主な用途 | HTTP・SSH・DB接続 | DNS・動画・VoIP・QUIC |

## Webエンジニアが遭遇する場面

**`Connection refused`**：相手のポートでサーバーが動いていない、またはファイアウォールで弾かれた。TCPのSYNを送ったがRSTが返ってきた状態。

**`Connection timed out`**：SYNを送ったが何も返ってこない。ファイアウォールがパケットをドロップしているか、ルートが存在しない。

**`TIME_WAIT` の大量発生**：短命なHTTPコネクションを大量に張るサーバーで発生しやすい。`keep-alive` の活用・`SO_REUSEADDR` の設定・カーネルパラメータのチューニングで対処する。

**ポート番号の衝突**：`Address already in use` エラーは、同じポートを複数のプロセスが使おうとしているか、TIME_WAITでポートが占有されているときに発生する。
