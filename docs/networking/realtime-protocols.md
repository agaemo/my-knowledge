# リアルタイム通信プロトコル

## なぜ存在するか

HTTPはクライアントからリクエストを送らないとサーバーからデータを受け取れない（リクエスト・レスポンス型）。チャット・通知・株価など「サーバーから能動的にデータを送りたい」場面でこの制約を回避するために複数の手法が登場した。

## 3つのアプローチ比較

| | Long Polling | SSE | WebSocket |
|-|-------------|-----|-----------|
| 方向 | サーバー→クライアント（擬似） | サーバー→クライアント | 双方向 |
| プロトコル | HTTP | HTTP（`text/event-stream`） | ws:// / wss:// |
| 接続 | リクエストを都度張り直す | 1接続を維持 | 1接続を維持 |
| オーバーヘッド | 高（HTTPヘッダーを毎回送る） | 低 | 最低 |
| 再接続 | 手動実装 | 自動（`retry`フィールド） | 手動実装 |
| HTTP/2対応 | ○ | ○（多重化の恩恵あり） | 一部制限あり |
| プロキシ通過 | 容易 | 容易 | 設定が必要なことがある |

## Long Polling

```
クライアント → リクエスト → サーバー（データが来るまでレスポンスを保留）
クライアント ← レスポンス ← サーバー（データが来たら返す or タイムアウト）
クライアント → 即座に次のリクエスト
```

**メリット**：通常のHTTPで実装できる。  
**デメリット**：接続が頻繁に張り直されるためオーバーヘッドが大きい。スケールしにくい。

**いつ使うか**：WebSocketが使えない環境（古いロードバランサ・プロキシ）やシンプルな通知の実装。

## SSE（Server-Sent Events）

サーバーから一方向にイベントを送り続けるHTTPストリーム。`Content-Type: text/event-stream` で接続を保持し、`data:` 形式で送信する。

```
data: {"message": "hello"}\n\n
data: {"message": "world"}\n\n
```

**メリット**：ブラウザ標準API（`EventSource`）で簡単に実装できる。接続断時の自動再接続付き。  
**デメリット**：クライアント→サーバー方向は通常のHTTPリクエストで別途送る必要がある。

**いつ使うか**：通知・フィード更新・AIチャットのストリーミング応答（ChatGPTのような逐次表示）。

## WebSocket

初回はHTTPでハンドシェイクし（`Upgrade: websocket`）、その後 TCP接続上で独自のフレームプロトコルに切り替える。双方向にいつでも送信できる。

```
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: ...

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
```

**メリット**：低レイテンシの双方向通信。ゲーム・コラボ編集・チャットに最適。  
**デメリット**：ステートフルな接続のため、スケールアウト時にスティッキーセッションかPub/Subが必要。

**いつ使うか**：クライアントとサーバーが頻繁に双方向通信するケース（チャット・オンラインゲーム・共同編集）。

## 選び方

```
双方向かつ低レイテンシが必要？
  → WebSocket

サーバー→クライアントの一方向でよい？
  → SSE（ブラウザ向け）

WebSocket/SSEが使えない環境（制約あり）？
  → Long Polling
```
