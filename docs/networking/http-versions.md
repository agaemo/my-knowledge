# HTTP バージョンの変遷

## なぜ存在するか

HTTP/1.1 の構造的な限界（線頭ブロッキング・冗長ヘッダー）がパフォーマンスのボトルネックになったため、段階的に改善された。各バージョンが「何の問題を解決したか」を理解すると、チューニング方針の判断に使える。

## 各バージョンの比較

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|-|----------|--------|--------|
| プロトコル | テキスト | バイナリ | バイナリ（QUIC） |
| 多重化 | なし（1接続1リクエスト） | あり（1接続で並列） | あり |
| ヘッダー圧縮 | なし | HPACK | QPACK |
| HoLブロッキング | TCPレベルで発生 | TCPレベルで発生 | なし（UDPベース） |
| 接続確立 | TCP + TLS | TCP + TLS | QUIC（0-RTT可） |
| サーバープッシュ | なし | あり（廃止傾向） | あり（廃止傾向） |

## HTTP/1.1 の問題点

**線頭ブロッキング（Head-of-Line Blocking）**
1つのリクエストが完了するまで次のリクエストを送れない。ブラウザはこれを回避するために同じドメインに対して複数のTCP接続を張る（通常6本）が、接続確立コストがかかる。

**ヘッダーの冗長性**
毎リクエストで `Cookie` や `User-Agent` など同じヘッダーをテキストで送り続ける。

## HTTP/2 が解決したこと

**多重化（Multiplexing）**
1本のTCP接続上で複数のリクエスト・レスポンスを並列に送受信できる。ストリームと呼ばれる論理チャネルに分割される。

**ヘッダー圧縮（HPACK）**
差分のみ送信。繰り返し登場するヘッダーは参照番号で代替。

**ただしTCPレベルのHoLは残る**
パケットロスが起きると、TCP層でそのパケットの再送を待つ間、後続のストリームも止まる。

## HTTP/3 が解決したこと

**トランスポートをUDPベースのQUICに変更**
QUICは各ストリームを独立して管理するため、1ストリームのパケットロスが他のストリームを止めない。

**0-RTT接続**
過去に接続したことのあるサーバーには、ハンドシェイクなしでデータを送り始められる。

## いつ使うか

- **既存サービスのパフォーマンス改善**：まずHTTP/2対応を確認する。多くのCDN・nginxはデフォルト対応済み
- **モバイル・不安定回線**：HTTP/3（QUIC）が効果的。Cloudflare・Google・AkamaiはHTTP/3を有効にしている
- **gRPC**：HTTP/2が前提（ストリーミングRPCに多重化を利用）
- **WebSocket**：HTTP/1.1または HTTP/2のストリームでアップグレード
