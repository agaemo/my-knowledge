# ゼロデイ攻撃

## 何か

開発者・ベンダーがまだ把握していない脆弱性（ゼロデイ脆弱性）を突く攻撃。
修正パッチが存在しない状態で行われるため、従来の「パッチを当てれば防げる」対策が通用しない。

「ゼロデイ」とは、脆弱性が公開されてからパッチが出るまでの猶予が **0日** であることを指す。

## なぜ危険か

通常の脆弱性対応では「脆弱性が発見される → ベンダーが修正する → パッチを適用する」という流れがある。
ゼロデイ攻撃はこの流れの最初のステップが存在しない状態、つまり**守る側が脆弱性の存在を知らないまま攻撃を受ける**。

発見された脆弱性が公開されずに闇市場で売買されることもあり、国家レベルの攻撃者が保有・悪用するケースもある。

## 脆弱性が公開されるまでの流れ

```
発見 → 悪用（ゼロデイ期間）→ 公開・報告 → パッチリリース → パッチ適用
                ↑
        この期間が「ゼロデイ」
```

脆弱性が公開された後も、パッチが適用されるまでの間は攻撃可能な状態が続く（n-day攻撃）。

## 対策

ゼロデイはコードの書き方では防げない。多層防御と早期検知が基本方針になる。

- **依存ライブラリの最小化**: 使うライブラリが少ないほど攻撃面（アタックサーフェス）が小さくなる
- **最小権限の原則**: 侵害された場合でも、権限を最小にしておくことで被害範囲を限定する
- **WAF（Web Application Firewall）**: 既知の攻撃パターンに加え、異常なリクエストを検知・遮断する
- **異常検知・監視**: 通常と異なる挙動を早期に検知する
- **ネットワーク分離**: 重要なシステムへの経路を絞り、侵害が横展開しにくい構成にする
- **迅速なパッチ適用**: ゼロデイが公開された瞬間からパッチ適用までの時間を短くする体制を作る
