# ファイアウォールルール

ネットワークトラフィックを「送信元IP・宛先IP・ポート・プロトコル・方向」の組み合わせで許可・拒否するルールセット。OSやネットワーク機器に実装され、通信の入口・出口を制御する。

## なぜ存在するか

ネットワークに接続されたシステムは原則すべてのポートへの接続を受け付けられる状態になっている。ファイアウォールルールは「必要な通信だけを許可し、それ以外を遮断する」最小権限の原則をネットワーク層で実現するための仕組み。

## ステートレスとステートフル

### ステートレスフィルタリング

パケット1つを単独で判断する。送信元・宛先・ポート・フラグだけを見るため、コネクションのコンテキストを持たない。

```
ルール: 宛先ポート80 → 許可
→ HTTP リクエストもレスポンスも個別に評価する
```

### ステートフルフィルタリング

コネクションテーブルを持ち、確立済みセッションの戻りパケットを自動的に許可する。現代のファイアウォールのほとんどはこちら。

```
外部 → 内部へのポート80（ESTABLISHED/RELATED）→ 許可
→ 内部から開始したHTTP通信の応答は自動的に通過する
```

## 方向（インバウンド・アウトバウンド）

| 方向 | 意味 | 例 |
|------|------|----|
| インバウンド | 外部 → 自システムへの着信 | SSHポート22への接続 |
| アウトバウンド | 自システム → 外部への発信 | curl での外部API呼び出し |
| フォワード | 自システムを経由する通過通信 | ルーター・NAT環境での中継 |

## Linuxでの実装（iptables / nftables）

```
# iptablesの例
# 確立済みセッションの通信を許可
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# SSH（22番）を許可
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# HTTP/HTTPS を許可
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# それ以外のインバウンドを拒否
iptables -A INPUT -j DROP
```

nftables はiptablesの後継で、ルール評価が高速・構文が統一されている。

## ペネトレーションテストでの意味

攻撃者はファイアウォールルールの「隙間」を探す。

- **許可されているポートを悪用**: HTTP(80)やHTTPS(443)は通常許可されているため、C2通信をHTTPSに乗せてフィルタを回避する
- **アウトバウンドが緩い環境**: インバウンドは厳しいが、アウトバウンドが制限されていない場合、リバースシェルが有効になる
- **エグレスフィルタリングの欠如**: 外部への通信を制限していないとデータ窃取が容易になる
