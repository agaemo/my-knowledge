# 暗号化の基礎

データを第三者が読めない形に変換する技術。「なぜ TLS が必要か」「なぜパスワードをハッシュで保存するか」の根拠になる概念。

## 3つの基本概念

### 共通鍵暗号（対称暗号 / Symmetric Encryption）

暗号化と復号に**同じ鍵**を使う。

```
平文 + 鍵A → 暗号文 → 鍵A → 平文
```

- **代表アルゴリズム:** AES（Advanced Encryption Standard）— 現在の標準
- **速い** — 大量のデータを暗号化するのに適する
- **課題:** 鍵をどう相手に安全に渡すか（鍵配送問題）

**使われる場面:** ファイルの暗号化・DB の暗号化・TLS の通信データ暗号化（鍵交換後）

### 公開鍵暗号（非対称暗号 / Asymmetric Encryption）

**公開鍵**（誰でも知ってよい）と**秘密鍵**（自分だけが持つ）のペアを使う。

```
暗号化: 平文 + 公開鍵 → 暗号文
復号:   暗号文 + 秘密鍵 → 平文
```

- **代表アルゴリズム:** RSA・楕円曲線暗号（ECDSA）
- **遅い** — 計算コストが高いため、大量データには使わない
- **鍵配送問題を解決** — 公開鍵は誰に渡してもよい

**使われる場面:**
- TLS の鍵交換（共通鍵を安全に共有するために使う）
- デジタル署名（秘密鍵で署名 → 公開鍵で検証）
- JWT の RS256 署名

### ハッシュ関数（Hash / 一方向関数）

入力から固定長の値を生成する。**元に戻せない（不可逆）**。

```
「password123」→ SHA-256 → 「ef92b778...」  （復元不可）
```

- **代表アルゴリズム:** SHA-256・bcrypt・Argon2
- 同じ入力からは常に同じ出力（決定的）
- 出力から入力を逆算することが計算上不可能

**使われる場面:** パスワードの保存・データの整合性確認・JWT の署名

## TLS（HTTPS）での組み合わせ

公開鍵暗号と共通鍵暗号を組み合わせて使う。

```
1. クライアントがサーバーの公開鍵を取得（証明書）
2. 公開鍵暗号で「セッション鍵（共通鍵）」を安全に交換
3. 以降の通信は共通鍵暗号（AES）で高速に暗号化
```

公開鍵暗号で「鍵配送問題」を解決し、共通鍵暗号で「速度」を確保する。

## パスワードの保存

パスワードは平文でも、単純な SHA-256 でも保存してはいけない。

```
Bad: password → SHA-256 → ハッシュをDBに保存
     → レインボーテーブル攻撃で解読される（よくあるパスワードのハッシュは計算済み）

Good: password + ランダムなソルト → bcrypt/Argon2 → ハッシュをDBに保存
     → ソルトが違えばハッシュが違う → 同じパスワードでも別の値になる
     → bcrypt/Argon2 は意図的に遅い（総当たりを困難にする）
```

**bcrypt・Argon2 を使う理由:** 計算を意図的に遅くすることで、総当たり攻撃のコストを上げる設計になっている。

## デジタル署名

「この内容を、この人が書いた」ことを証明する仕組み。JWT・コードの署名・証明書に使われる。

```
署名:   データのハッシュ + 秘密鍵 → 署名値
検証:   署名値 + 公開鍵 → ハッシュを復元 → データのハッシュと照合
```

JWT の場合: ヘッダー + ペイロードをサーバーの秘密鍵で署名 → クライアントは公開鍵で検証するだけで、DBを参照せずにトークンの正当性を確認できる。

## 使い分けまとめ

| 目的 | 使う技術 |
|---|---|
| 通信の暗号化 | TLS（公開鍵で鍵交換 + AES で本通信） |
| パスワード保存 | bcrypt / Argon2（ソルト付きハッシュ） |
| データの改ざん検知 | SHA-256 ハッシュ（チェックサム） |
| 認証トークン | JWT（HS256: 共通鍵署名 / RS256: 公開鍵署名） |
| ファイル・DB の暗号化 | AES（共通鍵暗号） |
