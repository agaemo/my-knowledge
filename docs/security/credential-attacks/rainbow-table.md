# レインボーテーブル攻撃

ハッシュ値と元の文字列の対応を事前計算したテーブルを使って、ハッシュから元のパスワードを逆引きする攻撃。

## 仕組み

ハッシュ関数は一方向で `hash("password") → 5f4dcc3b...` は計算できても逆算は不可能。しかし大量の文字列とそのハッシュ値の対応表（レインボーテーブル）を事前に作っておけば、ハッシュ値を検索するだけで元の文字列がわかる。

```
漏洩したハッシュ: 5f4dcc3b5aa765d61d8327deb882cf99
テーブル検索 → "password"
```

単純なハッシュ（MD5・SHA-1など）を使ったパスワード保存では、よく使われるパスワードのハッシュが一覧化されており、DBが流出した瞬間に即座に解析される。

## なぜ単純なハッシュでは不十分か

ハッシュは「改ざん検知」や「データ整合性確認」のために設計されており、高速なことが要件。パスワード保存に使うと大量試行が容易になる。MD5は現代のGPUで毎秒数百億回以上計算できる。

## 対策

- **ソルト**: ユーザーごとにランダムな文字列（ソルト）をパスワードに付加してからハッシュする。同じパスワードでも異なるハッシュ値になるため事前計算テーブルが無効化される
- **低速ハッシュ（bcrypt・Argon2・scrypt）**: 計算コストを意図的に高くし、大量試行を困難にする。GPUで高速化される汎用ハッシュ（MD5・SHA-256）はパスワード保存に使ってはならない
- **ペッパー**: 全ユーザー共通の秘密値をパスワードに付加する。ソルトとは別にサーバー側の秘密として管理し、DBが単体で流出しても解析されない（[暗号化の基礎](/security/encryption)参照）

ソルトはレインボーテーブルを無効化するが、ブルートフォース攻撃そのものは防がない。bcrypt/Argon2はどちらの攻撃も困難にする。
