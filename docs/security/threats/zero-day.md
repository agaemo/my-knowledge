# ゼロデイ攻撃・N-day攻撃

## 脆弱性対応の流れと攻撃タイミング

```
発見 → 悪用          → 公開・CVE採番 → パッチリリース → パッチ適用完了
       ↑                               ↑
  ゼロデイ期間                      N-day期間
（パッチが存在しない）           （パッチはあるが未適用）
```

## ゼロデイ攻撃

開発者・ベンダーがまだ把握していない脆弱性（ゼロデイ脆弱性）を突く攻撃。
修正パッチが存在しない状態で行われるため、「パッチを当てれば防げる」対策が通用しない。

「ゼロデイ」とは、脆弱性が公開されてからパッチが出るまでの猶予が **0日** であることを指す。

通常の脆弱性対応では「発見 → ベンダーが修正 → パッチを適用」という流れがある。ゼロデイ攻撃は守る側が脆弱性の存在を知らないまま攻撃を受けるため、対応の起点が存在しない。

発見された脆弱性が公開されずに闇市場で売買されることもあり、国家レベルの攻撃者が保有・悪用するケースもある。

## N-day攻撃

CVEとして公開・採番された**既知の脆弱性**を、パッチが適用されていないシステムに対して突く攻撃。
「N」はCVE公開からの経過日数を表す。

パッチはすでに存在するにもかかわらず、多くの組織がパッチ適用を後回しにする。攻撃者はCVEが公開された直後から自動スキャンツールで未パッチのシステムを探し始めるため、公開から数時間〜数日で悪用が始まるケースも珍しくない。

ゼロデイより技術的な難易度が低く、CVEの情報・既存のエクスプロイトコードが公開されているため攻撃者にとって取り組みやすい。実際の被害件数はゼロデイより N-day の方が多い。

## 対策

どちらの攻撃もコードの書き方では直接防げない。多層防御・迅速なパッチ適用・監視が基本方針になる。

- **迅速なパッチ適用**: CVE公開後できる限り早くパッチを適用する体制を作る。N-day攻撃への最も直接的な対策
- **依存ライブラリの管理**: Dependabot・Snyk などで既知の脆弱性を継続的に検出し、更新漏れをなくす
- **依存ライブラリの最小化**: 使うライブラリが少ないほど攻撃面（アタックサーフェス）が小さくなる
- **最小権限の原則**: 侵害された場合でも、権限を最小にしておくことで被害範囲を限定する
- **WAF（Web Application Firewall）**: 既知の攻撃パターンを検知・遮断する
- **異常検知・監視**: 通常と異なる挙動を早期に検知する
- **ネットワーク分離**: 重要なシステムへの経路を絞り、侵害が横展開しにくい構成にする
