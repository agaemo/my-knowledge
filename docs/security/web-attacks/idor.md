# IDOR（Insecure Direct Object Reference）

リソースへの参照（IDなど）をユーザーに直接公開し、認可チェックなしで他ユーザーのリソースにアクセスできる脆弱性。「壊れたアクセス制御（Broken Access Control）」の代表的なパターン。

## 仕組み

URLやパラメータのIDを書き換えるだけで他ユーザーのデータにアクセスできる。

```
GET /api/orders/12345   → 自分の注文
GET /api/orders/12346   → 他人の注文（IDを+1するだけ）

GET /api/invoices?id=abc-123  → 自分の請求書
GET /api/invoices?id=abc-124  → 他人の請求書
```

開発時は「ログインしているなら自分のデータしか見ない」と思いがちだが、リクエストのIDを書き換えても認可チェックが走らない実装になっていることが多い。

## 対策

- **リソースごとの認可チェック**: リクエストされたリソースの所有者とリクエスト者が一致するか毎回確認する。認証（誰か）と認可（何をしていいか）は別の処理
- **推測しにくいID**: 連番の整数ではなくUUID v4などを使う。推測を困難にするが、認可チェックの代替にはならない
- **間接参照のマッピング**: ユーザーには意味のないキーを渡し、サーバー側でリソースにマッピングする

OWASPトップ10の「アクセス制御の不備（Broken Access Control）」に分類され、実際の脆弱性報告で最も頻度が高いカテゴリの一つ。
