# ヌルバイト汚染

文字列にヌルバイト（`\x00`）を挿入することで、言語・ライブラリ間の文字列終端の解釈のずれを突く攻撃手法。C言語系の関数はヌルバイトを文字列の終端として扱うため、高水準言語側の検証を通過しながらファイルシステムやC拡張に渡る値を改ざんできる。

## なぜ起きるか

PHP・Python・Java などの高水準言語は文字列をバイト列として扱い、途中のヌルバイトを特別視しない。一方、OSのファイルシステム操作やCで書かれた拡張ライブラリはヌルバイトを文字列終端と見なす。この解釈のずれが攻撃面になる。

```
PHPのバリデーション:  "shell.php\x00.jpg" → 末尾が .jpg と判定 → 検証通過
OSのファイル書き込み: "shell.php\x00.jpg" → \x00 で終端 → shell.php として保存
```

## ファイルアップロードへの応用

拡張子チェックを回避する典型的な使い方。

```
アップロードファイル名: shell.php%00.jpg
                              ↑ URLエンコードされたヌルバイト
```

サーバーが拡張子を見て `.jpg` と判定しても、保存時にC系の関数を通ると `shell.php` として書き込まれる。

## ディレクトリトラバーサルへの応用

パス操作でも使われる。

```
入力:   ../../../../etc/passwd%00.jpg
判定:   .jpg で終わる → 許可
実際:   ../../../../etc/passwd として読み込まれる
```

## 影響を受けるバージョン

PHPでは 5.3.4 以降でファイル操作関数のヌルバイト処理が修正された。古いバージョンや独自のC拡張を使う環境では現在も有効なことがある。

## 対策

- **パスや拡張子を許可リストで検証する**: ヌルバイトを含む文字列は許可リストにマッチしない
- **ヌルバイトを明示的に除去・拒否する**: 入力値にヌルバイトが含まれていたらエラーとして扱う
- **言語・ライブラリを最新に保つ**: 既知の修正が適用されているバージョンを使う
