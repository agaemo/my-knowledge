# ファイルアップロード脆弱性

ファイルアップロード機能を持つWebアプリで、アップロードするファイルの種類・内容を適切に検証しないと、サーバー上で任意のコードを実行できるファイルを設置される脆弱性。

## なぜ起きるか

Webアプリはファイルをアップロードさせ、そのファイルをWebルート配下に保存することがある。そこにPHPやJSP等のサーバーサイドスクリプトを置けてしまい、なおかつURLで直接アクセスできる状態であれば、攻撃者はそのファイルを「Webシェル」として実行できる。

```
攻撃の流れ:
1. shell.php をプロフィール画像としてアップロード
2. サーバーが /uploads/shell.php に保存
3. ブラウザで https://example.com/uploads/shell.php?cmd=id にアクセス
4. サーバー上で id コマンドが実行され結果が返る
```

## Webシェル

ファイルアップロード脆弱性を悪用した場合の典型的な到達点。サーバーサイドスクリプトをアップロードしてURLでアクセスすることで、ブラウザからサーバー上のOSコマンドを実行できる状態になる。

```php
<?php system($_GET['cmd']); ?>
```

このような1行のPHPファイルをアップロードして実行できれば、`?cmd=id`・`?cmd=ls` のようにクエリパラメータ経由でコマンドを渡せる。より高機能なWebシェルはファイルマネージャ・コード実行・リバースシェル接続などをGUI形式で提供するものもある。

Webシェルを足場に権限昇格・横断移動へ進むため、初期侵入の入口として攻撃者に好まれる。

## 検証が不十分なケース

- **`accept` 属性のみ**: HTMLの `<input accept="image/*">` はブラウザのファイル選択ダイアログを制限するだけ。DevToolsで属性を削除するか、curlやBurp Suiteでリクエストを直接送れば無効化できる。サーバー側に検証がなければ意味がない
- **拡張子のみチェック**: `.php` を `.php.jpg` や `shell.PhP` に変えて回避される
- **Content-Typeのみチェック**: リクエストのMIMEタイプはクライアント側で自由に変更できる
- **マジックバイトのみチェック**: 画像ファイルのヘッダーを持つPHPファイルを作れる（ポリグロットファイル）

## 悪用される条件

脆弱性が実際の攻撃になるには複数の条件が重なる必要がある。

| 条件 | 内容 |
|------|------|
| 実行可能な場所に保存される | Webルート配下かつスクリプトエンジンが処理する場所 |
| URLで直接アクセスできる | アップロードパスが推測・列挙できる |
| サーバーがファイルを実行する | Apacheが `.php` をPHPとして処理するなど |

## 対策

- **ファイルをWebルートの外に保存する**: URLで直接アクセスできない場所に置く。ダウンロード時はアプリ経由で提供する
- **ファイル名をサーバー側で生成する**: UUIDなど推測不能な名前に変え、拡張子は許可リストのものだけ付与する
- **コンテンツを検証する**: 画像なら画像ライブラリで再エンコードし、元のバイナリを保存しない
- **実行権限を持たせない**: アップロードディレクトリのファイルをスクリプトとして実行しないようサーバーを設定する
