# SSRF（Server-Side Request Forgery）

サーバーに外部URLへのリクエストを発行させ、外部からアクセスできない内部リソースにアクセスさせる攻撃。

## 仕組み

Webアプリが「URLを受け取ってその内容を取得する」機能（OGPプレビュー・Webhook・画像変換・URLスクリーンショットなど）を持つとき、攻撃者は内部ネットワークのURLを渡す。

```
正常なリクエスト: fetch("https://example.com/image.png")

攻撃:
  fetch("http://169.254.169.254/latest/meta-data/iam/security-credentials/")
  → AWSのIAMロール認証情報が取得できる

  fetch("http://localhost:6379/")
  → 内部Redisサーバーに接続できる

  fetch("http://internal-api/admin/users")
  → 外部公開していない管理APIにアクセスできる
```

クラウド環境では `169.254.169.254`（AWSメタデータエンドポイント）から IAM ロールの認証情報が取得できるため特に危険。

## 対策

- **URLの検証**: ホスト名を許可リストで管理し、内部IPアドレス（`10.x.x.x`・`172.16.x.x`・`192.168.x.x`・`127.x.x.x`・`169.254.x.x`）を拒否する
- **DNSリバインディング対策**: DNS解決後のIPアドレスも検証する（ドメインが一見外部に見えても内部IPに解決されるケースがある）
- **メタデータエンドポイントの保護**: AWSではIMDSv2（トークンベース）を強制してv1を無効化する
- **ネットワーク分離**: アプリサーバーから内部ネットワークへの不要な経路をファイアウォールで遮断する
