# WAF（Web Application Firewall）

WebアプリへのHTTPトラフィックを検査し、既知の攻撃パターンにマッチするリクエストをブロックする防御コンポーネント。通常のファイアウォールがIPアドレス・ポートレベルで判断するのに対し、WAFはHTTPの中身（URL・ヘッダー・ボディ）を見て判断する。

## なぜ存在するか

アプリケーションコードの脆弱性をすべてゼロにすることは現実的でない。WAFはアプリの前段に置くことで、既知の攻撃パターンを防御層として追加する。脆弱性の修正が追いつかない間の緩和策としても機能する。

## 仕組み

```
インターネット → WAF → Webアプリ

WAFがリクエストを検査:
  - SQLインジェクションのシグネチャ（' OR 1=1 など）
  - XSSのパターン（<script> タグなど）
  - ディレクトリトラバーサル（../ の連続など）
  → マッチすればブロック、しなければ通過
```

## ルールの種類

| 種類 | 内容 |
|------|------|
| シグネチャベース | 既知の攻撃パターンと照合。OWASP Core Rule Set（CRS）が代表例 |
| レートリミット | 単位時間あたりのリクエスト数を制限してブルートフォースを緩和 |
| 地理的フィルタ | 特定の国・地域からのアクセスをブロック |
| カスタムルール | アプリ固有の条件（特定パスへのアクセス制限など）を追加 |

## 限界

WAFは万能ではなく、過信すると危険。

- **シグネチャのバイパス**: エンコード・大文字小文字・コメント挿入など、パターンを少し変えるだけでルールを回避できることがある
- **誤検知**（False Positive）: 正常なリクエストがブロックされてサービスに影響が出る
- **既知パターンのみ**: ゼロデイや独自ロジックの脆弱性は検出できない
- **暗号化された通信の限界**: TLS終端がWAFより後ろにある場合、中身を検査できない

WAFはあくまで追加の防御層であり、アプリケーション側の入力バリデーション・エスケープ・パラメータ化クエリを代替するものではない。

## 配置パターン

| 配置 | 内容 |
|------|------|
| クラウド型 | Cloudflare・AWS WAF・Akamai など。CDNと統合されることが多い |
| オンプレミス型 | ネットワーク境界に物理・仮想アプライアンスを設置 |
| ホスト型 | アプリサーバー上のモジュール（mod_security など）として動作 |
