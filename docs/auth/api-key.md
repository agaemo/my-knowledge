# APIキー認証

## 何か

事前に発行したランダムな文字列（APIキー）をリクエストに含めることでアクセスを識別・認証する方式。

リクエストヘッダー（`X-API-Key` など）やクエリパラメータとしてAPIキーを送る。

## なぜ存在するか

OAuth2やJWTはユーザーが介在することを前提とした認証フロー（ログイン・コールバック等）を持つ。
サーバー間通信や開発者向けAPIでは、このフローが不要であり、シンプルな認証手段としてAPIキーが使われる。

## 特徴

**シンプルさ**: ヘッダーに文字列を付けるだけ。認証ライブラリや複雑なフローが不要。

**スコープ管理の難しさ**: 「このキーはどのリソースに何ができるか」という粒度管理を自前で実装する必要がある。OAuthのようなスコープ仕様が標準化されていない。

**失効と再発行**: キーが漏洩したら即座に無効化・再発行できる設計が重要。有効期限のないAPIキーは漏洩したまま長期間使われるリスクがある。

**誰が使っているかの追跡**: APIキーごとにレート制限・利用統計・アクセスログを管理することで、どのクライアントがどう使っているかを把握できる。

## いつ使うか

- サーバー間通信（ユーザー認証が不要なバックエンドAPI）
- 開発者向けの外部公開API（天気API・地図APIなど）
- BtoB連携（パートナー企業が自社システムから呼ぶ）

ユーザーが操作するWebアプリには向かない。APIキーをフロントエンドのコードに置くと漏洩リスクがある。
