# 認証方式の使い分け

## 一覧と一言まとめ

| 方式 | 一言 |
|---|---|
| [Basic 認証](/auth/basic-auth) | ユーザー名・パスワードをヘッダーに乗せる最もシンプルな方式 |
| [セッション認証](/auth/session-auth) | サーバーがセッションを保持。従来型Webアプリの標準 |
| [APIキー認証](/auth/api-key) | 事前発行トークンによるシンプルな認証。サーバー間・外部API向け |
| [JWT](/auth/jwt) | トークン自体に情報を持つ。ステートレスでスケールしやすい |
| [Bearer 認証](/auth/bearer) | JWT や OAuth2 トークンをヘッダーに乗せる HTTP の規約 |
| [OAuth2](/auth/oauth2) | 認可の委譲。「別サービスのリソースへのアクセスを許可する」 |
| [OIDC](/auth/oidc) | OAuth2 + 認証。「誰としてログインしているか」を標準化 |
| [SAML](/auth/saml) | エンタープライズSSO。XMLベースで企業システム間の認証連携 |

## 場面別の使い分け

### ユーザーがログインするWebアプリを作りたい

```
小〜中規模・管理画面  →  セッション認証
SPA + API 構成       →  JWT（Bearer で送る）
```

### 「Google でログイン」などのソーシャルログインを実装したい

```
→ OIDC（OAuth2 の上に認証を追加したもの）
```

### 外部サービスのAPIを使いたい（GitHub・Slack など）

```
ユーザーの代わりに操作  →  OAuth2（認可コードフロー）
自分のシステムが操作    →  OAuth2（Client Credentials）
```

### 開発者向けAPIを外部に公開したい

```
→ APIキー認証
```

### 会社のSaaSツール（Salesforce・Slackなど）をIdPと繋げたい

```
→ SAML（既存エンタープライズIdPがある場合）
→ OIDC（新規構築・IdPが対応している場合）
```

### ちょっとした社内ツールを手軽に非公開にしたい

```
→ Basic 認証（Nginx・Cloudflare などサーバー側で設定）
```

## 比較表

### 状態管理

| 方式 | 状態の持ち方 | スケール |
|---|---|---|
| セッション認証 | サーバー（DB/Redis）が保持 | セッションストアが必要 |
| JWT | クライアント（トークン内）が保持 | ステートレスで水平スケール可 |
| Basic 認証 | なし（毎回送る） | スケールしやすい |
| APIキー | サーバーがキー情報を保持 | 軽量 |

### 即時ログアウト

| 方式 | 即時無効化 |
|---|---|
| セッション認証 | 可能（セッションを削除） |
| JWT | 難しい（有効期限まで有効） |
| Basic 認証 | 難しい（ブラウザがキャッシュ） |
| APIキー | 可能（キーを失効） |
| OAuth2 / OIDC | トークン失効エンドポイント次第 |

### 向いているクライアント

| 方式 | ブラウザ | モバイル | サーバー間 |
|---|---|---|---|
| Basic 認証 | △（ダイアログUXが悪い） | × | ○ |
| セッション認証 | ◎ | △（Cookie管理が必要） | × |
| JWT + Bearer | ○ | ◎ | ◎ |
| OAuth2 / OIDC | ◎ | ◎ | ○ |
| APIキー | × | △ | ◎ |
| SAML | ◎ | △ | × |

## よくある混同

**OAuth2 は認証ではない**
OAuth2 は「誰に何を許可するか」の認可プロトコル。「誰がログインしているか」を知るには OIDC が必要。

**JWT は認証方式ではない**
JWT はトークンの形式（データ構造）であり、認証フロー自体はセッション認証や OAuth2 などが行う。JWT はその結果を運ぶ入れ物。

**Bearer は認証システムではない**
Bearer は「Authorization ヘッダーにトークンを乗せる」という HTTP の規約。中身のトークンが JWT であれ OAuth2 アクセストークンであれ、運び方の話。
