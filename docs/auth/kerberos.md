# Kerberos

## 何か

信頼できる第三者（KDC: Key Distribution Center）が発行する「チケット」を使って相互認証を行うプロトコル。パスワードそのものをネットワークに流さずに本人確認できる。

## なぜ存在するか

MITが開発した、信頼できないネットワーク上でもパスワードを平文同然の形で送らずに認証したいという要求から生まれた。クライアントだけでなくサーバー側も本人確認できる「相互認証」を実現する点が、当時の認証方式との大きな違いだった。後に Windows Active Directory の標準認証プロトコルとして採用され、エンタープライズのSSO基盤の事実上の標準になった。

## 仕組み

1. クライアントが KDC の AS（Authentication Server）にログイン情報を提示し、TGT（Ticket Granting Ticket）を取得する
2. TGT を使って KDC の TGS（Ticket Granting Service）にサービスチケットを要求する
3. 発行されたサービスチケットを目的のアプリケーションサーバーに提示してアクセスする
4. 各ステップは対称鍵暗号で保護され、パスワード自体は一度もネットワーク上に流れない

TGT を最初に取得しておけば、その後は複数のサービスにパスワード入力なしでアクセスできる。これが Kerberos ベースの SSO の正体。

## 特徴

**相互認証**: クライアントがサーバーを騙れないだけでなく、サーバーもクライアントになりすませない。なりすましサーバーへの接続を防げる。

**チケットには有効期限がある**: 一定時間で失効するため、KDC・クライアント・サーバー間の時刻同期（NTPなど）がずれるとチケットの検証に失敗する。運用上の典型的な障害要因。

**ドメイン環境前提**: Windows Active Directory や Linux の FreeIPA など、KDC を中心としたドメイン構成があることが前提。インターネット越しの不特定多数向けサービスには向かない。

**RADIUSとの違い**: RADIUS は「ネットワークへの接続」を中央サーバーに問い合わせる仕組みだが、Kerberos は「ドメイン内の複数サービス」へのSSOを実現する仕組み。レイヤーが異なり、組み合わせて使われることもある。

## いつ使うか

- Windows Active Directory ドメイン環境でのSSO
- 社内のLinux/Unixサーバー群を統合認証したい場合（NFS・SSHなどKerberos対応サービスとの連携）
- 大学・研究機関など、MIT発祥の歴史的経緯から Kerberos が標準になっているネットワーク
