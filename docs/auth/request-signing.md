# リクエスト署名（Request Signing）

APIリクエストに署名を付与することで、「正規の送信者からのリクエストであること」「送信後に改ざんされていないこと」を受信側が検証できる仕組み。`sig` や `signature` というパラメータ名で渡されることが多い。

## なぜ必要か

APIキー認証は「正しいキーを持っているか」しか確認できない。リクエストの内容が途中で改ざんされても検知できないし、キーが漏れれば誰でもなりすませる。

リクエスト署名は **リクエスト全体にキーを絡めてハッシュを計算する** ため、以下が保証される。

- **真正性：** 秘密鍵を持つ送信者だけが正しい署名を生成できる
- **完全性：** パラメータを1文字でも変えると署名が一致しなくなる
- **リプレイ攻撃への耐性：** タイムスタンプを署名対象に含めることで、過去のリクエストの再送を防げる

## 仕組み（HMAC）

署名の計算には **HMAC（Hash-based Message Authentication Code）** が広く使われる。

```
署名の生成（送信側）:
  1. 署名対象の文字列を組み立てる
     例: "POST\n/api/orders\ntimestamp=1716451200&amount=1000"
  2. HMAC-SHA256(秘密鍵, 署名対象文字列) を計算
  3. Base64 や Hex でエンコードして sig パラメータとして送る

検証（受信側）:
  1. 受け取ったパラメータから同じ文字列を組み立てる
  2. 同じ秘密鍵で HMAC を計算する
  3. 送られてきた sig と一致するか比較する
```

秘密鍵はサーバーとクライアントだけが持つ。署名の計算結果から秘密鍵を逆算することはできない。

## 署名対象に含めるもの

何を署名対象に含めるかは API 設計によって異なるが、一般的には以下を組み合わせる。

| 要素 | 目的 |
|---|---|
| HTTPメソッド・パス | エンドポイントの改ざんを防ぐ |
| クエリパラメータ・ボディ | パラメータの改ざんを防ぐ |
| タイムスタンプ | リプレイ攻撃を防ぐ（古い署名を弾く） |
| ノンス | 同じ秒内の重複リクエストを防ぐ |

## 代表的な採用例

**AWS Signature Version 4：** AWS のすべての API で使われる署名方式。日付・リージョン・サービス名を階層的に使ったキー導出が特徴。

**Stripe Webhook 署名：** Stripe からのWebhookリクエストが本物かを検証するために `Stripe-Signature` ヘッダーに署名が付く。受け取り側がペイロードを同じ手順で署名し一致を確認する。

**Slack Signing Secret：** Slack からのリクエストを `X-Slack-Signature` ヘッダーで検証する。タイムスタンプを含めてリプレイ攻撃を防ぐ設計。

**Twitter / X API v1：** OAuth 1.0a の署名方式として採用。現在は OAuth 2.0 ベアラートークンに移行しているが、リクエスト署名の代表的な事例として知られる。

## API キー認証との違い

| | APIキー認証 | リクエスト署名 |
|---|---|---|
| 検証対象 | キーの一致 | キー＋リクエスト内容の一致 |
| 改ざん検知 | できない | できる |
| リプレイ攻撃 | 防げない | タイムスタンプで防げる |
| 実装コスト | 低い | やや高い |

機密性の高いリクエスト（決済・個人情報の変更など）や、Webhook の受信検証のように「送信元を信頼する必要がある」場面でリクエスト署名が選ばれる。
