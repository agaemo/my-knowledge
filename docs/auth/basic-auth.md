# Basic 認証

## 何か

HTTP の標準仕様（RFC 7617）に定義された最もシンプルな認証方式。
ユーザー名とパスワードを `:` でつなぎ、Base64 エンコードして `Authorization` ヘッダーに付与する。

```
Authorization: Basic dXNlcjpwYXNzd29yZA==
                      ↑ base64("user:password")
```

## なぜ存在するか

HTTP はステートレスなプロトコルであり、リクエストをまたいで「誰が送ったか」を識別する仕組みが必要だった。
Basic 認証はその最初期の解として HTTP 仕様に組み込まれた。ブラウザが標準でサポートしており、追加のライブラリなしに動作する。

## 仕組み

1. クライアントが保護されたリソースにリクエストを送る
2. サーバーが `401 Unauthorized` と `WWW-Authenticate: Basic realm="..."` を返す
3. ブラウザがユーザー名・パスワードの入力ダイアログを表示する
4. クライアントが認証情報を Base64 エンコードして再リクエスト
5. サーバーがデコードして検証し、正しければリソースを返す

以降のリクエストでは、ブラウザが自動的に `Authorization` ヘッダーを付与し続ける。

## 注意点

**Base64 は暗号化ではない**: エンコードされた文字列は誰でも元に戻せる。HTTPS なしで使うと盗聴されそのまま使われる。**HTTPS は必須**。

**ログアウトができない**: ブラウザはセッションが終わるまで認証情報を保持し続ける。明示的なログアウト機能を実装することが難しい。

**パスワードが毎回送られる**: セッションや JWT と異なり、毎リクエストにパスワード相当の情報が含まれる。

## ブラウザのダイアログ

URLを直接ブラウザに入力してアクセスすると、ブラウザがユーザー名・パスワードの入力を求めるダイアログを表示することがある。

::: info なぜダイアログが出るのか
ブラウザはURL遷移時に `Authorization` ヘッダーなしでリクエストを送る。サーバーが `401 Unauthorized` と `WWW-Authenticate: Basic` を返すと、ブラウザがその応答を受け取り、標準のUIとして認証ダイアログを自動的に表示する仕組み。
:::

入力して認証が通ると、ブラウザはそのセッション中の以降のリクエストに `Authorization` ヘッダーを自動付与し続ける。タブを閉じるまで再入力は不要。

## いつ使うか

- 社内ツール・管理画面などシンプルに非公開にしたい場合
- ローカル開発環境の保護
- サーバー間通信（ユーザーが介在しないシステム）

ユーザー向けのWebアプリには向かない。ログアウトのUXが悪く、パスワードを常時送り続ける設計になるため。
