# RADIUS

## 何か

ネットワーク機器（VPNゲートウェイ・Wi-Fiアクセスポイント・スイッチなど）への接続を、中央の認証サーバーに問い合わせて許可する AAA（Authentication, Authorization, Accounting）プロトコル。

## なぜ存在するか

1990年代、ダイヤルアップISPは多数のアクセスサーバー（NAS: Network Access Server）を抱えていたが、ユーザー情報をそれぞれに分散して持たせると管理が破綻する。RADIUS は認証ロジックを1台の中央サーバーに集約し、NAS側は「問い合わせるだけ」の薄いクライアントにする仕組みとして生まれた。この設計思想は今も Wi-Fi の 802.1X 認証や VPN 認証で使われ続けている。

## 仕組み

1. クライアント（PC・スマホ）が NAS（Wi-Fi AP・VPNゲートウェイ・スイッチなど）に接続を試みる
2. NAS が RADIUS サーバーに `Access-Request` を送信する（ユーザー名・暗号化されたパスワードなど）
3. RADIUS サーバーが認証し、`Access-Accept` または `Access-Reject` を返す
4. Accept の場合、VLAN 割り当てやアクセス可能時間帯などの認可情報も一緒に返せる
5. 接続の開始・終了を `Accounting-Request` として記録する（利用ログ・課金用）

NAS と RADIUS サーバーの間は事前共有した shared secret で保護されるが、パスワード以外のフィールドは保護が弱く、プロトコル自体が古い設計であることに注意が必要。

## 特徴

**AAA を1つのプロトコルで扱う**: 認証・認可・アカウンティングを一括で提供する。VPN や Wi-Fi の利用ログを取りたい場面に向く。

**バックエンド連携が前提**: RADIUS サーバー自身がユーザー情報を持つこともあるが、多くは LDAP や Active Directory に問い合わせる「橋渡し役」として使われる。

**UDP ベース**: ポート 1812（認証・認可）/ 1813（アカウンティング）。コネクションレスなため、タイムアウトやリトライの設計が必要。

**Webの認証プロトコルとは層が違う**: OAuth2 や SAML は「アプリケーション」へのアクセスを扱うが、RADIUS は「ネットワークそのもの」への接続を扱う。Wi-Fi に繋がった後にアプリへログインする際は別途 OIDC や SAML が使われる、という分業になる。

## いつ使うか

- 企業の有線/無線LAN認証（802.1X 環境でのユーザー認証）
- VPN接続時のユーザー認証
- ネットワーク機器（ルーター・スイッチ）の管理者ログインを中央集権化したい場合
