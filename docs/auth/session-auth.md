# セッション認証

## 何か

ログイン後にサーバーがセッションを生成し、そのIDをCookieでクライアントに渡す認証方式。
以降のリクエストではCookieのセッションIDを使ってサーバーが状態を参照する。

## なぜ存在するか

HTTP はステートレスなため、リクエストをまたいで「誰がログインしているか」を保持する仕組みが必要だった。
セッション認証はサーバー側でその状態を管理することでこれを解決した。
Webの黎明期から使われてきた最も一般的な認証方式。

## 仕組み

1. ユーザーがログイン情報を送信する
2. サーバーが認証を検証し、セッションを生成（セッションIDとユーザー情報をサーバーに保存）
3. セッションIDをCookieとしてクライアントに返す
4. 以降のリクエストでブラウザが自動的にCookieを送信する
5. サーバーがセッションIDをキーにユーザー情報を参照して処理する

## 特徴

**即時無効化が可能**: セッションをサーバーから削除すれば即座にログアウトできる。JWT にはないこの性質は、強制ログアウトや権限剥奪が必要なユースケースで重要。

**スケールの課題**: セッション情報をサーバーに保持するため、複数サーバー構成ではセッションの共有が必要になる。Redis などのセッションストアを別途用意するのが一般的。

**CSRF への注意**: Cookieは同一オリジンのリクエストに自動付与されるため、CSRF攻撃の対象になりやすい。CSRFトークンや `SameSite` Cookie 属性による対策が必要。

## JWTとの比較

| | セッション認証 | JWT |
|---|---|---|
| 状態の保持場所 | サーバー（DB/Redis） | クライアント（トークン内） |
| スケール | セッションストアが必要 | ステートレスでスケールしやすい |
| 即時無効化 | 簡単 | 難しい（ブラックリストが必要） |
| 向いている場面 | 従来のWebアプリ・管理画面 | API・マイクロサービス・SPA |

## いつ使うか

- 従来型のサーバーレンダリングWebアプリ
- 管理画面など即時ログアウトが必要な場面
- スケール要件が単純なサービス（セッションストア不要の小規模構成）
