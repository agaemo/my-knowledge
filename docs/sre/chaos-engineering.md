# カオスエンジニアリング

本番環境に意図的に障害を注入し、システムの耐障害性を実験で検証する手法。

## なぜ存在するか

障害は防げない。ネットワーク遅延・サーバークラッシュ・依存サービスの停止は必ず起きる。問題は「障害が起きたとき、システムが想定通りに振る舞うかどうか」が、実際に障害が起きるまでわからないこと。

カオスエンジニアリングは「驚き」を事前に消費する。制御された条件で意図的に壊すことで、未知の弱点を本番障害として経験する前に発見する。

Netflix が AWS 移行時に、インスタンスが突然停止してもストリーミングが続くことを検証するために **Chaos Monkey** を開発したのが起源。

## 実験の進め方

```
1. 定常状態を定義する
   例: エラー率 < 0.1%、p99 レイテンシ < 200ms

2. 仮説を立てる
   例: 「商品サービスが落ちても注文サービスはデグレードして動き続ける」

3. 障害を注入する
   例: 商品サービスのポッドを1台落とす

4. 観察・比較する
   定常状態が維持されているか確認する

5. 元に戻し、結果から改善する
```

## 注入できる障害の種類

| カテゴリ | 例 |
|---|---|
| ネットワーク | レイテンシ追加・パケットロス・帯域制限 |
| インスタンス | プロセスクロック・EC2/Pod のシャットダウン |
| 依存サービス | 特定サービスへの接続断・エラー返却 |
| リソース | CPU 高負荷・メモリ圧迫・ディスク枯渇 |
| リージョン | AZ 障害・DNS 障害 |

## ゲームデーとの違い

| | カオスエンジニアリング | ゲームデー |
|---|---|---|
| 頻度 | 定期的・継続的 | 数ヶ月に1回 |
| 実施者 | ツールが自動実行 | 人が手動で実施 |
| 目的 | 継続的な検証 | 大規模シナリオ訓練 |

## 関連パターン

カオスエンジニアリングで弱点が見つかったら、[サーキットブレーカー](/sre/resilience/circuit-breaker)・[バルクヘッド](/sre/resilience/bulkhead)・[タイムアウト](/sre/resilience/timeout) で対処する。[SLO](/sre/slo-sli-sla) と組み合わせることで「どこまで壊れていいか」の基準が明確になる。

## ツール

- **Chaos Monkey**（Netflix OSS）— AWS EC2 インスタンスのランダム停止
- **Litmus** — Kubernetes ネイティブのカオスエンジニアリング
- **AWS Fault Injection Service (FIS)** — AWS マネージドの障害注入サービス
- **Gremlin** — SaaS 型のカオスエンジニアリングプラットフォーム
