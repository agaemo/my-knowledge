# ポストモーテム

## 何か

インシデント（障害・重大な問題）が発生した後に、原因・影響・再発防止策を記録・共有するプロセスと文書。
"事後検証"とも呼ばれる。Googleが SRE Book で普及させた。

## Blameless（非難なし）の原則

「誰が悪いか」ではなく「なぜシステムがそうなったか」を問う。
個人を責める文化では、人は隠蔽し、組織は学習できなくなる。

エンジニアは当時の情報と制約の中で最善の判断をしていた、という前提に立つ。

## ポストモーテム文書の構成

| セクション | 内容 |
|---|---|
| 概要 | インシデントの一文要約・重大度・影響範囲 |
| タイムライン | 検知→対応→復旧の時系列。UTC統一が望ましい |
| 根本原因 | 直接原因と、それを可能にした構造的要因 |
| 影響 | ユーザー影響・ビジネス影響・継続時間 |
| 対応の振り返り | うまくいったこと・改善できること |
| アクションアイテム | 担当者・期限付きの具体的な改善タスク |

## 5 Whys（なぜなぜ分析）

表面的な原因から「なぜ？」を繰り返して根本原因に辿り着く手法。
ただし「なぜ人がミスしたか」ではなく「なぜシステムがミスを許したか」を問う方向で使う。

## 実施タイミング

インシデント収束後、記憶が新鮮なうちに（通常72時間以内）実施する。
規模・重大度に関わらず定期的に行う組織ほど、同じ障害を繰り返さない。

## アクションアイテムの質

「注意する」「確認を徹底する」はアクションアイテムとして無効。
「アラートを追加する」「ロールバック手順を自動化する」のように、人間の判断に依存しない対策にする。
