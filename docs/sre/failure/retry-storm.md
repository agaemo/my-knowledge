# リトライストーム

## 何か

障害中のサービスに対して、多数のクライアントが一斉にリトライを繰り返し、回復しかけたサービスを再びダウンさせる現象。
Thundering Herd（雷鳴る群れ）とも呼ばれる。

## なぜ起きるか

1. サービスAが一時的に過負荷で落ちる
2. クライアント（数百〜数千台）が即座にリトライを開始
3. 過負荷が増幅され、サービスAが回復できない
4. サービスAの回復 → 再びリトライ集中 → 再ダウン、を繰り返す

単純な「失敗したらすぐリトライ」実装が引き起こす。

## 対策

### Exponential Backoff（指数バックオフ）

リトライのたびに待機時間を倍増させ、サービスに回復の時間を与える。

```
1回目失敗 → 1秒後にリトライ
2回目失敗 → 2秒後にリトライ
3回目失敗 → 4秒後にリトライ
4回目失敗 → 8秒後にリトライ ...
```

### Jitter（ジッター）

バックオフに乱数を加えて、クライアント間のリトライタイミングをばらつかせる。
全クライアントが同じ間隔でリトライすると、バックオフしても結局一斉に来る。

```
待機時間 = random(0, min(cap, base * 2^attempt))
```

### サーキットブレーカーとの組み合わせ

リトライストームが発生しそうな状況では[サーキットブレーカー](../resilience/circuit-breaker)がOpenになり、クライアント側でリトライ自体を止める。

### リトライ上限とDead Letter Queue

無限リトライは避け、上限回数を超えたら DLQ に積んで後で処理する。

## リトライすべきでないケース

- `400 Bad Request`（クライアントのバグ）
- `401 Unauthorized`（認証エラー）
- `422 Unprocessable Entity`（バリデーションエラー）

リトライして意味があるのは`500`系・ネットワーク断・タイムアウトに限る。
