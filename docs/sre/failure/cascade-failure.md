# カスケード障害

## 何か

一部のコンポーネントの障害が連鎖的に広がり、システム全体を巻き込む現象。
分散システム特有のリスクで、局所的な問題が全障害に発展する。

## なぜ起きるか

### 過負荷の転送

障害インスタンスへのリクエストが生きているインスタンスに集中し、そこも過負荷で落ちる。
これが繰り返されて全インスタンスが倒れる。

### タイムアウト待ちの積み重ね

応答が遅い依存先を待ち続けるスレッド・コネクションが増加し、リソースを枯渇させる。

### リトライストーム

クライアントが一斉にリトライすることで、回復しかけたサービスに再び大量のリクエストが集中する。
→ [リトライストーム](./retry-storm)

## 典型的なシナリオ

```
DB レスポンス遅延
  → API サーバーのスレッド枯渇（コネクション待ち）
    → フロントエンドへのレスポンス遅延
      → フロントエンドのタイムアウト増加
        → ユーザーが再リクエスト → 負荷さらに増加
          → API サーバー全台ダウン
```

## 対策

| パターン | 役割 |
|---|---|
| [サーキットブレーカー](../resilience/circuit-breaker) | 障害のある依存先への呼び出しを遮断 |
| [バルクヘッド](../resilience/bulkhead) | 障害の波及範囲をコンポーネント間で隔離 |
| [タイムアウト設計](../resilience/timeout) | 応答を無限に待たず、早期に失敗を確定させる |
| [バックプレッシャー](../resilience/backpressure) | 処理能力を超えた入力を上流で抑制 |
| [リトライストーム対策](./retry-storm) | Exponential Backoff + Jitter |
