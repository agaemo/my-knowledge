# タイムアウト設計

## 何か

依存先（外部API・DB・マイクロサービスなど）への呼び出しに制限時間を設け、超過したら失敗として扱う仕組み。

## なぜ必要か

タイムアウトがないと、応答しない依存先を待ち続けるスレッド・コネクションが際限なく積み重なる。
リソースが枯渇し、呼び出し元まで巻き込む[カスケード障害](../failure/cascade-failure)の引き金になる。

**「タイムアウトしない」は「無限に待つ」と同義。**

## タイムアウトの種類

| 種類 | 対象 |
|---|---|
| **接続タイムアウト** | TCP接続の確立（3ウェイハンドシェイク）にかかる時間 |
| **読み取りタイムアウト** | 接続後、レスポンスの受信完了までの時間 |
| **全体タイムアウト** | 接続〜受信完了までの合計時間 |

接続タイムアウトは短め（数秒）、読み取りタイムアウトは処理内容に応じて設定する。

## 設定値の決め方

依存先の正常時レスポンス時間の **p99〜p999 を基準に、余裕を持たせた値** にする。
平均値を基準にすると、通常の遅延変動でもタイムアウトが頻発する。

```
タイムアウト値 ≒ p99 レスポンスタイム × 2〜3
```

## タイムアウト伝播（タイムバジェット）

マイクロサービスでは呼び出しが連鎖する。
上流のタイムアウトが5秒なら、その内側の下流呼び出しはそれより短くする必要がある。

```
クライアント (5s)
  → サービスA (3s)
    → サービスB (1s)
      → DB (500ms)
```

上流のタイムアウトを超えた時点でレスポンスは捨てられるため、残り時間を下流に伝えて早期に打ち切る設計が重要。

## サーキットブレーカーとの関係

タイムアウトは個々の呼び出しを保護する。
[サーキットブレーカー](./circuit-breaker)はタイムアウトが続いた場合に呼び出し自体を止める。

両者は補完関係にあり、セットで使うのが基本。
