# サーキットブレーカー

## 何か

障害のある依存先への呼び出しを一時的に遮断するパターン。
電気回路のブレーカーと同じ発想で、過電流（障害）を検知したら回路を開いて保護する。

マイクロサービス間の通信や外部API呼び出しに適用する。

## なぜ必要か

依存先が遅延・障害中のとき、タイムアウトまで待ち続けるスレッドが積み重なりリソースが枯渇する。
ブレーカーで早期に失敗を返すことで、呼び出し元を守り[カスケード障害](../failure/cascade-failure)を防ぐ。

## 3つの状態

```
         エラー率が閾値超過                  タイムアウト経過
  Closed ─────────────────→ Open ────────────────────→ Half-Open
    ↑                                                       │
    │              成功                                     │
    └───────────────────────────────────────────────────────┘
                   失敗 → Open に戻る
```

| 状態 | 動作 |
|---|---|
| **Closed**（通常） | リクエストをそのまま通す。エラー率を監視 |
| **Open**（遮断中） | リクエストを即座に失敗させる。依存先を呼ばない |
| **Half-Open**（試験中） | 一部のリクエストを通して回復を確認する |

## 設定パラメータ

- **エラー率の閾値**：Openに移行する失敗率（例: 50%）
- **ウィンドウサイズ**：閾値を判定するリクエスト数・時間
- **Openの継続時間**：Half-Openに移行するまでの待機時間
- **Half-Openの試験数**：回復確認に使うリクエスト数

## フォールバックとの組み合わせ

Open状態で即失敗させるだけでなく、キャッシュ返却・デフォルト値返却・別エンドポイントへの切り替えを組み合わせると、ユーザー影響を最小化できる。
