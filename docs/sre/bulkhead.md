# バルクヘッド（Bulkhead）

## 何か

障害の影響をコンポーネント間で隔離し、波及を防ぐパターン。
船の隔壁（bulkhead）から命名。1区画が浸水しても他の区画が無事なら沈まない。

## なぜ必要か

リソース（スレッド・コネクション・メモリ）を共有していると、1つの機能の障害が全体のリソースを食い潰す。
バルクヘッドはリソースを機能・テナント・用途ごとに分離し、障害の影響範囲を限定する。

## 実装パターン

### スレッドプール分離

機能ごとに独立したスレッドプールを割り当てる。

```
決済 API 呼び出し    → スレッドプール A（最大10スレッド）
在庫 API 呼び出し    → スレッドプール B（最大10スレッド）
通知 API 呼び出し    → スレッドプール C（最大5スレッド）
```

決済APIがタイムアウトしてスレッドプールAを使い果たしても、在庫・通知は影響を受けない。

### コネクションプール分離

DBやキャッシュへのコネクションを用途別に分割する。
読み取り専用クエリと書き込みクエリで別プールにするのも一形態。

### プロセス・サービス分離

最も強い隔離。クリティカルな機能を別プロセス・別サービスに切り出し、ノイジーネイバー問題を根本から排除する。

## サーキットブレーカーとの違い

| | バルクヘッド | サーキットブレーカー |
|---|---|---|
| 対象 | リソースの枯渇 | 障害のある依存先への呼び出し |
| 効果 | 被害範囲の限定 | 連鎖的な失敗の遮断 |
| 組み合わせ | 補完関係 — 両方使うのが基本 |

## トレードオフ

分離するほど障害耐性は上がるが、リソースの利用効率が下がる。
アイドル時のスレッドプールは無駄になる。用途の重要度とトラフィック量に応じて設計する。
