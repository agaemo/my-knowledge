# エラーバジェット

## 何か

SLOから導かれる「許容できる障害・ダウンタイムの残量」。信頼性への投資と新機能リリースのトレードオフを議論するための共通指標。

```
エラーバジェット = 1 - SLO

例：SLO 99.9% → 月に約44分のダウンタイムを許容できる
```

## なぜ存在するか

開発チームと運用チームの対立は「リリースしたい vs 安定させたい」という目標の不一致が原因だった。
エラーバジェットはこの議論を「どちらが正しいか」ではなく「バジェットが残っているか」という定量的な判断に変える。

## 使い方

バジェットの残量によってチームの行動指針が変わる。

| 状態 | 意味 | 行動 |
|---|---|---|
| バジェット残あり | SLOの範囲内で動いている | 積極的にリリース・実験できる |
| バジェット残わずか | SLOの限界に近い | リリース頻度を下げ、信頼性改善を優先 |
| バジェット枯渇 | SLOを下回った | 新機能をフリーズし、信頼性改善に集中 |

## エラーバジェットポリシー

バジェット枯渇時に何をするかをあらかじめ決めておくのがエラーバジェットポリシー。
チームが感情的な判断ではなく事前に合意したルールに従って動けるようにする。

典型的な内容：
- バジェットが50%を下回ったら週次でレビューする
- バジェットが枯渇したら新機能のリリースを停止する
- バジェットが枯渇した原因となったインシデントのポストモーテムを必須にする

## SLOとの関係

エラーバジェットはSLOの裏返しであり、SLOなしには成立しない。

```
SLO を高く設定しすぎる → バジェットが少なくなりリリースが遅くなる
SLO を低く設定しすぎる → バジェットが多くなるがユーザー体験が悪化する
```

SLOは「100%に近いほどよい」ではなく、ビジネスとユーザー体験のバランスから決める。

→ [SLI / SLO / SLA](/sre/slo-sli-sla)
