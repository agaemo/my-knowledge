# SLI / SLO / SLA

## 何か

サービスの信頼性を定量的に管理するための3層の概念。SREの中核をなす。

| 用語 | 意味 |
|---|---|
| **SLI**（Service Level Indicator） | 実際に測定する指標 |
| **SLO**（Service Level Objective） | SLIに対して定める目標値 |
| **SLA**（Service Level Agreement） | 顧客との契約。違反時のペナルティを伴う |

## SLI（何を測るか）

サービスの健全性を表す定量的な測定値。

典型的なSLI：
- **可用性**：成功レスポンス数 / 全リクエスト数
- **レイテンシ**：リクエストの99パーセンタイル応答時間
- **スループット**：単位時間あたりの処理リクエスト数
- **エラー率**：5xx レスポンスの割合

平均値より**パーセンタイル（p99・p999）**の方が実態を反映しやすい。

## SLO（どこまで許容するか）

SLIに対する目標値。チームが守るべき内部の約束。

例：
- `可用性 ≥ 99.9%（30日ローリング）`
- `p99 レイテンシ < 500ms`

**100%のSLOは設定しない。** 達成不可能なうえ、イノベーションを妨げる。

## エラーバジェット

`エラーバジェット = 1 - SLO`

SLO 99.9% なら月に約44分の許容ダウンタイムがある。
これを「信頼性への投資」と「新機能リリース」のバランスを議論する共通言語として使う。

- バジェット残あり → 積極的にリリース・実験できる
- バジェット枯渇 → 新機能をフリーズし、信頼性改善に集中

## SLA（顧客との契約）

SLOより緩い値を設定するのが一般的。違反すると返金・ペナルティが発生する。
SLO と SLA の間にバッファを設けることで、内部対応の猶予を確保する。

```
現実 < SLO（内部目標）< SLA（契約）
```

## なぜ重要か

「信頼性を上げるほどよい」は正しくない。信頼性にはコストがかかる。
SLO/エラーバジェットを使うことで、信頼性の水準を組織として合意し、エンジニアリングの優先度を定量的に判断できる。
