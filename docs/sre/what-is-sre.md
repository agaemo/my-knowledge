# SREとは何か

## 何か

**Site Reliability Engineering**。ソフトウェアエンジニアリングの手法でシステムの信頼性を設計・維持するアプローチ、およびそれを担うロール。

2003年にGoogleで生まれ、*Site Reliability Engineering*（O'Reilly、2016年）として公開されたことで業界全体に広まった。

## なぜ生まれたか

2000年代初頭、ソフトウェア開発の現場には構造的な対立があった。

**開発チーム（Dev）** の目標は「新機能を速くリリースすること」。  
**運用チーム（Ops）** の目標は「システムを安定させること」。

運用チームは変化を嫌う。変化こそが障害の原因だからだ。開発チームは変化を求める。変化こそが価値だからだ。この対立は多くの組織で「開発 vs 運用」という文化的断絶を生んだ。

GoogleのVPエンジニアリングである **Ben Treynor Sloss** は「ソフトウェアエンジニアが運用チームを設計したらどうなるか」という問いを立て、これを実践した。それがSREの起源となった。

核心となる考え方は「**運用の問題をソフトウェアエンジニアリングで解く**」こと。人が手作業でやっていることをコードに置き換え、障害を個人の失敗ではなくシステムの問題として扱う。

## SREが解決した問題

### 「人が頑張る」運用の限界

従来の運用は属人的だった。ベテランが経験で障害を察知し、手順が口頭で伝わり、特定の人がいないと夜中の障害に対応できない。サービスが10倍になっても人を10倍雇えるわけではなく、スケールしない。

SREはこれをコードと自動化で解く。監視・デプロイ・障害対応の手順をすべてコードにする。

### 障害が「誰かのせい」になる文化

従来の運用では障害が起きると犯人を探す。犯人が見つかると「気をつけます」で終わり、同じ障害が繰り返される。

SREは障害をシステムの問題として扱う。「誰がミスをしたか」ではなく「なぜシステムがそのミスを防げなかったか」を問う。これが**ポストモーテム文化**の核心だ。

### 開発と運用の目標が噛み合わない

SREは**エラーバジェット**という概念でこれを解決した。「このサービスは月に43分まで落ちていい（SLO 99.9%）」と定める。バジェットが余っているなら積極的にリリースできる。バジェットを使い切ったら新機能より信頼性改善に集中しなければならない。「開発 vs 運用」の対立が、共通の数値を使った議論に変わる。

## SREとDevOpsの違い

| | DevOps | SRE |
|---|---|---|
| 何か | 文化・哲学・運動 | 具体的な実装方法・ロール |
| 目標 | 開発と運用の壁をなくす | 信頼性をエンジニアリングで担保する |
| 起源 | 業界全体のムーブメント | Google発の具体的プラクティス |
| 関係 | 抽象的な原則 | DevOpsの原則を具体化したもの |

## トイル（Toil）

SREが特に重視する概念。**手作業・繰り返し・自動化できるのにしていない作業**のことを指す。

以下に当てはまる作業がトイルであり、自動化の優先候補になる：

- 手作業で行っている
- 繰り返し発生する（週次・月次・障害のたびなど）
- 自動化できる（技術的に不可能ではない）
- サービスの成長に比例して増える（スケールしない）

トイルへの対応が業務の50%を超えたら問題と見なす。

## 自動化の段階

トイルの自動化は一気には進まない。段階的に進化する。

| 段階 | 状態 | 特徴 |
|---|---|---|
| 1 | 手動対応 | 手順書すら存在せず、担当者の記憶と経験に依存 |
| 2 | 手順書による標準化 | Runbookが存在し誰でも同じ手順で対応できる |
| 3 | 部分的な自動化 | スクリプト化された手順を人がトリガーする |
| 4 | 完全な自動化 | 人の介入なしに問題が解決され、事後通知が届く |
| 5 | 自律的なシステム | 問題が起きる前にシステムが予測して対処する |

多くのシステムはStage 3〜4の間にある。すべてを一気にStage 5にしようとせず、段階的に進めることが重要。
