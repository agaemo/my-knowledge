# バックプレッシャー

## 何か

処理能力を超えた入力が来たとき、その過負荷を上流に伝えてレートを抑制するメカニズム。
水道管の「背圧」から命名。

単にリクエストを捨てる（ロードシェディング）とは異なり、上流に「遅くして」と伝える点が特徴。

## なぜ必要か

入力を無制限に受け入れると：
- キューがメモリを使い果たす
- レイテンシが際限なく増大する（キューイング遅延）
- やがてOOMクラッシュやGCストームが発生する

バックプレッシャーにより、システムが許容範囲内で動作し続けることを保証する。

## 実装パターン

### 同期的バックプレッシャー

呼び出し元をブロック（待機させる）ことで自然にレートを制限する。
DBコネクションプールのサイズが上限に達したら新規取得をブロックする仕組みがこれに当たる。

### キューと容量制限

有界キュー（サイズ上限あり）を使い、満杯になったら上流に失敗を返す。
無界キューはバックプレッシャーを持たない。

### Reactive Streams / Flow Control

RxJava・Project Reactor・Kotlin Flow などのリアクティブライブラリはバックプレッシャーを仕様として持つ。
ダウンストリームが `request(n)` で「n件処理できる」と上流に通知し、それ以上流れてこないようにする。

### HTTP/2 フロー制御

HTTP/2 はプロトコルレベルでウィンドウサイズによるフロー制御を持つ。
受信側のバッファが減ると送信側の送出量が自動的に抑制される。

## ロードシェディングとの使い分け

| | バックプレッシャー | ロードシェディング |
|---|---|---|
| 方針 | 上流を遅くする | リクエストを捨てる |
| 向いている場面 | 上流が制御可能な内部システム | 外部クライアントからの流入 |
| トレードオフ | 上流が詰まる可能性 | 一部ユーザーへの明示的な拒否 |

実際は両方を組み合わせて使う。内部では backpressure、外部向けには rate limiting + 503 返却。
