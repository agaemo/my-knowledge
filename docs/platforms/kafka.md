# Apache Kafka

イベントストリーミングプラットフォーム。大量のリアルタイムデータを耐久性を持って発行・保存・処理するために設計されている。

## なぜ存在するか

従来のメッセージキューはプロデューサーとコンシューマーが密結合になりやすく、スケールや再処理が難しかった。Kafkaはプロデューサーとコンシューマーを完全に分離することで、同一イベントを複数のコンシューマーが独立して何度でも読み取れるようにした。

## いつ使うか

- 複数のシステム間でリアルタイムにデータを連携したい
- 高スループット・低レイテンシのイベント処理が必要
- イベントの永続保存と再処理が必要
- マイクロサービス間の非同期通信基盤を作りたい

## 主要コンポーネント

### トピック

イベントを分類・保存する単位。ファイルシステムのフォルダに相当する。複数のプロデューサーとコンシューマーが同時にアクセスできる。

### パーティション

トピックを分割した「バケット」。水平スケールを実現する核心的な設計。同一キーを持つイベントは同じパーティションに書き込まれるため、順序が保証される。

### ブローカー

Kafkaクラスタを構成するサーバー。データの保存とメッセージ配信を担当する。

### レプリケーション

データを複数のブローカーに複製して耐障害性を確保する。本番環境ではレプリケーションファクター3が標準。

## コアAPI

| API | 役割 |
|-----|------|
| Producer API | イベントをトピックに発行する |
| Consumer API | トピックのイベントを購読・処理する |
| Kafka Streams API | ストリーム処理アプリケーションを構築する |
| Kafka Connect API | 外部システム（DB・ファイル等）と双方向に連携する |
| Admin API | トピック・ブローカーの管理操作 |

## 設計上の重要な特性

- **プロデューサー/コンシューマーの分離**: 互いに存在を知らない。コンシューマーの処理速度に関係なく発行できる
- **オフセット管理**: コンシューマーはどこまで読んだかをオフセットで管理する。障害後の再起動や再処理が容易
- **イベントの保持期間**: メッセージキューと違い、消費後もイベントが削除されない。保持期間を設定して過去に遡った処理が可能

## 主な用途

- 金融取引のリアルタイム処理
- IoTデバイス・センサーデータの収集と分析
- ログ・監査証跡の集約
- マイクロサービス間のイベント駆動連携
- データパイプライン（ETL）の非同期化
