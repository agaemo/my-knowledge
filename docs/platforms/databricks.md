# Databricks

## なぜ生まれたか

大規模データ処理の標準である Apache Spark は強力だが、クラスター構築・管理・チューニングが複雑で、データエンジニア以外には扱いにくかった。また、データエンジニアリング（ETL）・機械学習・SQL 分析がそれぞれ別のツールで行われ、データのサイロ化と重複管理が問題になっていた。

Databricks は Spark の開発者（UCバークレー AMPLab チーム）が創業し、これらの課題をマネージドプラットフォームとして解決するために設計された。

## 何ができるか

| 用途 | 内容 |
|------|------|
| データエンジニアリング | テラバイト以上の ETL パイプライン。Spark ジョブをノートブックや Workflow で管理 |
| 機械学習 | MLflow による実験管理・モデル管理。Feature Store でのフィーチャー共有 |
| SQL 分析 | Databricks SQL（旧 SQL Analytics）でデータアナリストが直接クエリ |
| ストリーミング | Structured Streaming による準リアルタイム処理 |

## Delta Lake

Databricks が OSS として開発したストレージフォーマット。データレイク（S3・Azure Blob・GCS）の上に ACID トランザクションを持たせる。

- **なぜ必要か**: 通常の Parquet/CSV はトランザクションを持たないため、書き込み中の読み取りや失敗時の中途半端な状態が残る
- **提供するもの**: ACID 保証・スキーマ管理・タイムトラベル（過去バージョンへの参照）・Upsert（MERGE 操作）

## Snowflake との比較

両者はしばしば競合として比較される。

| | Databricks | Snowflake |
|-|-----------|-----------|
| 強み | ML・複雑な変換・Spark 資産の活用 | SQL 分析・データウェアハウス・BI ツール連携 |
| 弱み | SQL ユーザーには学習コストが高い | ML ワークロードは後付け感がある |
| 向いている組織 | データエンジニア・ML エンジニアが主体 | データアナリスト・BI チームが主体 |

## アーキテクチャの特徴

クラウドネイティブ設計で AWS・Azure・GCP 上で動作する。「コントロールプレーン（Databricks 管理）」と「データプレーン（ユーザーのクラウドアカウント）」に分離されており、データ自体はユーザーのクラウドストレージに留まる設計になっている。

## いつ選ぶか

- データ量がテラバイト以上でバッチ処理のスループットが課題になっている
- ETL パイプラインと ML モデル学習を同じ基盤で管理したい
- 既存の Spark ジョブをクラウドに移行したい

小〜中規模のデータ量（数十 GB 以下）であれば、BigQuery や Snowflake の方がシンプルで導入コストが低い。
