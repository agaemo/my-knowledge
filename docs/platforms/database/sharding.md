# シャーディング（Sharding）

データを複数のデータベースノードに分散して保存する水平スケーリングの手法。

## なぜ必要になるか

単一の DB サーバーはハードウェアの限界がある。ストレージ・CPU・ネットワーク帯域のどれかが先に詰まる。

**垂直スケール（スケールアップ）** でサーバーのスペックを上げることは最初の対策として有効だが、上限がある。コストも指数関数的に増える。

```
垂直スケール:
  [DB サーバー 4コア 32GB] → [DB サーバー 64コア 512GB]
  ↑ 限界がある。コストが急増する

水平スケール（シャーディング）:
  [DB ノード A] + [DB ノード B] + [DB ノード C]
  ↑ ノードを追加することで容量を増やせる
```

## 仕組み

データを「シャードキー」と呼ぶ値に基づいて複数のノードに分割する。

```
シャードキー: user_id

user_id 1〜100万     → シャード A
user_id 100万〜200万 → シャード B
user_id 200万〜300万 → シャード C
```

クエリ時はシャードキーを元に「どのノードに問い合わせるか」を決定する。アプリケーションかミドルウェアがこのルーティングを担う。

## シャーディング方式

### レンジシャーディング（Range-based）

値の範囲でデータを分割する。上の例がこれにあたる。

```
日付でシャードする例:
  2023年のデータ → シャード A
  2024年のデータ → シャード B
  2025年のデータ → シャード C
```

特定範囲のスキャンが速い。ただし新しいデータが特定シャードに集中しやすい（ホットスポット問題）。

### ハッシュシャーディング（Hash-based）

シャードキーのハッシュ値でデータを分散する。

```
hash(user_id) % 3 = 0 → シャード A
hash(user_id) % 3 = 1 → シャード B
hash(user_id) % 3 = 2 → シャード C
```

データが均等に分散されるためホットスポットが起きにくい。一方で範囲スキャンは全シャードを見る必要がある。

## シャーディングの課題

シャーディングは「スケール」と「複雑さ」のトレードオフ。

**クロスシャードクエリ：** 複数のシャードにまたがる JOIN・集計は各シャードに問い合わせて結果をアプリ側でまとめる必要がある。パフォーマンスが落ちる。

**分散トランザクション：** 1つのトランザクションが複数シャードに書く場合、ACID を保証するのが難しい。2フェーズコミットなどの仕組みが必要になる。

**シャードの再分割：** データが増えてシャードを追加するとき、既存データを再配置（リシャーディング）する必要がある。ダウンタイムや高コストが伴う。

**運用の複雑さ：** 複数 DB サーバーの管理・監視・バックアップが単一 DB より格段に複雑になる。

## アプリ管理 vs 自動シャーディング

### アプリケーション側でシャードを管理する

開発者がシャードキーとルーティングロジックを実装する。柔軟だが複雑さが増す。

```
# アプリ側でシャードを決定する例
def get_db_connection(user_id):
    shard_index = hash(user_id) % NUM_SHARDS
    return connections[shard_index]
```

### ミドルウェアで管理する

Vitess（MySQL）・Citus（PostgreSQL）などのミドルウェアがシャーディングを透過的に処理する。アプリは単一 DB に接続しているように書ける。

### NewSQL で自動化する

[NewSQL](/platforms/database/newsql)（CockroachDB・TiDB など）は自動シャーディングを組み込みで提供する。アプリ側でシャードを意識せず、DB が自動でデータを分散・再配置する。分散トランザクションも ACID で保証される。

```
シャーディングの進化:
  手動管理（複雑） → ミドルウェア → NewSQL（自動・ACID 保証）
```

## いつシャーディングを検討するか

```
検討するサイン:
  ✓ 単一 DB のディスクが上限に近づいている
  ✓ 書き込みのスループットがボトルネックになっている
  ✓ 垂直スケールのコストが現実的でなくなってきた

まず試すこと:
  1. インデックスの最適化・クエリチューニング
  2. 読み取りレプリカの追加（読み込みを分散）
  3. キャッシュ層の導入
  → それでも足りなければシャーディングを検討する
```

シャーディングは強力だが、引き返しにくい設計変更を伴う。本当に必要になるまで導入しないのが原則。
