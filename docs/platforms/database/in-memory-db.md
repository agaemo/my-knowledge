# インメモリDB

## なぜ存在するか

ディスクベースのDBはデータをストレージに永続化するため、読み書きにI/Oコストがかかる。
インメモリDBはデータをRAM上に保持することで、**ディスクI/Oを排除して極めて高速なアクセスを実現する**。

トレードオフとして、プロセスが落ちればデータが消える（揮発性）。
この特性を逆手に取り、「速さが必要で永続化は必須でない」用途に特化して使われる。

## 代表: Redis

最も広く使われるインメモリデータストア。単純なKVSを超え、多様なデータ構造と機能を持つ。

### データ構造

```
String  → キャッシュ・カウンター・セッション
List    → キュー・最新N件のログ
Set     → タグ・ユニーク集合
Sorted Set → ランキング・スコア付きデータ
Hash    → オブジェクト（ユーザー情報など）
Stream  → イベントログ・メッセージキュー
```

### 主な用途

**キャッシュ**

DBへの重いクエリ結果を一時保存し、同じリクエストを高速に返す。

```
GET /products/123
  → Redis に hit → 即返却（数百μs）
  → Redis に miss → DB に問い合わせ → Redis に保存 → 返却
```

```bash
SET product:123 '{"name":"..."}' EX 300   # 300秒で自動失効
GET product:123
```

**セッション管理**

複数サーバーでセッションを共有するためのセッションストア。

```bash
SET session:abc123 '{"userId":1,"role":"admin"}' EX 3600
```

**レートリミット**

一定時間内のリクエスト数をカウントする。

```bash
INCR ratelimit:user:1:2024010112    # 時間単位のキーでカウント
EXPIRE ratelimit:user:1:2024010112 3600
```

**Pub/Sub**

チャンネルへのメッセージ送受信。リアルタイム通知・チャットなどに使う。

```bash
SUBSCRIBE notifications:user:1
PUBLISH notifications:user:1 '{"type":"message","text":"Hello"}'
```

**分散ロック**

複数サーバーが同じリソースに同時アクセスしないよう制御する。

```bash
SET lock:order:456 "process-1" NX EX 10   # NX: 存在しない場合のみセット
```

### 永続化オプション

完全揮発でなく、ある程度の耐障害性が必要な場合に使える。

| 方式 | 内容 |
|---|---|
| RDB（スナップショット） | 定期的にディスクに書き出す。再起動後に復元可。直前のデータは失う |
| AOF（Append Only File） | 全書き込み操作をログに追記。RDBより復元精度が高いがファイルが大きくなる |
| なし | 最速。キャッシュ用途など消えても許容できる場合 |

## 他DBとの比較

### Redis vs MongoDB

| 観点 | Redis | MongoDB |
|---|---|---|
| データ保存場所 | RAM（オプションでディスク永続化） | ディスク |
| データモデル | KVS・多様なデータ構造 | ドキュメント（JSON） |
| 速度 | 極めて高速（μs〜ms） | 高速（ms〜） |
| 永続性 | 基本は揮発性 | 永続 |
| クエリ能力 | シンプル（キー操作中心） | 複雑なクエリ・集計が可能 |
| 向く用途 | キャッシュ・セッション・カウンター | コンテンツ管理・検索・柔軟なスキーマのデータ |

Redisは「速さと揮発性」、MongoDBは「柔軟なスキーマと永続化」。併用するケースが多い。

### Redis vs RDB（PostgreSQL など）

| 観点 | Redis | RDB |
|---|---|---|
| 速度 | 極めて高速 | 相対的に遅い（ディスクI/O） |
| 永続性 | 基本は揮発性 | 永続 |
| トランザクション | 限定的 | ACID保証 |
| 向く用途 | キャッシュ・一時データ | 業務データ・複雑なリレーション |

RDBの前段にRedisを置き、**キャッシュ層として負荷を吸収する**構成が定番。

### Redis vs Memcached

| 観点 | Redis | Memcached |
|---|---|---|
| データ構造 | 多様（String・List・Set・Hash など） | String のみ |
| 永続化 | 対応 | 非対応 |
| Pub/Sub | 対応 | 非対応 |
| クラスタ | Redis Cluster で対応 | 対応 |
| 向く用途 | 多機能・汎用 | シンプルなキャッシュ専用 |

単純なキャッシュのみならMemcachedでも十分だが、現在はRedisが事実上の標準。

## いつ使うか

- DBへの同じクエリが繰り返し発行されてパフォーマンスが問題になっている（キャッシュ）
- 複数サーバー間でセッションを共有したい
- リアルタイムのカウンター・ランキングが必要
- 非同期処理のキューやPub/Subが必要
- レートリミットの実装

**単独では使わない**。永続DBと組み合わせ、速度が必要な部分だけRedisに任せるのが基本。
