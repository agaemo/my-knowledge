# RDBMS（リレーショナルDB）

## なぜ存在するか

業務システムでは「データの整合性」が最重要課題だった。例えば、銀行の送金処理では「口座Aの残高を減らし、口座Bの残高を増やす」という2つの操作が**必ず両方成功するか、両方失敗するか**でなければならない。どちらか一方だけ成功するとデータが壊れる。

RDBMSはこの問題をトランザクション（ACID保証）で解決した。さらに、データを表（テーブル）として構造化し、テーブル間の関係（リレーション）をSQL で表現できるようにした。

## ACID保証

| 特性 | 意味 |
|---|---|
| **Atomicity（原子性）** | トランザクション内の操作はすべて成功するか、すべて失敗するか（部分的な状態にならない） |
| **Consistency（一貫性）** | トランザクション前後でデータは常に整合した状態を保つ |
| **Isolation（独立性）** | 並行して実行されるトランザクションは互いに干渉しない |
| **Durability（永続性）** | コミットされたデータはシステム障害後も失われない |

## 行志向ストレージ

RDBMSはデータを行単位でディスクに格納する。

```
ID | 名前   | 年齢 | 部署
1  | 田中   | 30   | 営業
2  | 佐藤   | 25   | 開発
3  | 鈴木   | 35   | 営業
```

1行分のデータがまとまって格納されているため、**特定の1件を取得・更新するOLTPに最適**。ただし「全行の年齢の平均を出す」ような集計は、不要なカラムも読み込むため非効率になる。

## 正規化とリレーション

データの重複を排除し、整合性を保つための設計手法。

```sql
-- 非正規化（部署名を直接持つ）: 部署名が変わると全行を更新する必要がある
users(id, name, department_name)

-- 正規化（外部キーで参照）: 部署テーブルを1か所変更するだけで済む
users(id, name, department_id)
departments(id, name)
```

外部キー制約により、存在しない `department_id` を登録しようとするとDBがエラーを返す（参照整合性）。

## 代表的なRDBMS

| DB | 特徴 |
|---|---|
| **PostgreSQL** | 高機能・拡張性が高い。JSONサポート・全文検索など。デファクト標準 |
| **MySQL / MariaDB** | Webアプリで広く使われてきた。シンプルで速い |
| **SQLite** | サーバー不要の組み込み型。モバイル・小規模アプリ・テスト用途 |
| **Oracle DB** | エンタープライズ向け。高可用性・サポート重視 |

## OLTPとしての用途

RDBMS は **OLTP（Online Transaction Processing）** に分類される。特定レコードへの短い読み書きが高頻度で発生するワークロードに向く。

```
ユーザー登録・更新・削除
注文の作成・キャンセル
残高の加減算
セッション管理
```

一方、「過去1年間の売上を月別・商品カテゴリ別に集計する」ような分析クエリ（OLAP）は苦手。そのような用途には列志向の分析DBを使う。

## いつ使うか

- 複数テーブルをまたぐ更新でACIDが必要（送金・在庫管理・予約）
- データの整合性・参照整合性を厳密に保ちたい
- 複雑なJOINやサブクエリが必要
- 業務システム・ユーザーデータの主データストアとして

::: tip PostgreSQL を選ぶ理由
新規プロジェクトであれば PostgreSQL がデファクトスタンダード。JSON型・全文検索・ウィンドウ関数・拡張機能（pgvector 等）が充実しており、多くのユースケースをカバーできる。
:::
