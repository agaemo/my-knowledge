# NoSQL

## なぜ存在するか

RDBは構造化データと一貫性に強いが、大規模な水平スケール・スキーマレスなデータ・特定アクセスパターンへの最適化が苦手。
NoSQLはこれらの制約を緩めることで、特定のユースケースにおいてスケールと柔軟性を実現する。

「NoSQL = RDBの代替」ではなく、**データモデルとアクセスパターンに合わせて選ぶ**のが基本。

## データモデルの種類

### ドキュメント型

JSONライクなドキュメントをコレクションに格納する。スキーマが柔軟で、ネストしたデータを自然に表現できる。

```json
{
  "_id": "user_123",
  "name": "Alice",
  "address": {
    "city": "Tokyo",
    "zip": "100-0001"
  },
  "tags": ["premium", "active"]
}
```

- 代表: MongoDB, Firestore
- 向くケース: スキーマが頻繁に変わる・ネストしたデータを一度に取得したい・コンテンツ管理・ユーザープロフィール

### キーバリュー型

シンプルなキーとバリューのペア。極めて高速で、単純な読み書きに最適。

```
SET session:abc123 '{"userId": 1, "role": "admin"}' EX 3600
GET session:abc123
```

- 代表: Redis, DynamoDB（KVモード）
- 向くケース: セッション管理・キャッシュ・リアルタイムランキング・レートリミットのカウンター

### カラム型（ワイドカラム）

行と列の概念はあるが、列を動的に追加でき、書き込みと大量データのスキャンに強い。

```
Row Key: user_123
  → profile:name = "Alice"
  → activity:2024-01 = "{...}"
  → activity:2024-02 = "{...}"
```

- 代表: Apache Cassandra, Google Bigtable, HBase
- 向くケース: 時系列データ・ログ・IoTセンサーデータ・書き込みが多く読み取りパターンが固定

### グラフ型

ノード（エンティティ）とエッジ（関係）でデータを表現する。多対多の関係をトラバースするクエリが高速。

```cypher
// Cypher (Neo4j)
MATCH (u:User)-[:FOLLOWS]->(f:User)-[:LIKES]->(p:Post)
WHERE u.id = 'alice'
RETURN p
```

- 代表: Neo4j, Amazon Neptune
- 向くケース: SNSのフォロー関係・レコメンデーション・不正検出・知識グラフ

## RDBとの比較

| 観点 | RDB | NoSQL |
|---|---|---|
| スキーマ | 固定（事前定義） | 柔軟（スキーマレスが多い） |
| 一貫性 | ACID保証 | モデルによる（結果整合性が多い） |
| スケール | 垂直スケールが基本 | 水平スケールに強い |
| クエリ | 汎用SQL・JOINが強力 | アクセスパターンに特化 |
| 向くデータ | 構造化・関係が複雑 | 非構造化・大量・特定パターン |

## いつ選ぶか

- アクセスパターンが明確で、そのパターンに特化した性能が必要なとき
- スキーマが頻繁に変わる、または事前に定義できないとき
- 水平スケールが前提で、RDBのシャーディング複雑さを避けたいとき

逆に、**複雑なJOIN・トランザクション・集計が必要なら RDB の方が適している**。
