# ACID とトランザクション

データベースにおけるトランザクションの信頼性を保証する4つの性質。

## トランザクションとは

複数の読み書き操作をひとまとまりとして扱う単位。「全部成功するか、全部失敗するか」を保証する。

```
銀行送金の例:
  1. 送金元の残高を -1000 円
  2. 送金先の残高を +1000 円

1 だけ成功して 2 が失敗すると 1000 円が消える
→ 両方を原子的に扱う必要がある
```

## ACID の4性質

### Atomicity（原子性）

トランザクション内の操作はすべて成功するか、すべて失敗する。途中の状態は外から見えない。

### Consistency（一貫性）

トランザクション前後でデータはスキーマ制約・整合性制約を満たした状態を保つ。
例：外部キー制約・NOT NULL 制約を違反した状態で確定しない。

### Isolation（分離性）

並行する複数トランザクションは互いに干渉しない。あるトランザクションが見る中間状態は、別のトランザクションが変更中の状態ではない。

### Durability（永続性）

コミットされたデータはシステム障害後も失われない。WAL（Write-Ahead Logging）などで保証される。

## 分離レベル

Isolation は「どこまで他のトランザクションの変更が見えるか」を制御する。厳しいほど整合性は高いが、同時実行性が低下する。

| レベル | 読み取り対象 | 発生しうる異常 |
|---|---|---|
| Read Uncommitted | コミット前の変更も読む | Dirty Read |
| Read Committed | コミット済みのみ | Non-Repeatable Read |
| Repeatable Read | トランザクション開始時点の値 | Phantom Read |
| Serializable | 完全に直列化 | なし |

多くの RDBMS のデフォルトは **Read Committed**（PostgreSQL・SQL Server）または **Repeatable Read**（MySQL InnoDB）。

## BASE との対比

[NoSQL](/platforms/database/nosql) や分散システムは ACID の代わりに BASE を採用することがある。

| | ACID | BASE |
|---|---|---|
| 一貫性 | 強い整合性 | 結果整合性 |
| 可用性 | 制限される場合がある | 優先する |
| 主な用途 | RDBMS | 分散 KVS・ドキュメント DB |

[CAP 定理](/architecture/cap-theorem)の観点では、ACID は一貫性を優先し、BASE は可用性を優先する設計方針。

## マイクロサービスでの課題

複数サービスにまたがるトランザクションは ACID が保証できない（それぞれが別の DB を持つため）。代わりに Saga パターンで補償トランザクションを設計する。
