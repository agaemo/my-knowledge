# CMS（コンテンツ管理システム）

## なぜ存在するか

Web サイトのコンテンツ（記事・画像・製品情報など）を更新するたびにエンジニアが HTML を書き換えるのは非効率で、ビジネスの速度を落とす。
CMS はコンテンツの編集・管理を非エンジニアが行えるようにし、コンテンツと実装を分離する。

## 従来型 CMS（モノリシック CMS）

コンテンツの管理・保存・レンダリングまで一体で担うシステム。WordPress・Drupal が代表例。

```
[管理画面] → [CMS（DB + テンプレートエンジン）] → [HTML を配信]
```

**特徴**:
- セットアップが速く、テーマ・プラグインで機能拡張できる
- フロントエンドとバックエンドが密結合している
- スケーリングやモダンフロントエンドとの統合が難しい

## ヘッドレス CMS

コンテンツ管理（"ボディ"）だけを担い、表示（"ヘッド"）はフロントエンドに委ねる API ファーストな設計。
Contentful・Sanity・Strapi・microCMS が代表例。

```
[管理画面] → [CMS（DB + REST/GraphQL API）] → [任意のフロントエンド]
                                                ├── Web（Next.js・Nuxt）
                                                ├── モバイルアプリ
                                                └── IoT・デジタルサイネージ
```

**特徴**:
- コンテンツを複数チャネル（Web・アプリ・メール）に同時配信できる
- フロントエンドの技術選択が自由
- 管理画面・エディタ UX はフロントエンドより高品質なものが多い
- コンテンツ変更だけでフロントエンドのビルドが走る（SSG との相性が良い）

## 従来型 vs ヘッドレス

| 観点 | 従来型 CMS | ヘッドレス CMS |
|---|---|---|
| 表示層 | CMS が管理 | フロントエンドが自由に実装 |
| 配信チャネル | Web のみ | Web・アプリ・複数媒体 |
| フロント技術 | CMS 依存（PHP テーマなど） | 自由（React・Vue・モバイルなど） |
| セットアップ速度 | 速い | フロント実装が別途必要 |
| スケーリング | 難しい | CDN・SSG で対応しやすい |
| コスト | 安価〜無料（OSS） | SaaS 型は従量課金が多い |

## コンテンツモデル設計

ヘッドレス CMS の核はコンテンツモデル（スキーマ）設計。型安全な API を生成できるものもある。

```ts
// Contentful のコンテンツモデル例（型生成後）
interface BlogPost {
  title: string;
  slug: string;
  body: Document; // Rich Text
  author: Author;
  publishedAt: string;
  tags: string[];
}
```

## SSG との組み合わせ

ヘッドレス CMS は静的サイトジェネレーター（Next.js・Astro）と組み合わせることが多い。

1. コンテンツ更新 → CMS が Webhook を送信
2. CI/CD がビルドをトリガー
3. 静的ファイルを CDN に配信

コンテンツ更新のたびに再ビルドが走るため、ページ数が増えると**ビルド時間**が課題になる。ISR（Incremental Static Regeneration）や SSR で対応する。

## いつ使うか

**従来型 CMS（WordPress など）が向いている状況**:
- エンジニア工数を最小化してコンテンツサイトを立ち上げたい
- プラグインエコシステムで機能を補いたい
- SEO・ブログ・企業サイトなど Web 単一チャネル

**ヘッドレス CMS が向いている状況**:
- コンテンツを複数チャネルに配信したい
- モダンフロントエンドフレームワーク（Next.js・Nuxt）と組み合わせたい
- マーケターがコンテンツ更新し、エンジニアが表示を独立して開発したい
- スケーラビリティ・パフォーマンスが重要な大規模サイト

**CMS が向かない状況**:
- コンテンツが動的でリアルタイム性が高い（チャット・ダッシュボード）
- コンテンツ構造が非常に複雑・カスタムで汎用エディタに合わない
