# ログ集約

## 何か

複数のサービス・サーバーから生成されるログを1か所に集めて検索・分析できるようにする仕組み。
サービスが増えると各サーバーにSSHしてログを確認するのが不可能になるため必要になる。

## 2つの設計思想

ログ集約には「ログの中身までインデックス化するか」で大きく方針が分かれる。

### フルテキストインデックス型（Elasticsearch + Kibana）

ログの**すべてのフィールドをインデックス化**する。

```
ログ: { "level": "error", "message": "DB timeout", "userId": "u123" }
         ↓ インデックス化
  level / message / userId すべて高速検索可能
```

- **メリット**: 任意のフィールドで瞬時に全文検索できる
- **デメリット**: インデックスが大きくなりストレージコストが高い。大量ログには不向き

### ラベルインデックス型（Grafana Loki）

ログ自体はインデックス化せず、**ラベル（メタデータ）だけをインデックス化**する。

出典: [Grafana Loki Overview](https://grafana.com/docs/loki/latest/get-started/overview/)

```
ラベル: { "service": "order-api", "env": "prod" }  ← インデックス化
ログ本文: "DB timeout for userId=u123"              ← 非インデックス（圧縮保存）
```

- **メリット**: インデックスが小さく、S3などオブジェクトストレージに安価に保存できる
- **デメリット**: ログ本文の検索はラベルで絞り込んだ後にgrep相当の処理が走るため遅い

## 比較

| 観点 | Elasticsearch + Kibana | Grafana Loki |
|---|---|---|
| インデックス戦略 | フルテキスト | ラベルのみ |
| ストレージコスト | 高い | 低い（S3等） |
| 検索速度 | 任意フィールドで高速 | ラベル絞り込み後にgrep |
| 向いている用途 | 複雑なクエリ・ビジネス分析 | 大量ログの低コスト保存 |
| Grafanaとの親和性 | 連携可能 | ネイティブ統合 |
| Prometheusとの親和性 | 別系統 | 設計思想が共通（ラベル） |

## ログの収集パイプライン

```
[各サービス]
    └→ [エージェント（Fluent Bit / Grafana Alloy）]
           └→ [集約サーバー（Loki / Elasticsearch）]
                  └→ [可視化（Grafana / Kibana）]
```

エージェントをサーバーにインストールし、ログをプッシュ方式で送る。

## いつ何を選ぶか

**Loki を選ぶとき**
- すでに Prometheus + Grafana でメトリクスを監視している
- ログ量が多くストレージコストを抑えたい
- ログをラベル（service・env・pod名など）で絞り込む使い方が中心

**Elasticsearch を選ぶとき**
- ログの中身（エラーメッセージ・ユーザーIDなど）で自由に全文検索したい
- ログをビジネス分析・監査ログとしても活用する
- 検索速度が重要

**マネージドサービスを選ぶとき**
- AWS → CloudWatch Logs
- GCP → Cloud Logging
- 全部まとめて → Datadog

自前でログ基盤を運用するコストが高いため、クラウドに乗っているならマネージドサービスが第一候補になることが多い。
