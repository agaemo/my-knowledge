# Zabbix

## 何か

インフラ・ネットワーク機器・サーバーの**死活監視・閾値監視**に特化したオープンソースの監視プラットフォーム。
「サーバーが落ちていないか」「CPU・メモリが限界に近づいていないか」を継続的に監視し、異常時にアラートを発する。

## なぜ存在するか

アプリケーションのオブザーバビリティ（Prometheus + Grafana 等）は、アプリ自身がメトリクスを公開することを前提にしている。
しかし、ルーター・スイッチ・ストレージ・古いサーバーは自らメトリクスを公開しない。
Zabbix はこれらを**エージェントや SNMP でポーリング**することで、アプリ側の改修なしに監視できる。

## Prometheus との違い

| 観点 | Zabbix | Prometheus |
|---|---|---|
| 収集方式 | Pull（ポーリング）・Push（Zabbix送信側） | Pull（スクレイピング） |
| 対象 | インフラ・NW機器・OS | アプリケーション・コンテナ |
| 設定方法 | Web UI からGUIで設定 | YAML ファイル |
| データ保持 | 内蔵DB（MySQL / PostgreSQL） | TSDB（時系列DB） |
| 得意なこと | SNMP・エージェントレス監視・長期保存 | 高頻度収集・ラベルベースのクエリ（PromQL） |
| 苦手なこと | 動的なコンテナ環境のスケール追従 | NW機器・エージェントレス監視 |

## いつ使うか

- オンプレミスのサーバー・ネットワーク機器を監視したい
- 監視対象がメトリクスエンドポイントを公開できない（SNMP のみ対応等）
- 開発者よりもインフラ・運用チームが主体で管理する
- 商用サポート付きの OSS が必要（Zabbix 社が提供）

## いつ使わないか

- Kubernetes 上のマイクロサービス監視 → Prometheus + Grafana の方が向いている
- 高頻度（秒単位）のメトリクス収集・PromQL での分析 → Prometheus
- クラウドネイティブ環境全体をカバーしたい → Datadog / CloudWatch

## アーキテクチャ

```
[監視対象]                    [Zabbix]
 ├── Zabbix Agent ──────────→  Zabbix Server
 ├── SNMP デバイス ─── SNMP →  Zabbix Server
 └── ICMP（ping）──────────→  Zabbix Server
                                    │
                              Zabbix DB（MySQL等）
                                    │
                              Zabbix Web（管理UI）
```

- **Zabbix Agent**: 監視対象サーバーにインストールするデーモン。CPU・メモリ・ディスク等を収集
- **SNMP 監視**: エージェントを入れられないNW機器に対して SNMP プロトコルで情報取得
- **エージェントレス**: ICMP（死活）・SSH・Telnet などでエージェントなしに監視も可能

## Zabbix のアラート設計

### トリガー

条件式でアラート発火を定義する。

```
{host:system.cpu.util[,idle].avg(5m)} < 10
→ CPU 空きが 5分平均で 10% を下回ったら発火
```

### アクション

トリガー発火時の動作を定義。メール・Slack・スクリプト実行などを設定できる。

## Prometheus + Grafana との併用

Zabbix と Prometheus は排他ではなく、役割分担して共存させるケースが多い。

```
インフラ層（サーバー・NW機器）  → Zabbix で監視
アプリ層（コンテナ・サービス）   → Prometheus + Grafana で監視
```

それぞれの得意領域を活かし、アラートは PagerDuty / OpsGenie などで統合する。
