# QA エンジニア

ソフトウェアの品質を保証するための活動に専念するロール。「テストをする人」という理解は正しいが、それは仕事の一部に過ぎない。

## 「テスターとの違い」という問い

かつてのソフトウェア開発では、テスターはリリース前に動作確認をするだけの役割だった。QA（Quality Assurance）エンジニアはこれとは異なる。

| | テスター（旧来） | QA エンジニア |
|---|---|---|
| 関与タイミング | 開発完了後 | 要件定義から関与 |
| 仕事 | 動作確認・バグ報告 | 品質設計・テスト戦略・プロセス改善 |
| 視点 | 「バグを見つける」 | 「バグが入り込まない仕組みを作る」 |
| 成果 | バグの発見数 | リリース後の障害率・品質指標の改善 |

QA エンジニアの本質は「品質を後から検査する」ではなく、**「品質をプロセスに組み込む」**こと。

## なぜ必要か

「開発者がテストを書けばいい」という考え方は部分的に正しいが、限界がある。

**認知バイアス：** 実装した人は、自分の思い込みと同じ方向でテストを書きがち。想定外の使い方・エッジケースを見落としやすい。

**視点の違い：** 開発者はシステムの内側（実装）から考える。QA は外側（ユーザーの行動・ビジネスルール）から考える。この2つの視点が揃うことでカバレッジが広がる。

**トレードオフの調整：** 開発者は「速く動くものを作る」インセンティブが強い。QA は「リリースしてよい状態かどうか」を独立して判断する役割を担う。

**テスト戦略の設計：** 何を・どこまで・どの順番でテストするかは専門的な判断が必要。テスト設計の質が低いと、テストが増えても品質が上がらない状況が起きる。

## QA エンジニアの仕事

### テスト設計

「どのようにテストするか」を設計する。境界値・同値分割・デシジョンテーブルなどの技法を使い、少ないケースで多くのリスクをカバーするテスト仕様を作る。

### リスク分析

すべてをテストすることはできない。「どこが壊れたら最もインパクトが大きいか」「変更の影響が波及しやすい箇所はどこか」を分析し、テストの優先度と深度を決める。

### テスト自動化

リグレッションテスト（変更によって既存機能が壊れていないかの確認）を自動化し、CI/CD に組み込む。テストコードを書くため、プログラミングスキルが必要。

### 品質指標の設計・可視化

バグ発見率・リリース後の障害件数・テストカバレッジ・MTTR（平均復旧時間）などを指標化し、チームが品質の現状を把握できるようにする。

### プロセス改善

バグが入り込むパターンを分析し、コードレビューのガイドライン整備・設計段階での品質チェックリスト導入など、上流でバグを防ぐ仕組みを整える。

## QA の変遷

```
ウォーターフォール時代:
  [設計] → [開発] → [QA フェーズ] → [リリース]
  QA はリリース前の最後のゲート。問題が見つかると手戻りが大きい。

アジャイル時代:
  各スプリントに QA が参加し、都度テストする。
  開発チームに1〜2名埋め込む「Embedded QA」が主流になる。

現在:
  「Shift Left」— テストを左（上流）にシフトする考え方が広まる。
  要件定義・設計の段階から QA が関与し、バグが入り込む前に止める。
```

## SDET（Software Development Engineer in Test）

QA の専門化が進んだ形。プログラミングスキルが高く、テストフレームワーク・CI/CD パイプラインの構築・テスト基盤の整備を担う。

| | QA エンジニア | SDET |
|---|---|---|
| 強み | テスト設計・品質プロセス | テスト自動化・ツール開発 |
| 主な仕事 | 探索的テスト・品質改善・リスク分析 | テストフレームワーク構築・CI 組み込み |
| ポジション | 多くのチームに埋め込まれる | 共通基盤として機能する |

Google が SDET という呼称を普及させた。実際の現場では QA エンジニアと SDET の境界は曖昧で、両方をこなす人も多い。

## チームへの組み込み方

**独立した QA チーム：** QA がサービスを横断して品質を管理する。開発チームから独立しているため客観性は高いが、コンテキストのキャッチアップに時間がかかる。

**Embedded QA（埋め込み型）：** QA エンジニアが特定の開発チームに所属する。チームのコンテキストを深く理解でき、要件定義からテスト設計に関与しやすい。現在の主流。

どちらが良いかは組織規模・プロダクトの複雑さ・QA の人数による。少人数なら Embedded、大規模なら横断チームとの組み合わせが現実的。

## いつ採用を検討するか

```
QA エンジニアが必要なサイン:
  ✓ リリース後に本番障害・ユーザー報告バグが繰り返し起きている
  ✓ 開発者がテストを書く時間が取れず、品質への自信が下がっている
  ✓ 機能追加のたびに既存機能が壊れる（リグレッション）が頻発している
  ✓ チームが「何をどこまでテストすべきか」判断できていない
```

開発者が多いチームに QA を1名加えるだけで、バグ発見の視点・テスト自動化の基盤・品質プロセスの整備が一気に進むことが多い。
