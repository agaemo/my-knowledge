# ミューテーションテスト

## なぜ存在するか

カバレッジ率は「そのコードがテストで通過したか」しか測れない。アサーションが弱かったり欠けていたりしても、カバレッジは 100% になる。

ミューテーションテストは、テストコードではなく**プロダクションコードに意図的なバグを埋め込み**、テストがそれを検出できるか確認する手法。テストの品質そのものを評価する。

```
通常のカバレッジ計測:
  テスト → プロダクションコード（変更なし） → pass ✓

ミューテーションテスト:
  テスト → プロダクションコード（バグを埋め込んだ） → fail ✓（バグを検出できた）
                                                   → pass ✗（テストが弱い = 生き残りミュータント）
```

## ミュータントの例

```ts
// 元のコード
function isAdult(age: number): boolean {
  return age >= 18;
}

// ミュータント1: 条件を変える
return age > 18;   // テストが 18 を検証していれば検出できる

// ミュータント2: 演算子を変える
return age <= 18;  // テストが true/false を両方検証していれば検出できる

// ミュータント3: 戻り値を反転
return !(age >= 18);
```

各ミュータントに対してテストスイートを実行し：
- **Killed（駆除）**: テストが fail → テストはそのバグを検出できる
- **Survived（生存）**: テストが pass のまま → テストの抜け穴

## ミューテーションスコア

```
ミューテーションスコア = Killed 数 / 総ミュータント数 × 100
```

カバレッジ 100% でもミューテーションスコアが低ければ、テストの品質が不十分。

## ツール

### Stryker（JavaScript / TypeScript）

```bash
npm install --save-dev @stryker-mutator/core @stryker-mutator/jest-runner
npx stryker run
```

```json
// stryker.config.json
{
  "testRunner": "jest",
  "mutate": ["src/**/*.ts", "!src/**/*.spec.ts"],
  "thresholds": { "high": 80, "low": 60, "break": 50 }
}
```

実行後、HTML レポートで生き残ったミュータントの一覧と箇所を確認できる。

### PITest（Java）

Maven/Gradle プロジェクトで使う Java 向けの代表的なミューテーションテストツール。

## 生き残りミュータントへの対処

すべてのミュータントを駆除する必要はない。対処の判断基準：

| 状況 | 対処 |
|---|---|
| アサーションが弱い | テストに境界値・反転ケースを追加する |
| 到達不能コード | コードを削除する |
| 意図的に検証しない箇所 | ミュータントを除外設定（`@Ignore`）する |

## いつ使うか

**有効な場面**:
- カバレッジが高いのにバグが多発するとき（テストの質の診断）
- ビジネスロジックの核心部分（金額計算・権限判定）の品質保証
- テストの書き方をチームで改善したいとき

**コスト面の注意**:
- 全ファイルに適用すると実行時間が長くなる（ミュータント数 × テスト実行回数）
- CI では重要なモジュールに絞って実行するのが現実的
