# スナップショットテスト

## なぜ存在するか

UI コンポーネントや CLI の出力・シリアライズ結果は、「正しい状態」を期待値として文字列で書き下すのが難しい。

スナップショットテストは、**初回実行時の出力を「正解」として保存**し、以降の実行でその差分を検出する。意図しない変化（リグレッション）を自動で発見できる。

```
初回実行:
  コンポーネントをレンダリング → スナップショット（.snap ファイル）を生成・保存

2回目以降:
  コンポーネントをレンダリング → 保存済みスナップショットと比較
  → 差分なし: pass
  → 差分あり: fail（意図しない変化を検出）
```

## Jest でのスナップショットテスト

### React コンポーネント

```tsx
import { render } from '@testing-library/react';

test('Button コンポーネントのスナップショット', () => {
  const { container } = render(<Button label="送信" disabled={false} />);
  expect(container).toMatchSnapshot();
});
```

初回実行で生成されるスナップショットファイル：

```
// __snapshots__/Button.test.tsx.snap
exports[`Button コンポーネントのスナップショット 1`] = `
<div>
  <button
    class="btn btn-primary"
  >
    送信
  </button>
</div>
`;
```

### スナップショットの更新

意図的な変更（デザイン修正など）のあとは更新コマンドを実行：

```bash
jest --updateSnapshot
# または
jest -u
```

## インラインスナップショット

`.snap` ファイルを作らず、テストコード内にスナップショットを埋め込む。差分のレビューがしやすい。

```ts
test('ユーザーオブジェクトのシリアライズ', () => {
  const user = serialize({ id: 1, name: 'Alice' });
  expect(user).toMatchInlineSnapshot(`
    {
      "id": 1,
      "name": "Alice",
    }
  `);
});
```

## スナップショットテストの限界

```tsx
// スナップショットは変化を検出するが、変化が「正しいか」は判断しない
// ↓ このテストは常に pass する（壊れていても更新すれば通る）
test('何かが変わったら検出する', () => {
  expect(render(<MyComponent />).container).toMatchSnapshot();
});
```

**スナップショットの更新を機械的に承認しない**のがポイント。差分をレビューして、意図した変更かどうかを確認する。

## いつ使うか

**有効な場面**:
- コンポーネントの構造（DOM 構造・クラス名）の安定化
- シリアライズ・JSON 変換の出力検証
- CLI コマンドの出力検証

**向かない場面**:
- 頻繁に変わる UI（スナップショットの更新が多く、ノイズになる）
- スタイル（CSS）の検証 → ビジュアルリグレッションテストの方が適切
- ビジネスロジックの検証 → 意図を明示した通常のアサーションを使う

## ビジュアルリグレッションテストとの違い

| 観点 | スナップショットテスト | ビジュアルリグレッションテスト |
|---|---|---|
| 検出対象 | HTML 構造・テキストの変化 | 見た目（ピクセル）の変化 |
| 実行速度 | 速い | 遅い（ブラウザ起動が必要） |
| CSS の変化 | 検出できない | 検出できる |
