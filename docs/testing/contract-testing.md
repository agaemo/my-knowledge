# 契約テスト（Contract Testing）

サービス間の API インターフェース（契約）を自動的に検証するテスト手法。  
マイクロサービス環境で「統合してみたら動かなかった」を防ぐ。

## なぜ存在するか

マイクロサービスではサービス間の通信が増え、インターフェースの不整合が問題になる。

```
問題のシナリオ:
  User Service が /api/users/:id のレスポンスに email フィールドを追加
  Order Service はこれを知らず、古い型定義で動いている

  → ローカルのユニットテストは通る（モックを使っているため）
  → 統合してみて初めて壊れることに気づく
```

E2E テストで発見できるが、E2E は実行が遅く・環境依存が強く・壊れやすい。  
契約テストは「E2E より軽く・ユニットテストより現実的」な中間の検証手段。

## Consumer-Driven Contract Testing

**コンシューマー（API を呼ぶ側）が期待するインターフェースを「契約」として定義し、プロバイダー（API を提供する側）がその契約を満たしているか検証する**手法。

```
Consumer（Order Service）:
  「/api/users/:id を呼んだら id と name が返ってくることを期待する」
  → この期待を Pact ファイル（契約）として記録

Provider（User Service）:
  「Order Service の契約を満たしているか？」を自分のテストで検証する
  → 実際にリクエストを再生して、期待通りのレスポンスを返せるか確認
```

## Pact

契約テストの代表的なフレームワーク。コンシューマーが Pact ファイルを生成し、プロバイダーがそれを検証する。

```
フロー:
  1. Consumer テスト実行 → pact ファイル生成（JSON）
  2. pact ファイルを Pact Broker に publish
  3. Provider テスト実行 → Pact Broker から契約を取得して検証
  4. 検証結果を Pact Broker に記録
  5. CI でデプロイ前に「can-i-deploy」で互換性チェック
```

```typescript
// Consumer 側（Order Service）
const interaction = {
    state: 'user 123 exists',
    uponReceiving: 'a request for user 123',
    withRequest: { method: 'GET', path: '/api/users/123' },
    willRespondWith: {
        status: 200,
        body: { id: 123, name: like('Alice') }, // 型の一致を検証
    },
};
```

## ユニットテスト・E2E テストとの違い

| | ユニットテスト | 契約テスト | E2E テスト |
|---|---|---|---|
| 対象 | 単一サービスの内部 | サービス間のインターフェース | システム全体の動作 |
| 速度 | 速い | 速い | 遅い |
| 環境 | 不要 | 不要（モック通信） | 全サービスが必要 |
| 発見できる問題 | ロジックのバグ | インターフェースの不整合 | 統合した際の問題 |

## いつ使うか

- マイクロサービスが増えてサービス間の連携が複雑になってきたとき
- 「API を変更したら他のサービスが壊れた」が繰り返し起きているとき
- E2E テストの環境構築・実行コストが高くなってきたとき

小規模なモノリスや、サービスが2〜3つ程度の段階では過剰になりやすい。チームが独立してデプロイする段階から価値が出てくる。
