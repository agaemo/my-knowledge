# 負荷テスト・パフォーマンステスト

## なぜ存在するか

機能テスト（単体・統合・E2E）は「正しく動くか」を検証するが、「どれだけの負荷に耐えられるか」「遅くなっていないか」は検証しない。

本番でユーザーが急増したとき、あるいはリファクタリング後に性能が劣化したとき、機能テストは何も教えてくれない。負荷テストはシステムの**性能特性と限界**を明らかにする。

## テストの種類

| 種類 | 目的 | やり方 |
|---|---|---|
| **負荷テスト**（Load Test） | 想定ユーザー数での性能確認 | 期待ピーク負荷を一定時間かける |
| **ストレステスト**（Stress Test） | 限界点の特定 | 徐々に負荷を上げてシステムが壊れる点を探す |
| **スパイクテスト**（Spike Test） | 急激な負荷増加への耐性 | 短時間で急激に負荷を増やす |
| **ソークテスト**（Soak Test） | 長時間稼働での劣化検出 | 通常負荷を数時間〜数日かける（メモリリーク検出など） |
| **ベンチマーク** | 変更前後の性能比較 | 同条件で計測して差分を見る |

## ツール：k6

JavaScript でシナリオを書く負荷テストツール。CLI で実行でき CI に組み込みやすい。

```js
// load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 20 },   // 30秒かけて20ユーザーまで増やす
    { duration: '1m',  target: 20 },   // 1分間 20ユーザーを維持
    { duration: '10s', target: 0 },    // 10秒かけて0に戻す
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // 95パーセンタイルが 500ms 未満
    http_req_failed:   ['rate<0.01'],  // エラー率 1% 未満
  },
};

export default function () {
  const res = http.get('https://api.example.com/users');
  check(res, { 'status is 200': (r) => r.status === 200 });
  sleep(1);
}
```

```bash
k6 run load-test.js
```

## 測定する指標

| 指標 | 意味 | 目安 |
|---|---|---|
| **レイテンシ（p50/p95/p99）** | リクエストの応答時間のパーセンタイル | p95 < 500ms が一般的な目標 |
| **スループット（RPS）** | 秒あたりの処理リクエスト数 | ピーク時の想定値を定義する |
| **エラー率** | 失敗したリクエストの割合 | < 1% |
| **CPU・メモリ使用率** | サーバーリソースの消費状況 | 限界値に近づかないか監視 |

p50 だけ見ていると遅いユーザーを見落とす。**p95・p99 を必ず確認する**。

## いつ実行するか

| タイミング | 目的 |
|---|---|
| リリース前 | 本番スケールに耐えられるかの確認 |
| インフラ変更後 | スケールアップ・DB 変更の効果測定 |
| パフォーマンス改善後 | ビフォーアフターの比較 |
| 定期実行（週次など） | 性能劣化の早期検出 |

## いつ使うか

**有効な場面**:
- ユーザー数の増加・イベントのトラフィックスパイクが予測されるとき
- API のレスポンスタイムにSLAがあるとき
- アーキテクチャ変更（DB・キャッシュ追加など）の効果を測りたいとき

**注意点**:
- 本番環境に直接かけると実害が出る。専用の負荷テスト環境を用意する
- 外部API（決済・メール）への呼び出しはモックするかテストアカウントを使う
