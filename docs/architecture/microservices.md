# マイクロサービス

アプリケーションを小さな独立したサービス群に分割し、それぞれが独自のプロセスで動作・独立してデプロイできるアーキテクチャスタイル。

## モノリスとの対比

```
モノリス:
  [UI + ビジネスロジック + DB アクセス] が1つのプロセス
  → シンプルだが、スケールや変更が全体に影響する

マイクロサービス:
  [注文サービス] [商品サービス] [ユーザーサービス] [決済サービス]
  → 各サービスが独立。それぞれが独自の DB を持つことが多い
```

## なぜ存在するか

モノリスが成長すると以下の問題が起きる。

**デプロイの依存:** 決済の小さな変更のために、全サービスをまとめてデプロイしなければならない。リスクが高く、リリース頻度が下がる。

**スケールの非効率:** 検索機能だけが重い場合も、アプリ全体をスケールアウトするしかない。

**技術の固定:** 全体が同じ言語・フレームワークに縛られる。

**チームの結合:** 大きなコードベースを複数チームが触ると、コンフリクトと調整コストが増える。

マイクロサービスはこれらを解決するために、**サービスを独立した単位に分割する**。

## メリット

- **独立したデプロイ:** 各サービスが独自のリリースサイクルを持てる
- **技術の自由度:** サービスごとに最適な言語・DBを選べる
- **スケールの最適化:** 負荷の高いサービスだけスケールアウトできる
- **チームの自律性:** コンウェイの法則に従い、チームがサービスを所有する

## デメリット・課題

**分散システムの複雑さ:** ネットワーク越しの通信は失敗する。タイムアウト・リトライ・サーキットブレーカーが必要になる。

**データの整合性:** サービスごとに DB を持つと、またがるトランザクションが難しい。[Saga パターン](/architecture/cqrs-event-sourcing) などで対処する。

**運用コスト:** 10のサービスは10の CI/CD・監視・ログ・デプロイ設定が必要。[オブザーバビリティ](/observability/)・[Kubernetes](/dev-tools/infra/kubernetes) などの基盤が不可欠。

**ネットワーク遅延:** サービス間通信がボトルネックになる。モノリスより遅くなるケースがある。

## サービスの分割方法

**ドメイン（境界づけられたコンテキスト）で分割する**のが基本。[DDD](/architecture/ddd) の考え方を使う。

```
EC サイトの例:
  注文コンテキスト → 注文サービス
  商品コンテキスト → 商品サービス
  ユーザーコンテキスト → ユーザーサービス
  決済コンテキスト → 決済サービス
```

分割が細かすぎると「ナノサービス」問題が起き、サービス間の通信が増えすぎる。適切な粒度を見つけることが重要。

## いつ採用するか

マイクロサービスは万能ではなく、**準備ができていない段階での採用は失敗しやすい**。

```
採用を検討する条件:
  ✓ モノリスで実際に痛みが出ている（デプロイ競合・スケール問題）
  ✓ チームが複数あり、独立して動けるようにしたい
  ✓ CI/CD・コンテナ・監視の基盤が整っている
  ✓ サービスの境界（ドメイン）が明確に見えている

時期尚早なケース:
  ✗ 立ち上げ直後・PMF 前（要件がまだ変わる）
  ✗ チームが1〜2人
  ✗ 運用基盤が整っていない
```

## モノリスから移行する場合

いきなり全部を分割するのではなく、[ストラングラーフィグパターン](/architecture/strangler-fig) で段階的に切り出す。まず [モジュラーモノリス](/architecture/modular-monolith) で内部を整理してから分割するのが現実的。

```
モノリス → モジュラーモノリス → マイクロサービス（一部から）
```
