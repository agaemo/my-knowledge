# CQRS / Event Sourcing

## CQRS（Command Query Responsibility Segregation）

読み取り（Query）と書き込み（Command）のモデルを分離するパターン。  
Greg Young が提唱。DDD・イベント駆動アーキテクチャと組み合わせて使われることが多い。

### なぜ存在するか

通常のアーキテクチャでは、読み取りと書き込みが同じモデル・同じDBを共有する。

```
問題:
  書き込みに最適なデータ構造と読み取りに最適なデータ構造は異なる
  → 在庫の更新（正規化・整合性重視）と
    ダッシュボード表示（非正規化・JOINを避けたい）を同じモデルで扱うと両立しない
```

CQRS は「書き込み」と「読み取り」を別のモデルに分離することでこれを解決する。

```
Command（書き込み）側:
  整合性・バリデーション・ビジネスルールに最適化
  正規化された DB（RDBMS）

Query（読み取り）側:
  パフォーマンス・柔軟な検索に最適化
  非正規化された読み取り専用ビュー（Redis・Elasticsearch・別テーブル）
```

### トレードオフ

| メリット | デメリット |
|---|---|
| 読み取り・書き込みを独立してスケール | 実装が複雑になる |
| 読み取りを目的別に最適化できる | 読み取りモデルの同期ラグ（結果整合性） |
| 書き込みロジックがシンプルになる | 2つのモデルを維持するコスト |

**小規模なアプリには不要。** 複雑なドメインロジック・高負荷な読み取りがある場合に有効。

## Event Sourcing

状態をそのまま保存するのではなく、**状態を変化させたイベントの履歴を保存する**パターン。現在の状態はイベントを順番に再生して導出する。

### 通常の状態保存との違い

```
通常（State-Based）:
  DB に現在の状態だけ保存
  UPDATE orders SET status = 'shipped' WHERE id = 1;
  → 「いつ・なぜ shipped になったか」が消える

Event Sourcing:
  イベントを追記していく
  INSERT INTO events VALUES ('OrderShipped', {orderId: 1, at: '...', by: '...'})
  → 全履歴が残る。状態はイベントを再生して導出
```

### なぜ存在するか

- **監査ログが完全に残る** — 誰が何をいつ変えたか、後から追える
- **過去の状態を再現できる** — 「3日前の時点での残高」を計算できる
- **バグの原因追跡** — どのイベントが問題を引き起こしたか特定できる
- **イベントを他のサービスに配信できる** — イベントストリームとして使える

### イメージ

```
銀行口座の例（通常）:
  残高: 1,000円  ← これだけ

銀行口座の例（Event Sourcing）:
  AccountOpened    { amount: 0 }
  MoneyDeposited   { amount: 5,000 }
  MoneyWithdrawn   { amount: 3,000 }
  MoneyDeposited   { amount: 2,000 }
  → 再生すると残高 4,000 円
```

### トレードオフ

| メリット | デメリット |
|---|---|
| 完全な変更履歴 | イベント数が増えると再生が遅くなる（スナップショットで対応） |
| 過去の任意時点に戻れる | クエリが複雑（集計・検索が難しい） |
| イベント駆動と相性がいい | スキーマ変更が難しい（既存イベントとの互換性） |

## CQRS と Event Sourcing の組み合わせ

2つは独立したパターンだが、相性がよいため一緒に語られることが多い。

```
書き込み: コマンド → イベントを生成・保存（Event Sourcing）
読み取り: イベントを購読 → 読み取り専用のビューを構築（CQRS の Query モデル）
```

```
[ユーザー操作]
      ↓
[Command Handler]
      ↓ イベントを保存
[Event Store]
      ↓ イベントを発行
[Event Handler] → 読み取り用 DB を更新
                       ↓
              [Query Model（非正規化）]
                       ↑
              [読み取りAPI（高速）]
```

## いつ使うか

**CQRS:**
- 読み取りと書き込みの負荷が大きく異なる
- 複数の読み取りビューが必要（モバイル向け・管理画面向けなど）
- DDD でドメインロジックが複雑になってきたとき

**Event Sourcing:**
- 監査ログが業務要件として必要（金融・医療・EC）
- 過去の状態を再生する必要があるとき
- イベントを他システムに配信してリアルタイム連携したいとき
