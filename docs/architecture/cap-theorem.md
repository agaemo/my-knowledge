# CAP定理・分散システムの基礎

分散システムで「一貫性・可用性・分断耐性の3つを同時に満たすことはできない」という定理。  
データベースや分散システムの選択・設計判断の根拠になる概念。

## CAP 定理

2000年に Eric Brewer が提唱。

| 要素 | 意味 |
|---|---|
| **C — Consistency**（一貫性） | 全ノードが常に同じ最新データを返す |
| **A — Availability**（可用性） | 全てのリクエストが必ず応答を返す（エラーなし） |
| **P — Partition Tolerance**（分断耐性） | ネットワーク分断が起きても動作し続ける |

**分散システムではネットワーク分断（P）は避けられない前提。**  
よって実質的な選択は「C か A か」になる。

```
ネットワーク分断が発生したとき:

  CP（一貫性優先）: 古いデータを返すくらいならエラーを返す
  AP（可用性優先）: 古くてもよいので応答し続ける
```

## CP vs AP の選択

### CP（Consistency + Partition Tolerance）

一貫性を優先し、不確かな場合はエラーを返す。

- **例:** HBase・ZooKeeper・Etcd・MongoDB（デフォルト設定）
- **向いている場面:** 金融・在庫・認証など「古いデータを読むことが許容されない」領域

### AP（Availability + Partition Tolerance）

可用性を優先し、古いデータを返すことを許容する（結果整合性）。

- **例:** Cassandra・DynamoDB・CouchDB
- **向いている場面:** SNSのタイムライン・アクセスカウント・カート情報など「少し古くても動いている方がいい」領域

## 結果整合性（Eventual Consistency）

AP システムが採用する考え方。「最終的には全ノードが同じ状態に収束する」。

```
t=0: ノードA に書き込み → 残高 1,000円
t=1: ノードB を読み取り → まだ 0円（まだ伝播していない）
t=2: ノードB を読み取り → 1,000円（伝播完了）
```

「いつ一致するか」は保証しないが「いつかは一致する」ことを保証する。

## PACELC（CAP の発展）

CAP は分断が「起きているとき」だけを扱う。2012年に Daniel Abadi が分断が「起きていないとき」も含めた PACELC を提唱。

```
P（分断あり）: A か C かを選ぶ（CAP と同じ）
E（通常時）  : L（低レイテンシ）か C（一貫性）かを選ぶ
```

通常時でも「全ノードへの書き込み確認（強一貫性）」を取るとレイテンシが上がるトレードオフがある。

## 分散システムの代表的な問題

### 脳分割（Split Brain）

ネットワーク分断でノードが互いを「死んだ」と判断し、それぞれがリーダーとして動き出す状態。データの不整合が起きる。

**対策:** クォーラム（過半数のノードの合意を必須にする）

### 時刻の同期問題

分散システムでは各ノードの時計がわずかにズレる。イベントの順序を時刻だけで決定すると順序の逆転が起きる。

**対策:** 論理クロック（Lamport timestamps）・ベクタークロック

## データベース選択への影響

```
強一貫性が必要（金融・在庫）
  → PostgreSQL・MySQL（RDBMS）
  → Google Spanner・CockroachDB（分散 CP）

高可用性・スケール優先（タイムライン・セッション）
  → DynamoDB・Cassandra（AP、結果整合性）

柔軟なトレードオフが必要
  → MongoDB（設定で一貫性レベルを調整可能）
```

## いつ意識するか

- マイクロサービスで複数 DB をまたぐ処理を設計するとき
- 「在庫が一時的にマイナスになってもよいか」などの要件を判断するとき
- データベースを選定するとき
- レプリケーションラグによる問題が発生したとき（「書き込んだばかりなのに読めない」）
