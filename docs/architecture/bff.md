# BFF（Backend for Frontend）

フロントエンドの種類ごとに専用のバックエンド API を設ける設計パターン。

## なぜ存在するか

複数のクライアント（Web・モバイル・スマートTV など）が同じ API を共有すると、それぞれが必要なデータ量・形式・認証方式が異なるため、どのクライアントにとっても「半分使えない」API になりがちになる。

**オーバーフェッチ：** モバイルは小さい画面なのに、Web 向けの大きなレスポンスを受け取る。

**アンダーフェッチ：** 1画面を表示するために複数エンドポイントを叩く必要がある。

**変更の衝突：** 片方のクライアントに合わせた変更がもう片方を壊す。

BFF はクライアントごとに独立したバックエンド層を置くことで、各クライアントのニーズに最適化された API を提供する。

## 構造

```
         ┌─────────────┐   ┌────────────────────┐
Web App  │  Web BFF    │──▶│                    │
         └─────────────┘   │  マイクロサービス群   │
                            │  (注文・商品・ユーザー)│
Mobile   ┌─────────────┐   │                    │
App      │ Mobile BFF  │──▶│                    │
         └─────────────┘   └────────────────────┘
```

各 BFF は：

- 必要なサービスを呼び出してデータを集約（Aggregation）
- クライアントの形式に合わせてレスポンスを整形
- クライアント固有の認証・認可を担う

## メリット

- **クライアント最適化：** 各クライアントが必要なデータだけを取得できる
- **独立した変更：** Web の変更がモバイルに影響しない
- **フロントエンドチームの自律性：** BFF の変更をフロントエンドチームが自分で管理できる

## デメリット

- **重複コード：** 複数の BFF が似た処理を実装する可能性がある
- **管理コスト：** クライアントの数だけ BFF も増える
- **データ集約の遅延：** 複数サービスへの並列呼び出しが必要な場合、レイテンシの最適化が必要

## いつ使うか

```
使うケース:
  ✓ クライアントの種類が複数あり、API のニーズが異なる
  ✓ マイクロサービス構成で、フロントエンドからの呼び出しが増えている
  ✓ フロントエンドチームが自律して API 変更できる体制にしたい

使わないケース:
  ✗ クライアントが1種類のみ（BFF の価値が薄い）
  ✗ チームが小さく、BFF の管理コストが払えない
```

単一クライアントの場合は、[API Gateway](/backend/api/api-gateway) でルーティング・認証を担い、BFF を省くのが現実的。
