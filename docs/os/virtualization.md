# 仮想化（Virtualization）

1台の物理マシン上で、複数の独立した仮想マシン（VM）を動かす技術。CPUの仮想化支援機能（Intel VT-x・AMD-V）を利用し、それぞれのVMが専用のハードウェアを持つかのように振る舞う。

## なぜ存在するか

物理サーバーを1用途1台で使うと、CPUやメモリが余りリソース効率が悪い。仮想化により1台の物理サーバーで複数のOSを動かし、リソースを有効活用できる。加えて、VM単位でスナップショット・バックアップ・移行ができるため運用も柔軟になる。

## ハイパーバイザー（Hypervisor）

物理ハードウェアとVMの間に位置し、CPU・メモリ・ストレージ・ネットワークを各VMに割り当てて管理するソフトウェア。VMM（Virtual Machine Monitor）とも呼ばれる。

### Type 1（ベアメタルハイパーバイザー）

```
┌──────┐  ┌──────┐  ┌──────┐
│  VM  │  │  VM  │  │  VM  │
└──────┘  └──────┘  └──────┘
┌──────────────────────────┐
│      ハイパーバイザー      │  ← OSなしで直接ハードウェア上で動く
└──────────────────────────┘
┌──────────────────────────┐
│      物理ハードウェア      │
└──────────────────────────┘
```

ホストOSを介さないため高速・安定。主にサーバー・データセンターで使われる。

| 製品 | 用途 |
|------|------|
| **VMware ESXi** | エンタープライズ向けの代表的なType 1。vSphere/vCenterで複数ホストを管理 |
| **Microsoft Hyper-V** | Windows Server / Windows 10以降に内蔵。Type 1に分類される |
| **KVM** | Linuxカーネルに組み込まれたハイパーバイザー。クラウド基盤（OpenStack等）で広く使われる |
| **Xen** | AWSの初期インフラで使われていたOSSハイパーバイザー |

### Type 2（ホスト型ハイパーバイザー）

```
┌──────┐  ┌──────┐
│  VM  │  │  VM  │
└──────┘  └──────┘
┌──────────────────┐
│  ハイパーバイザー  │  ← ホストOS上のアプリとして動く
└──────────────────┘
┌──────────────────┐
│     ホストOS      │
└──────────────────┘
┌──────────────────┐
│   物理ハードウェア │
└──────────────────┘
```

通常のアプリとしてインストールできるため開発・検証用途に向く。ホストOSを介するぶんオーバーヘッドがある。

| 製品 | 用途 |
|------|------|
| **VMware Workstation / Fusion** | Windows/Mac向けの高機能な開発用VM |
| **VirtualBox** | OSSで無料。クロスプラットフォーム |
| **Parallels Desktop** | Mac向け。Apple SiliconでWindows ARMを動かせる |

## Hyper-V について

MicrosoftがWindowsに内蔵するハイパーバイザー。**Windows 10 Pro・Enterprise / Windows Server** で有効化できる（Homeエディションは不可）。

Type 1に分類されるが、インストール方法はType 2に近い（Windows上の機能として有効化する）。実態は、Hyper-Vを有効にするとWindowsホスト自体もHyper-Vの上で動くVMになる。

```
Hyper-V を有効化すると:
  物理ハードウェア
    └─ Hyper-Vハイパーバイザー
         ├─ Windows（管理パーティション / ルートパーティション）← ホストに見えていたWindowsがVMになる
         └─ VM（Linuxなど）
```

### Docker Desktopとの関係

Windows版Docker DesktopはLinuxコンテナを動かすために内部でLinux VMが必要になる。Hyper-Vが有効な環境ではそのVMをHyper-V上に作成する（またはWSL 2を使う）。Hyper-VとVirtualBoxは同時に動作できないことがある。

## VMware ESXiとvSphere

**ESXi**はVMwareのType 1ハイパーバイザー本体。物理サーバーにインストールして使う。

**vSphere**は複数のESXiホストをまとめて管理するプラットフォームの総称で、**vCenter Server**（管理サーバー）を中心に構成される。エンタープライズのデータセンターで広く使われてきたが、Broadcomによる買収後のライセンス変更で移行を検討する組織が増えている。

## VMとコンテナの違い

```
仮想マシン（VM）:               コンテナ:
┌────────────────┐             ┌──────┐ ┌──────┐ ┌──────┐
│  アプリ          │             │ アプリ│ │ アプリ│ │ アプリ│
│  ゲストOS        │             └──────┘ └──────┘ └──────┘
│  （カーネル含む）  │             ┌──────────────────────┐
└────────────────┘             │    コンテナランタイム     │
ハイパーバイザー                 │  （カーネルはホストと共有）│
物理ハードウェア                 └──────────────────────┘
                                      ホストOS カーネル
                                      物理ハードウェア
```

| | VM | コンテナ |
|-|----|---------|
| 起動時間 | 分単位 | 秒以下 |
| サイズ | GBオーダー（OS込み） | MBオーダー |
| 隔離レベル | カーネルレベルで完全分離 | プロセス・名前空間で分離 |
| ホストOSのカーネル | 不要（ゲストOS内蔵） | 共有 |
| 用途 | 異なるOSの実行・強い隔離が必要な場面 | アプリの実行環境・CI/CD・マイクロサービス |
