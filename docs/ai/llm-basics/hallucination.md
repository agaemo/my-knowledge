# ハルシネーション（Hallucination）

LLM が事実と異なる内容を、確信を持って生成する現象。「幻覚」とも呼ばれる。  
単なるバグではなく、LLM の学習の仕組みに由来する構造的な問題。

## なぜ起きるか

LLM は「次のトークンを予測する」モデルとして学習されている。事実を記憶しているのではなく、大量のテキストから統計的なパターンを学習している。

```
LLM の内部にあるもの: 「この文脈ではこのトークンが続きやすい」というパターン
LLM の内部にないもの: 「この情報は本当か」を判定する機構
```

自信満々に誤りを述べるのは、モデルが「それらしい答え」を生成するように最適化されているため。「わからない」と言うより、それらしい答えを作る方がパターン的に自然に見える。

## ハルシネーションの種類

**事実の捏造（Factual Hallucination）**  
存在しない論文・人物・法律・APIメソッドを自信満々に述べる。

```
ユーザー: 「〇〇に関する論文を教えて」
LLM:     「Smith et al. (2021) の研究では...」← 存在しない論文
```

**文脈への不忠実（Faithfulness Hallucination）**  
与えた文書・コードと矛盾する内容を生成する。RAG で取得したドキュメントの内容を誤って要約する場合など。

**時間的ハルシネーション（Temporal Hallucination）**  
知識カットオフより後の情報を「知らない」と言う代わりに、古い情報を最新として述べる。

## なぜ重要か

ハルシネーションが起きるという前提を理解することで、他の AI パターンが「なぜ存在するか」がわかる。

| 対策パターン | ハルシネーションとの関係 |
|---|---|
| RAG | 外部の正確なデータをコンテキストに注入し、根拠のある回答を生成させる |
| Evals | ハルシネーションの発生率を定量的に測定する |
| HITL | 重要な判断に人間の確認を挟む |
| Guardrails | 出力の事実確認・フィルタリング |
| 引用・ソース提示 | 根拠となる情報源を明示させ、検証可能にする |

## 軽減策

**モデルへの指示で減らす**
- 「わからない場合は『わかりません』と言ってください」と明示する
- 「根拠となる情報源を示してください」と求める
- 温度（temperature）を下げ、確率の高い出力に絞る

**設計で減らす**
- RAG で正確な情報をコンテキストに渡す
- 回答をスキーマで制約する（構造化出力）
- 長い連鎖推論（エージェント）ではステップごとに検証する

**完全には解消できない**  
現在の LLM アーキテクチャにおいて、ハルシネーションをゼロにする方法は存在しない。「どこで起きたら許容できるか」「どこで起きたら危険か」を設計上で分けることが現実的なアプローチ。
