# コンテキストエンジニアリング（Context Engineering）

LLM のコンテキストウィンドウに「何を・どの順で・どのような形式で入れるか」を設計する実践。  
Andrej Karpathy が 2025年に提唱し、プロンプトエンジニアリングの進化形として広まった。

## プロンプトエンジニアリングとの違い

**プロンプトエンジニアリング** は「どう指示するか」の技術。単発の入力をうまく書く。

**コンテキストエンジニアリング** は「コンテキスト全体を設計する」技術。何をコンテキストに含めるか・除外するか・どう構造化するかを管理する。

```
プロンプトエンジニアリング:
  「ステップごとに考えてください。出力は JSON 形式で...」

コンテキストエンジニアリング:
  システムプロンプト + 会話履歴 + RAG 結果 + ツール出力 + ユーザー入力
  = コンテキストウィンドウ全体の設計
```

プロンプト単体を磨くだけでなく、**コンテキストウィンドウという限られたリソースを何で埋めるか**がパフォーマンスを左右する。

## コンテキストウィンドウの構成要素

LLM が一度に受け取れる情報（コンテキストウィンドウ）は、複数の情報源から構成される。

```
┌─────────────────────────────────────────┐
│ コンテキストウィンドウ                  │
│                                         │
│ ① システムプロンプト                   │
│    ロール・制約・出力形式の指定         │
│                                         │
│ ② 会話履歴                             │
│    過去のやり取り（長くなると圧迫する） │
│                                         │
│ ③ RAG で取得した文書                   │
│    外部データベースから検索した情報     │
│                                         │
│ ④ ツール呼び出しの結果                 │
│    API レスポンス・コード実行結果       │
│                                         │
│ ⑤ Few-shot の例                        │
│    期待する入出力のサンプル             │
│                                         │
│ ⑥ ユーザーの入力                       │
└─────────────────────────────────────────┘
```

これらが限られたトークン数に収まるように設計する必要がある。

## なぜ重要か

シンプルな1回の質問応答ではプロンプトエンジニアリングで十分。  
コンテキストエンジニアリングが重要になるのは以下のケース：

**AI エージェント** — 複数ステップ・複数ツールを渡り歩くと、コンテキストが膨らみ続ける。何を保持し、何を捨てるかの管理が必要。

**長い会話** — 履歴を全部残すとウィンドウが埋まる。古い履歴を要約・圧縮して残すか、重要な部分だけ保持するかを設計する。

**RAG** — 検索結果を全部入れると無関係な情報でノイズが増える。どの文書を・何件・どの順で入れるかが精度に直結する。

## 主なテクニック

**コンテキスト圧縮**  
会話履歴が長くなったら要約して短縮する。重要な情報を保ちつつトークン数を削減する。

**選択的な RAG**  
検索でヒットしたものを全部入れるのではなく、関連スコアでフィルタリングして本当に必要な情報だけを注入する。

**構造化されたコンテキスト**  
情報を XML タグや見出しで区切り、モデルが情報の種類を区別しやすくする。

```xml
<system>あなたは...です</system>
<documents>
  <doc id="1">...</doc>
</documents>
<conversation>...</conversation>
<user_input>...</user_input>
```

**コンテキストの優先順位**  
LLM はコンテキストの先頭と末尾に注意を向けやすい（中間の情報を見落としやすい）。重要な情報を適切な位置に配置する。

**メモリの外部化**  
コンテキストに入れるのではなく、データベースに保存しておき、必要なときだけ取得する（エージェントの長期記憶）。

## いつ意識するか

- エージェントを構築するとき
- 会話が長くなるチャットボット
- RAG システムの精度を上げたいとき
- コンテキストウィンドウのコスト（トークン数）を最適化したいとき

単発のプロンプトや短い Q&A では、プロンプトエンジニアリングで十分。システムが複雑になるほど、コンテキスト全体の設計が重要になる。
