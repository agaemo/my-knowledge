# Vibe Coding

## 何か

「AIに自然言語で意図を伝え、AIが実装する」スタイルのコーディング。
2025年に Andrej Karpathy が命名・定義した。従来の「コードを書くための補完ツールとしてAIを使う」から、「AIが主体的に実装し、人間がレビュー・方向付けする」へのシフト。

## 従来の AI 補助コーディングとの違い

| | 従来（Copilot的） | Vibe Coding |
|---|---|---|
| 人間の役割 | コードを書く・AIが補完 | 意図を伝える・AIが実装 |
| AIの役割 | 補完・サジェスト | 設計〜実装〜デバッグ |
| 成果物の確認 | 書いた行を確認 | 動作・テストで確認 |
| 必要なスキル | コーディング力 | 設計力・レビュー力 |

## Vibe Coding が有効な場面

- プロトタイプ・PoC（動くものを素早く作りたい）
- 自分が詳しくない言語・フレームワークを触るとき
- ボイラープレートや反復作業
- テストコードの生成

## Vibe Coding が有効でない場面

::: warning 任せてはいけない領域
- セキュリティ・認証・課金など、バグが致命的な領域
- パフォーマンスが重要な低レベル処理
- AI が文脈を持ちきれない巨大なコードベース
:::

## 実践上の落とし穴

### 1. 生成コードを理解せずにマージする

AIが生成したコードは動いても「なぜそう書いたか」を理解していないと、後でデバッグできない。
**対策**: 生成コードは必ず読む。理解できない部分はAIに説明させる。

### 2. コンテキストが蓄積されず、AIが迷走する

会話が長くなると前半の決定をAIが忘れ、矛盾した実装が増える。
**対策**: CLAUDE.md・設計ドキュメントに決定事項を記録し、AIに毎回参照させる。

### 3. テストなしで「動いた」と判断する

「一見動く」コードが大量生成される。テストがないと品質の保証がない。

::: tip
TDD と組み合わせる。最低限、AI にテストも同時に書かせる。
:::

### 4. 技術的負債の急速な蓄積

Vibe Coding はスピードが出る分、設計の整合性が崩れやすい。
**対策**: 一定量の実装後にリファクタリングセッションを設ける。

## 効果的なワークフロー

```
1. 要件・設計を自分で考える（AIに任せない）
2. 小さい単位でAIに実装させる
3. 生成コードを読んでレビュー・修正指示
4. テストで動作確認
5. 定期的にコードベース全体を見直す
```

## Vibe Coding に必要な能力の変化

コーディング力よりも以下が重要になる：

- **設計力** — 何を作るかの判断・分解は人間が担う
- **レビュー力** — 生成コードの問題を見つける目
- **プロンプト力** — AIに正確な意図を伝える能力
- **コンテキスト管理** — AIが一貫した実装をできる環境の整備
