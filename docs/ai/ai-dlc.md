# AI 駆動開発ライフサイクル（AI-DLC）

## なぜ存在するか

従来のソフトウェア開発ライフサイクル（SDLC）は、すべての認知作業を人間が担う前提で設計されていた。AI コーディングツールの登場により、各フェーズでの作業分担と開発者の役割が変わりつつある。

AI-DLC は**AI を各フェーズに統合した開発プロセスの設計**を指す概念。単に「コードを書く部分だけ AI を使う」のではなく、要件定義からデプロイまでの全体で AI をどう活かすかを考える。

## 開発者の役割の変化

```
従来:  要件理解 → 設計 → 実装（コーディング）→ テスト → レビュー
AI-DLC: 要件理解 → 設計 → AI に実装させる → 検証・レビュー → 修正指示
```

開発者の役割が**「書く人」から「判断する人・検証する人」にシフト**する。コードを書くスキルより、AI に正確な意図を伝えるスキルと、AI の出力を評価するスキルが重要になる。

## フェーズ別の AI 活用

### 要件定義・設計

AI はアイデアの壁打ち・要件の整理・設計の代替案の提示に使える。人間が最終判断を下すが、検討の幅を広げる。

```
- 「この要件で考慮漏れはないか？」
- 「このアーキテクチャのトレードオフを整理して」
- 「類似の設計パターンを挙げて」
```

### 実装

Vibe Coding のアプローチで、自然言語の指示からコードを生成させる。ポイントは**小さい単位で反復すること**。大きなタスクを一度に渡すと検証が難しくなる。

→ 詳細は [Vibe Coding](/ai/vibe-coding) を参照

### テスト生成

実装済みのコードからテストケースを生成させる。特にエッジケース・境界値の網羅に有効。

```
- 「この関数のユニットテストを書いて。エッジケースを含めて」
- 「このAPIのインテグレーションテストを追加して」
```

ただし、AI が生成したテストは**仕様の確認なく通過するテストになりやすい**。テストの意図と期待値を人間がレビューする。

### コードレビュー

差分を AI に渡してセキュリティ問題・バグ・規約違反を検出させる。人間のレビュアーが見落としがちな網羅的なチェックに向く。

```
- 「このPRでセキュリティ上の問題はないか？」
- 「このコードでパフォーマンス上の懸念点は？」
```

### デプロイ・運用

インシデント対応・ログ解析・障害の原因調査に AI を活用できる。エラーメッセージとスタックトレースを渡して原因を絞り込む。

## AI-DLC を機能させる前提

### コンテキスト設計（Harness Engineering）

AI は与えられた情報の範囲でしか動けない。プロジェクトのルール・アーキテクチャ・禁止事項を環境として整備しておくことで、各フェーズでの出力品質が安定する。

→ 詳細は [Harness Engineering](/ai/harness-engineering) を参照

### 人間による検証（HITL）

AI の出力は正しいとは限らない。特に本番への影響が大きい操作（DB 変更・デプロイ・外部 API 呼び出し）では、人間の確認ステップを組み込む。

→ 詳細は [HITL](/ai/hitl) を参照

### 小さく反復する

AI に大きなタスクを一度に渡すと、問題の特定が難しくなる。1 機能・1 ファイル単位で反復し、各ステップで検証する。

## 向かないケース

- **曖昧なまま進める**: AI は曖昧な指示を補完するが、補完の方向が意図と違うことが多い。要件を言語化してから渡す
- **検証なしに積み上げる**: 前のステップの問題を確認せずに次を指示すると、誤りが連鎖する
- **全てを AI に委ねる**: セキュリティ設計・アーキテクチャの判断・ビジネスロジックの正しさは人間が責任を持つ
