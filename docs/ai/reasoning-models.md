# 推論モデル（Reasoning Models）

答える前に「考える」ステップを持つ LLM のカテゴリ。  
2024年に OpenAI が o1 を発表し、思考プロセスを明示的に生成するアプローチが広まった。

## 通常の LLM との違い

通常の LLM は入力を受け取ると、直接答えを生成する。

```
通常の LLM:
  プロンプト → [モデル] → 回答

推論モデル:
  プロンプト → [モデル] → <thinking>
                              ステップ1: ...
                              ステップ2: ...
                              確認: ...
                           </thinking> → 回答
```

「考える」部分（thinking tokens / scratchpad）は推論中に内部で生成され、最終的な回答の品質を高める。

## なぜ存在するか

複雑な問題は一発で正解を出すのが難しい。人間も難しい数学の問題を「見た瞬間に答えがわかる」のではなく、段階的に考えて解く。

LLM においても、中間ステップを明示的に生成させることで精度が上がることが実験的に示された（Chain of Thought の研究）。推論モデルはこの考え方を、モデルの学習レベルまで組み込んだもの。

### テスト時計算（Test-Time Compute）

従来の LLM の精度向上はモデルを大きくする（学習コストを増やす）ことで達成されてきた。推論モデルは別のアプローチを取る。

```
従来: モデルサイズを大きくする（学習コスト ↑）→ 精度 ↑
推論: 回答時に考える時間を増やす（推論コスト ↑）→ 精度 ↑
```

「考える時間」を増やすことで、小さいモデルでも難しい問題を解けるようになる。これを**テスト時スケーリング（Test-Time Scaling）**と呼ぶ。

## Chain of Thought との関係

**Chain of Thought（CoT）** はプロンプトエンジニアリングの技術。  
「ステップごとに考えてください」と指示することで、通常の LLM に推論ステップを出力させる。

**推論モデル** は CoT の考え方をモデルの学習に組み込んだもの。プロンプトで指示しなくても自動的に考える。

```
CoT:        プロンプトで「段階的に考えて」と指示する
推論モデル: 指示しなくても自動的に内部で推論する
```

## 主な推論モデル

| モデル | 提供元 | 特徴 |
|---|---|---|
| o1 / o3 / o4-mini | OpenAI | thinking tokens を内部生成。難易度に応じて思考量が変わる |
| Claude（extended thinking） | Anthropic | thinking ブロックを明示的に出力できる |
| DeepSeek-R1 | DeepSeek | オープンソースの推論モデル。o1 に匹敵する性能 |
| Gemini 2.0 Flash Thinking | Google | 高速な推論特化モデル |

## トレードオフ

| 観点 | 通常の LLM | 推論モデル |
|---|---|---|
| レイテンシ | 速い | 遅い（thinking の分だけ長い） |
| コスト | 低い | 高い（生成トークン数が多い） |
| 複雑な推論 | 苦手 | 得意 |
| 単純な質問 | 十分 | オーバースペック |

## いつ使うか

**推論モデルが向いている場面：**
- 数学・論理パズル・コードのデバッグ
- 複数ステップが必要な計画・分析
- 競合する要件のトレードオフ判断
- 法的・医療的な複雑な推論

**通常の LLM で十分な場面：**
- 要約・翻訳・文章生成
- 単純な Q&A・事実検索
- レイテンシが重要なリアルタイム応答
- 大量処理でコストを抑えたい場合

推論モデルはすべてにおいて優れているわけではない。単純なタスクに使うと遅くてコストがかかるだけになる。**難易度に応じてモデルを使い分ける**（ルーティング）のが実務的なアプローチ。
