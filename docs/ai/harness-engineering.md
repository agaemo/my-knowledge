# Harness Engineering

## なぜ存在するか

AI エージェント（モデル単体）はそのままでは制御が難しく、方向を誤ったり品質を保証できない。
ハーネスエンジニアリングは、エージェントを「手綱（harness）」で制御する仕組みを設計すること。

::: info Agent = Model + Harness
ハーネスとは「エージェント自体を除いたすべてのもの」。モデルにハーネスを組み合わせることで初めて有用なエージェントになる。
:::

馬に手綱をつけて正しい方向に導くように、AI エージェントに制御機構を組み込むことで、意図通りの出力・安全な行動・品質の担保を実現する。

ハーネスとは「エージェント自体を除いたすべてのもの」と定義される（Birgitta Böckeler / Martin Fowler）。

## 2つの制御メカニズム

### ガイド（フィードフォワード制御）

エージェントが行動する**前に**働く予測的な制御。初回から良い結果を出す確率を高める。

```
人間の意図・プロジェクトルール → ガイド → エージェントの行動
```

**例:**
- 常時コンテキストファイル（CLAUDE.md・.cursorrules・copilot-instructions.md）
- 権限設定（何を自律実行してよいか・禁止するか）
- アーキテクチャ制約のドキュメント
- 再利用可能なプロンプト・コマンド

### センサー（フィードバック制御）

エージェントが行動した**後に**働く観察的な制御。問題を検知して自己修正を促す。

```
エージェントの行動 → センサー → エラー・警告のシグナル → エージェントが修正
```

**例:**
- テスト・型チェッカー・リンター（行動直後に自動実行）
- CI パイプラインの検証
- コードレビューエージェント（AI によるセマンティック検査）

センサーが LLM に最適化されたシグナルを返すほど、エージェントが自己修正しやすくなる。

## 2つの制御タイプ

| タイプ | 特性 | 例 |
|---|---|---|
| **コンピュテーショナル** | 決定論的・高速（ms〜s）・信頼性が高い | テスト・型チェッカー・リンター |
| **インフェレンシャル** | AI によるセマンティック分析・低速・コスト高・非決定論的 | LLM-as-judge・コードレビューエージェント |

コンピュテーショナル制御を基盤にして、インフェレンシャル制御で補完するのが基本設計。

## 3つのハーネスカテゴリ

### 保守性ハーネス（Maintainability）

内部コード品質を規制する。ツールが最も整備されており、成熟度が高い。

- 構造的問題: 型チェッカー・静的解析（コンピュテーショナル）
- セマンティックな問題: コードレビューエージェント（インフェレンシャル）

### アーキテクチャ適合ハーネス（Architecture Fitness）

アーキテクチャ特性・パフォーマンス要件・観測可能性標準などの制約を強制する。

### 振る舞いハーネス（Behaviour）

機能的な正しさを保証する。最も難しいカテゴリで、AI 生成テストへの過信は禁物。

## ハーネス可能性（Harnessability）

すべてのコードベースがハーネスを適用しやすいわけではない。以下の環境はハーネスに向く。

- 強い型付き言語
- 明確なモジュール境界
- 確立されたフレームワーク

コードベースを「読み取り可能・ナビゲート可能・扱いやすい」状態に整えることが、効果的なハーネス設計の前提になる。

## 人間の役割：操舵ループ

ハーネスは一度作れば終わりではなく、継続的に改善するサイクルが重要。

```
エージェントが問題を起こす → ガイド・センサーを改善 → 再発防止
```

::: tip ハーネスの目標
人間の関与をなくすことではなく、**人間の判断が最も重要な場所に集中させる**こと。ハーネスは人間の経験を外部化できるが、完全には置き換えられない。
:::

## いつ使うか

- AI エージェントに自律的なタスクを任せる構成を作るとき
- エージェントの出力品質・安全性を担保したいとき
- チーム全体で AI エージェントを一貫して運用したいとき
