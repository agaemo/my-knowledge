# HITL（Human-in-the-Loop）

AIシステムの意思決定プロセスに人間を組み込む設計パターン。AIが判断・生成した結果を人間がレビュー・承認・修正してから次のステップに進む。

## なぜ存在するか

LLMは高精度だが誤りを犯す。特に高リスクな判断（医療・法律・金融・本番環境への操作）をAIに全委任すると、誤った出力がそのまま実行されてしまう。HITLはAIの自律性と人間の監督を組み合わせ、精度・安全性・信頼性を担保する。

## 介入のタイミング

### Before（事前承認）
AIが行動を起こす前に人間が承認する。最も保守的なアプローチ。

```
AI が提案 → 人間がレビュー → 承認/却下 → 実行
```

**例**: AIが生成したSQLを本番DBに実行する前にDBAが確認する。

### After（事後レビュー）
AIが出力した後に人間が品質チェックする。スループットを維持しつつ誤りを補足する。

```
AI が生成 → 実行 → 人間が品質チェック → フィードバック
```

**例**: カスタマーサポートのAI回答を送信後にオペレーターが確認し、問題があれば追加対応する。

### Exception（例外介入）
AIが低信頼度・異常ケースと判断した時だけ人間にエスカレーションする。通常ケースは全自動。

```
AI が判断 → 信頼度が閾値以上 → 自動実行
              信頼度が閾値未満 → 人間にエスカレーション
```

**例**: 感情分析で「中立」と判定されたチケットだけを人間がトリアージする。

## AIエージェントにおけるHITL

自律的に行動するAIエージェントでは、取り消し不可能な操作の前に人間の確認を挟む設計が重要。

```
# 不可逆な操作の前に確認を求める例
tools = [
    {
        "name": "delete_file",
        "require_approval": True,   # HITLフラグ
    },
    {
        "name": "read_file",
        "require_approval": False,
    },
]
```

### 確認すべき操作の例
- ファイル・レコードの削除
- 外部APIへのPOST・PATCH・DELETE
- メール・通知の送信
- 本番環境へのデプロイ
- 金融トランザクション

## フィードバックループとしてのHITL

人間のレビュー結果をモデルの改善に活用するサイクル。

```
AI 出力 → 人間がラベリング・修正 → データ蓄積 → Fine-tuning / Evals → モデル改善
```

EvalsのテストケースやファインチューニングのデータセットをHITLで収集するアプローチはLLMOpsの重要な要素。

## 自動化レベルのスペクトラム

| レベル | 説明 | 例 |
|--------|------|----|
| 完全手動 | AIは提案のみ、実行は常に人間 | AIドラフト→人間が送信 |
| HITL | 条件に応じて人間が介入 | 低信頼度時のみ確認 |
| 完全自動 | 人間介入なし | バッチ処理・監視アラート |

## いつ使うか

- 出力の誤りが重大な結果を招く（医療・法律・金融）
- 取り消し不可能な操作を含むエージェント
- モデルがまだ本番精度に達していない初期フェーズ
- コンプライアンス上、人間の承認記録が必要
- Fine-tuning・Evalsのデータを収集したい
