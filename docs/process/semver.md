# セマンティックバージョニング（semver）

バージョン番号を `MAJOR.MINOR.PATCH` の3つの数値で表し、それぞれの意味を明確に定義した規約。

## なぜ存在するか

バージョン番号の付け方がプロジェクトごとにバラバラだと、依存ライブラリを更新するときに「このバージョンアップは安全か？」の判断ができない。semverはバージョン番号に意味を持たせることで、変更の影響範囲を開発者が予測できるようにした。

## 構造

```
  1  .  2  .  3
  ↑     ↑     ↑
MAJOR MINOR PATCH
```

### PATCH（パッチ）
後方互換性のあるバグ修正。既存の動作を壊さない修正。

```
1.2.3 → 1.2.4
```

### MINOR（マイナー）
後方互換性のある機能追加。既存コードは壊れないが、新しいAPIや機能が増える。

```
1.2.3 → 1.3.0  （PATCHはリセット）
```

### MAJOR（メジャー）
後方互換性のない変更（破壊的変更）。既存コードが動かなくなる可能性がある。

```
1.2.3 → 2.0.0  （MINOR・PATCHはリセット）
```

## プレリリース

```
1.0.0-alpha.1   # アルファ版
1.0.0-beta.2    # ベータ版
1.0.0-rc.1      # リリース候補
```

正式リリース前のバージョンを示す。`1.0.0-alpha.1 < 1.0.0` として扱われる。

## 0.x.x（初期開発）

`0.x.x` はAPIが安定していない初期開発フェーズを示す。`0.x.0` のマイナーバージョンアップでも破壊的変更が起きる可能性がある。

```
0.1.0 → 0.2.0  # 破壊的変更があり得る
1.0.0           # 安定版としてのリリース
```

## バージョン範囲の指定（npm）

semverに基づいたバージョン範囲指定。

```json
{
  "dependencies": {
    "express": "^4.18.0",  // 4.x.x の最新（4.18.0以上5.0.0未満）
    "lodash": "~4.17.0",   // 4.17.x の最新（4.17.0以上4.18.0未満）
    "react": "18.2.0"      // 完全固定
  }
}
```

| 記号 | 意味 |
|------|------|
| `^` | MINORとPATCHの更新を許容（MAJORは固定） |
| `~` | PATCHの更新のみ許容（MINORは固定） |
| なし | 完全固定 |

## 破壊的変更の判断基準

以下は破壊的変更（MAJORを上げる）とみなす：

- 既存の公開APIの削除・リネーム
- 引数の型・数の変更
- 戻り値の型の変更
- 既存の動作の意味が変わる変更

以下はMINORで許容：

- 新しいエンドポイント・メソッドの追加
- オプション引数の追加
- デフォルト値の変更（既存コードに影響しない場合）

## いつ使うか

- ライブラリ・パッケージを公開するとき（npm・PyPI・RubyGemsなど）
- 社内ライブラリやAPIのバージョン管理
- CLIツールやSDKのリリース管理
