# ADR（Architecture Decision Records）

アーキテクチャ上の重要な意思決定を、その背景・理由・トレードオフとともに記録するドキュメント手法。  
Michael Nygard が 2011 年に提唱。

## なぜ存在するか

設計判断は往々にして「なぜそうしたか」が失われる。コードには「何をしているか」は残るが、「なぜその選択をしたか」は残らない。

半年後に「なぜ PostgreSQL にしたのか」「なぜ REST ではなく gRPC にしたのか」を調べると、当時の判断をした人がいなくなっていたり、Slack のログを掘り返す羽目になる。ADR はこの「意思決定の文脈」をコードと同じリポジトリに保存する。

## フォーマット

ADR は短く、1ページ程度に収める。代表的な構成：

```markdown
# ADR-0001: 認証に JWT を採用する

## Status
Accepted

## Context
モバイルアプリとWebアプリの両方から同一APIを使う必要がある。
セッション管理をサーバーサイドで持つとスケールアウト時にセッションの
共有問題が生じる。マイクロサービス間での認証情報の伝播も必要。

## Decision
JWT（JSON Web Token）を使ったステートレス認証を採用する。
トークンの有効期限は15分、リフレッシュトークンで再取得する。

## Consequences
- サーバー側でセッションを持たないため、水平スケールが容易になる
- トークンの即時無効化ができないため、有効期限を短くする必要がある
- リフレッシュトークンの管理（ローテーション・失効）が追加で必要になる
```

**Status の値：**

| 値 | 意味 |
|---|---|
| Proposed | 提案中、まだ決定していない |
| Accepted | 採用された |
| Deprecated | 廃止された（新規では使わない） |
| Superseded by ADR-XXXX | 別の ADR に置き換えられた |

## 置き場所

リポジトリの `docs/adr/` または `adr/` ディレクトリに連番で保存する。

```
adr/
├── 0001-use-jwt-for-auth.md
├── 0002-use-postgresql.md
└── 0003-adopt-grpc-for-internal-api.md
```

コードと同じリポジトリに置くことで、コードの変更と意思決定の記録が git 履歴上で紐づく。

## 何を記録するか

すべての技術的な判断を ADR にする必要はない。記録する価値があるのは：

- **後から変えるコストが高い決定** — DB の選択・認証方式・API スタイル・言語・フレームワーク
- **トレードオフが大きい決定** — 複数の選択肢があり、どちらにも一長一短がある
- **チームで議論が起きた決定** — 意見が割れた場合、なぜその結論になったかを残す
- **外部の制約に基づく決定** — セキュリティ要件・コスト・ライセンスなど

ルーティングの命名規則やコードフォーマットのルールは ADR より CLAUDE.md やスタイルガイドに書く。

## チームへの導入

1. まず過去の重要な決定を遡及的に ADR として書く（現状把握にもなる）
2. 「新しいアーキテクチャ上の決定をするときは ADR を書く」というルールを追加する
3. PR レビューに「この変更に ADR が必要か？」という観点を加える

ADR は重厚なドキュメントではなく、**短くていい**。書くコストが低いほど継続できる。
