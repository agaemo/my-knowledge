# Shape Up

## 何か

Basecamp（37signals）が提唱する独自の開発プロセス。
スクラムの「スプリント・バックログ・ベロシティ」を使わず、**6週間サイクル**で機能を完成させる。

アジャイルに疲れたチーム・スクラムがうまく機能していないチームの代替として注目されている。

## スクラムとの根本的な違い

| | スクラム | Shape Up |
|---|---|---|
| サイクル | 1〜2週間スプリント | 6週間サイクル |
| バックログ | 積み上がっていく | 存在しない |
| 見積もり | ストーリーポイント | アペタイト（予算） |
| 進捗管理 | バーンダウンチャート | ヒルチャート |
| 中断 | スプリント途中の割り込みあり | サイクル中は中断しない |

## 3つのフェーズ

```
Shaping（成形）  → Betting（賭け）  → Building（構築）
[6週間前に完了]    [2週間の冷却期間]   [6週間で完成]
```

### 1. Shaping（成形）

**誰がやるか**: シニアメンバー・経営者（開発チームではない）

何を作るかを「荒削りだが具体的な」レベルに整える。
詳細な仕様書ではなく、**解決策の輪郭**を定義する。

```
Shaped Pitch（提案書）の内容:
- 問題: 何が困っているか
- アペタイト: 何週間かけてよいか（見積もりではなく予算）
- 解決策: スクリーンの大まかなスケッチ
- リスク: 把握しているウサギの穴（複雑になりそうな箇所）
- 対象外: スコープに含めないことの明示
```

**アペタイト（Appetite）が重要**:
「この機能に2週間かけてよい」という予算を先に決める。
見積もり（どれくらいかかるか？）ではなく、**価値に対して払う意思がある時間**。

### 2. Betting Table（賭けのテーブル）

**誰がやるか**: 経営者・Product Owner

Shaping された提案書（Pitch）をレビューし、次の6週間で何を作るかを決める。
**バックログはない**。採用されなかった提案は廃棄される（本当に重要なら再提案される）。

「バックログは過去の誰かが重要だと思ったものリスト」という考え方で、積み上げない。

### 3. Building（構築）

**誰がやるか**: 開発チーム（Designer + Engineer のペア）

6週間で自律的に完成させる。Shaping が決めた輪郭の中で、詳細は開発チームが判断する。

#### ヒルチャート（Hill Chart）

スクラムのバーンダウンチャート（残タスク量）の代わりに使う進捗管理ツール。

```
          ○ ここが頂上（問題を理解した・解決策が見えた）
         / \
未知    /   \    完了に向かって進んでいる
の坂 ↗       ↘ 既知の坂
```

- **上り坂**: 「どうやって作るかまだわかっていない」フェーズ
- **下り坂**: 「作り方がわかった・あとは実装するだけ」フェーズ

バーンダウンが「残量」を示すのに対し、ヒルチャートは「不確実性の解消度」を示す。

## サイクルの時間配分

```
6週間 Building
↓
2週間 Cooldown（冷却期間）
  - チームが自由に使える時間
  - バグ修正・技術的負債・やりたかった改善
  - 次の Betting Table の準備
↓
次の 6週間
```

**Cooldown が重要**: スプリントと違い、強制的な休息と自由な時間を確保する。
「常にフル回転」ではなく、持続可能なペースを維持する。

## いつ Shape Up を選ぶか

- スクラムのセレモニーにコストを感じている
- バックログが手に負えないほど積み上がっている
- チームが「何を作るべきか」の判断に関与できる
- 6週間で完結できる機能単位で開発できる

## 向いていない場面

- 規制・法令によるフェーズ承認が必要な開発
- 機能が超大規模で6週間に収まらない
- 「要件を全部先に決めたい」ステークホルダーがいる

## Shape Up の本質

「何を作らないかを決める勇気」と「チームへの信頼」が前提にある。
バックログを捨て、Shaping の質で価値の高いものだけを作る。
