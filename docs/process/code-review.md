# コードレビュー

マージ前にチームメンバーがコードを確認するプロセス。「バグを見つける作業」と思われがちだが、それは目的の一部に過ぎない。

## なぜ行うか

**品質の担保：** 実装者が見落としたバグ・設計の問題・セキュリティの穴を別の目で発見する。

**知識の共有：** コードベースの変更をチーム全体が把握できる。「自分が担当した部分しか知らない」状態を防ぐ。

**設計の改善：** レビューを前提にコードを書くと、説明できない設計をしにくくなる。書く側の設計力が上がる。

**チームの標準化：** コーディングスタイル・命名・アーキテクチャの判断基準がチームで揃っていく。

**新人の育成：** レビューコメントが具体的なフィードバックとして機能し、経験の浅いメンバーの成長を加速する。

## レビュアーが見るべき観点

すべてを一度に確認しようとすると焦点が散漫になる。観点を分けると抜け漏れが減る。

**正確性：** バグはないか。エッジケース（空文字・null・上限値）を見落としていないか。

**設計：** 変更の粒度は適切か。責務が適切に分離されているか。将来の変更に耐えられる構造か。

**セキュリティ：** 入力値は検証されているか。SQL インジェクション・XSS などの脆弱性がないか。認証・認可のチェックが漏れていないか。

**可読性：** 名前から意図が伝わるか。複雑な条件分岐にコメントが必要ではないか。

**テスト：** テストが変更に対して適切に書かれているか。重要なケースが抜けていないか。

## 良いレビューコメントの書き方

レビューコメントの質がチームの心理的安全性に直結する。

**何が問題かより、なぜ問題かを書く：**
```
Bad:  この関数名は良くない
Good: この関数は副作用があるので、get〜 より update〜 の方が意図が伝わります
```

**提案をセットにする：**
```
Bad:  ここはもっと良い書き方があります
Good: ここは〇〇パターンを使うとテストしやすくなります。例: ...
```

**深刻度を示す：** 必須の修正なのか、好みの問題なのかを明示する。

```
[blocking] セキュリティ上の問題があるため修正が必要です
[nit]      個人的な好みですが、変数名を〇〇にするとより明確かもしれません
[question] この設計の意図を教えてください
```

**実装者を否定しない：** 「なぜこう書いたか」と批判するのではなく、「こういう理由でこうすると良いかも」と代替案を示す。

## レビューを受ける側の心構え

レビューはコードへのフィードバックであり、人格への評価ではない。コメントに対して防御的にならず、疑問があれば理由を聞く。

**コンテキストを先に書く：** PR の説明に「何を・なぜ変えたか」「どこを特に見てほしいか」を書いておくと、レビュアーの負担が下がり質が上がる。

**小さく分ける：** 巨大な PR は全体を把握するのが難しく、レビューの質が下がる。1つの PR が1つの目的を持つよう分割する。

## PR サイズの目安

大きな PR ほどレビューの質が下がる。人間の集中力には限界がある。

```
〜200行:  詳細なレビューが可能
200〜400行: レビュアーへの負担が増す。分割を検討
400行〜:  レビューの実効性が著しく下がる。必ず分割する
```

大規模なリファクタリングは「構造の変更」と「ロジックの変更」を別 PR に分けると双方が楽になる。

## レビュープロセスの設計

**誰がレビューするか：** 最低1人のレビュアーが標準。セキュリティ・パフォーマンスに関わる変更は担当領域の詳しい人をレビュアーに加える。

**いつマージするか：** 全コメントへの返答を必須にするチームと、`[nit]` 等の軽微なものは任意にするチームがある。基準をチームで決めておく。

**自動化との組み合わせ：** Linter・型チェック・テスト・セキュリティスキャンを CI で自動化し、人間のレビューを「機械が検出できないもの」に集中させる。

## レビュー文化がない組織で起きること

レビューなしで開発を続けると以下が起きやすい。

- **属人化の加速：** 誰も触ったことのないコードが増え、担当者が辞めると詰む
- **設計の劣化：** 誰も指摘しないまま場当たり的な実装が積み重なる
- **セキュリティの穴：** 複数の目で見れば気づいた問題が本番に出る
- **オンボーディングの困難：** 新しいメンバーがコードベースを理解する手がかりが減る
