# Git ブランチ戦略

チームでコードを並行開発するとき、ブランチをどう使い・いつマージするかのルール。チームの規模・リリース頻度・デプロイの方法によって最適な戦略が変わる。

## GitFlow

Vincent Driessen が 2010年に提唱。長期リリースサイクルのプロジェクト向けに設計された。

```
main（本番）
  └── develop（開発統合）
        ├── feature/xxx   ← 機能開発
        ├── release/1.0   ← リリース準備
        └── hotfix/xxx    ← 本番バグ修正
```

**ブランチの役割:**
- `main`: 常に本番リリース済みのコード
- `develop`: 次のリリースに向けた統合ブランチ
- `feature/*`: 機能ごとに切って develop にマージ
- `release/*`: リリース前の最終調整。main と develop 両方にマージ
- `hotfix/*`: 本番の緊急バグ修正。main と develop 両方にマージ

**向いている場面:**
- スマホアプリ・パッケージソフトなど、バージョン管理が明確な製品
- リリースの審査・承認プロセスがある

**課題:** ブランチが多く複雑。Web サービスのように継続的にデプロイする開発には重すぎる。

## GitHub Flow

GitHub が自社の開発フローとして公開した、シンプルな戦略。

```
main（常にデプロイ可能）
  └── feature/xxx  ← 機能開発 → PR → main にマージ → 即デプロイ
```

**ルール:**
1. main は常にデプロイ可能な状態を保つ
2. 新しい作業はすべて main から feature ブランチを切る
3. PR を出してレビュー・CI を通過したら main にマージ
4. マージしたら即デプロイ

**向いている場面:** Web サービス・SaaS のように継続的デリバリーをする開発。小〜中規模チーム。

## トランクベース開発（Trunk-Based Development）

Google・Facebook などが採用する、最もシンプルな戦略。全員が `main`（trunk）に直接コミットするか、非常に短命なブランチを使う。

```
main ← 全員が毎日コミット（またはごく短命なブランチ）
```

**特徴:**
- ブランチの寿命は最大1〜2日
- Feature Flags で未完成の機能を隠しながら main に入れる
- CI が常に動いており、main が壊れたら即修正

**メリット:** マージ地獄がない。統合の問題を早期に発見できる。  
**デメリット:** Feature Flags の管理が必要。CI・テストカバレッジが充実していないと成立しない。

**向いている場面:** CI/CD が整備されている・テストカバレッジが高い・大規模チーム。

## 比較

| | GitFlow | GitHub Flow | トランクベース |
|---|---|---|---|
| ブランチ数 | 多い | 少ない | 最小限 |
| リリース頻度 | 低い（バージョン管理） | 高い（随時） | 非常に高い（毎日〜複数回） |
| 複雑さ | 高い | 低い | 低い（ただし規律が必要） |
| 向いている規模 | 小〜中 | 小〜中 | 中〜大 |

## Feature Flags との関係

トランクベース開発と GitHub Flow では、未完成の機能を main に入れるために [フィーチャーフラグ](/deployment/feature-flags) を使う。

```
// 機能が完成していなくてもフラグで隠せる
if (featureFlags.isEnabled('new_checkout')) {
    return <NewCheckout />;
}
return <OldCheckout />;
```

## いつどれを選ぶか

```
スマホアプリ・パッケージソフト → GitFlow
Web サービス・小〜中規模チーム → GitHub Flow（最も一般的）
CI/CD が整備された大規模チーム → トランクベース開発
```

迷ったら **GitHub Flow** から始めるのが現実的。シンプルで多くのチームに合う。
