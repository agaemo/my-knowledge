# PoC・Prototype・MVP・Spike

「本格的に作る前に、小さく試す」アプローチの総称。似た言葉が複数あるが、**何を検証したいか**で使い分ける。

## 4つの概念

### PoC（Proof of Concept：概念実証）

**「これは技術的に実現できるか」を検証する。**

品質・設計・保守性は度外視して、可能性だけを確かめる。  
検証したら基本的に**捨てる**前提で作る。

```
問い: 「この外部 API を使って○○ができるか？」
      「この処理が要求されたレイテンシ内で収まるか？」
      「LLM でこの精度の分類ができるか？」

PoC で確認すること: できる / できない
PoC で確認しないこと: 保守性・テスト・エラーハンドリング
```

**失敗した PoC は成功。**「やってみたら無理だった」という情報は、本格開発に入る前に得られた最も価値ある知見。

### Prototype（プロトタイプ）

**「これはどういう体験・見た目になるか」を検証する。**

ユーザーや関係者に見せて、方向性を確認するのが主目的。UI モックアップから動くデモまで幅がある。

```
問い: 「このインターフェースで使いやすいか？」
      「この画面の流れで理解できるか？」

Prototype で確認すること: UX・デザイン・ユーザーの反応
Prototype で確認しないこと: スケーラビリティ・セキュリティ
```

PoC との違い：PoC は技術的実現可能性、Prototype はユーザー体験。両方作ることもある。

### MVP（Minimum Viable Product：最小限の製品）

**「ユーザーはこれを欲しがるか」を検証する。**

動くプロダクトとして実際のユーザーに提供し、フィードバックを得る。捨てる前提ではなく、**育てる前提**で作る。

```
問い: 「この機能にユーザーはお金を払うか？」
      「この解決策は本当に問題を解決しているか？」

MVP の条件:
  - 本物のユーザーが実際に使える
  - 最小限の機能だが、価値が伝わる
  - 学習が得られる設計になっている
```

**「最小限」は「粗雑」ではない。** MVP は捨てないので、基本的な品質・セキュリティは必要。

### Spike（スパイク）

Agile（特に XP）の用語。**時間を区切って技術的な不確実性を調査する**タスク。

```
Sprint の計画時: 「この機能の工数が見積もれない」
  → Spike: 2日間で調査して見積もれる状態にする
  → 結果を次の Sprint の計画に使う
```

PoC と近いが、Spike はより短期間・スコープが狭い。チケット（Issue）として管理される。

## 使い分け

| | 目的 | 成果物 | 捨てるか |
|---|---|---|---|
| **PoC** | 技術的実現可能性の確認 | 動くが汚いコード・計測結果 | 基本的に捨てる |
| **Prototype** | UX・方向性の確認 | デモ・モックアップ | 基本的に捨てる |
| **MVP** | 市場・ユーザーの反応を確認 | 実際に動くプロダクト | 育てる |
| **Spike** | 見積もり・設計の不確実性解消 | 調査結果・サンプルコード | 捨てる |

## よくある失敗

**PoC がそのまま本番になる**  
「とりあえず動いている」PoC が、時間・予算の都合でそのまま本番に使われる。品質・テスト・設計を無視して作ったコードが技術的負債の温床になる。PoC は明示的に「捨てる」判断をしてから本番実装を始める。

**MVP が最小でなくなる**  
「あれも入れよう、これも必要」と機能が膨らみ、リリースが遅れる。MVP は「これがないと最低限の価値が伝わらないか？」を問い続けて削ることが重要。
