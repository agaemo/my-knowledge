# アジャイル開発

## 何か

変化する要件に対応しながら、動くソフトウェアを継続的に届ける開発スタイル。
2001年の「アジャイルソフトウェア開発宣言」が起点。

## アジャイル宣言の4つの価値

```
プロセスやツール           より  個人と対話
包括的なドキュメント       より  動くソフトウェア
契約交渉                   より  顧客との協調
計画に従うこと             より  変化への対応
```

「右側にも価値はある。しかし左側をより重視する」という宣言。

## スクラム（Scrum）

アジャイルの代表的な実践フレームワーク。

### 役割

| 役割 | 責務 |
|---|---|
| Product Owner | 何を作るかを決める。バックログの優先順位を管理 |
| Scrum Master | プロセスを守る・チームの障害を取り除く |
| Development Team | 実際に作る（3〜9人が理想） |

### スプリント

**2週間（1〜4週間）の固定サイクル**で動くソフトウェアを届ける。

```
Sprint Planning → 開発 → Sprint Review → Sprint Retrospective → 次のスプリント
```

- **Sprint Planning**: このスプリントで何を作るかを決める
- **Daily Scrum**: 毎朝15分。昨日やったこと・今日やること・障害を共有
- **Sprint Review**: 作ったものをステークホルダーに見せる（デモ）
- **Retrospective**: プロセスの振り返り。「うまくいったこと・改善点」

### バックログ

```
Product Backlog        ← Product Owner が管理。全要件の優先順位付きリスト
    ↓ スプリント計画で選択
Sprint Backlog         ← このスプリントで対応するタスク
    ↓ 開発
Done（完成の定義を満たしたもの）
```

**完成の定義（Definition of Done）を明確にする**。
「コードが書けた」ではなく「テストが通り・レビューが済み・デプロイ可能な状態」など。

## カンバン（Kanban）

スプリントを使わず、継続的なフローで開発する。
WIP（Work In Progress）制限でボトルネックを可視化する。

```
Backlog → In Progress → Review → Done
           [WIP: 2]     [WIP: 1]
```

WIP 制限を設けることで「たくさん着手して全部中途半端」を防ぐ。

### スクラムとカンバンの使い分け

| | スクラム | カンバン |
|---|---|---|
| 向いている | 新機能開発・チームで動く | 運用・サポート・割り込みが多い |
| サイクル | スプリント固定 | 継続的フロー |
| 計画 | スプリント単位 | 随時 |

## よくある失敗

### スクラムの形だけ真似る（Scrum-but）

「スプリントはやるが Retrospective はしない」「Daily Scrum が進捗報告会になっている」
→ セレモニーには意味がある。省くならなぜか理由を持つ。

### バックログが肥大化する

誰も触らない要件が積み上がり、管理コストだけかかる。
→ 定期的に不要なものを削除する（バックログリファインメント）。

### ベロシティで進捗を測る

スプリントポイントの消化速度を KPI にすると、ポイント盛りが始まる。
→ ベロシティは計画ツール。管理指標にしない。

### 「アジャイルだから設計しない」という誤解

アジャイルは「変化に対応する」であり「設計しない」ではない。
大きな設計判断（アーキテクチャ・DB設計）は前倒しで行う。
