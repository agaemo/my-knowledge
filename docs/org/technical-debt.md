# 技術的負債（Technical Debt）

将来の開発速度と引き換えに、今の開発を早める選択をすること。Ward Cunningham が 1992年に提唱した比喩。

> 「今は簡単な実装で進め、後でより良い設計に直す」は、借金をして後で利子付きで返すことと同じ。

## なぜ存在するか（比喩として）

金融の負債と同様、技術的負債も意図的に取ることが合理的な場合がある。PoC を早く出したい・締め切りに間に合わせたい・要件が不確かなうちに実装するといった場面では、負債を取ることで短期的な価値を得られる。問題は**返済されないまま積み上がること**。

```
負債なし: 1機能を実装するのに 3日
負債あり: 1機能を実装するのに 6日（コードの複雑さが増すほど遅くなる）

→ 利子 = 将来の開発速度の低下
```

## Fowler の技術的負債マトリクス

Martin Fowler は「意図的かどうか」「無謀か慎重か」の2軸で分類した。

|  | **無謀**（Reckless） | **慎重**（Prudent） |
|---|---|---|
| **意図的**（Deliberate） | 「設計なんて考える時間はない」 | 「今は早く出す。後で直す」 |
| **無意識**（Inadvertent） | 「レイヤー分割って何？」 | 「これがベストプラクティスだとやっと気づいた」 |

慎重・意図的な負債（右上）は戦略的に取ることがある。無謀・無意識な負債（左下）は避けるべき。

## 主な種類

**設計の負債**  
場当たり的な設計が積み上がり、変更のたびに影響範囲が読めなくなる。

**テストの負債**  
テストがないまま変更が続く。リファクタリングが怖くなる。

**ドキュメントの負債**  
「なぜこうなっているか」が誰もわからない。新メンバーのキャッチアップコストが増大する。

**依存関係の負債**  
古いライブラリ・EOL になったフレームワーク。セキュリティリスクと移行コストが積み上がる。

## なぜ問題になるか

技術的負債は複利で増える。

```
機能追加のたびに複雑な部分を避けて書く
  → さらに複雑になる
  → 次の機能追加がさらに難しくなる
  → バグが増える
  → リリース頻度が下がる
  → 競争力が落ちる
```

「レガシーシステム」と呼ばれるものの多くは、返済されなかった技術的負債が積み上がった結果。

## 返済の戦略

**継続的なリファクタリング**  
機能追加のついでに周辺を少しずつ改善する。「ボーイスカウトルール（来たときよりも綺麗にして去る）」。大きな変更ではなく、日常の習慣として行う。

**ストラングラーフィグパターン**  
旧システムを少しずつ新しい実装に置き換えていく。一気に書き直さない。

**負債の可視化**  
返済するには「どこに負債があるか」を把握する必要がある。コードの複雑度計測・テストカバレッジ・依存関係の古さを定期的に確認する。

**返済バジェットを確保する**  
スプリントの一定割合（例: 20%）を技術的負債の返済に充てる。機能開発だけを優先し続けると返済の機会が永遠に来ない。

## いつ意識するか

- 機能追加のたびに「なぜここはこんな設計なのか」と思うことが増えたとき
- バグ修正が他のバグを生むことが増えたとき
- 新しいメンバーがコードを理解するのに異常に時間がかかるとき
- リリース頻度が下がってきたとき
