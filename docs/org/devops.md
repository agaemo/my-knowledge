# DevOps

## 何か

開発（Development）と運用（Operations）の壁を取り除き、ソフトウェアを**速く・安全に・継続的に**届けるための文化・プラクティス・ツールの総体。

2009年に Patrick Debois が "DevOpsDays" を開催したことで広まった。特定のツールや手法ではなく、**考え方の変革**が本質。

## なぜ存在するか

伝統的な組織では開発チームと運用チームが分離していた。

| | 開発チーム | 運用チームの関心 |
|---|---|---|
| 目標 | 新機能を早くリリース | システムの安定稼働 |
| リスク認識 | 変更 = 価値 | 変更 = リスク |
| 結果 | 「動いているのに運用が止める」 | 「開発が壊れたものを渡してくる」 |

この対立が[サイロ化](./silo.md)を生み、リリースが遅くなる。DevOps はこの構造的な問題を解消する。

## 中心的なプラクティス

**CI/CD（継続的インテグレーション / デリバリー）**  
コードをマージするたびに自動でビルド・テスト・デプロイする。手動作業を減らし、問題を早期に発見する。

**Infrastructure as Code（IaC）**  
インフラ構成をコードで管理する（Terraform・Ansible など）。再現性が上がり、環境の差異によるバグが減る。

**モニタリングとオブザーバビリティ**  
開発チームが本番の挙動を常に観察できる状態にする。「リリースして終わり」から「リリースして観る」へ。→ [Observability](/observability/)

**フィーチャーフラグ**  
コードのデプロイとフィーチャーの公開を分離する。問題が起きたとき即座に無効化できる。→ [フィーチャーフラグ](/deployment/feature-flags)

## SRE との関係

SRE（Site Reliability Engineering）は Google が提唱した、DevOps を実践するための**具体的な実装方法**の一つ。

| | DevOps | SRE |
|---|---|---|
| 性格 | 文化・哲学・ムーブメント | 具体的なロール・プラクティス・指標 |
| 提唱者 | コミュニティ主導 | Google |
| 主な関心 | 開発と運用の協力関係 | 信頼性の定量管理（SLO・エラーバジェット） |

「DevOps をどう実践するか」の答えの一つが SRE。→ [SRE](/sre/)

## コンウェイの法則・サイロとの関係

DevOps が目指すのは、Dev と Ops のサイロを解消すること。  
[コンウェイの法則](./conways-law.md)の観点では、Dev と Ops が分離した組織はシステムも「開発用」と「本番用」に分断されやすい。DevOps はチーム構造を変えることでアーキテクチャも変える。

## いつ使うか

- リリース頻度を上げたい（週次→日次→随時）
- 本番障害の原因が「環境差異」や「手動手順のミス」に集中している
- 開発チームが「自分たちのコードが本番でどう動いているか知らない」状態になっている
