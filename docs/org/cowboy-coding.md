# カウボーイ・コーディング

## 何か

設計・レビュー・テストなしに、個人の裁量だけでコードを書くスタイル。
「誰にも相談せず、好きに実装する」という行動パターンを指す。

チームや組織における行動スタイルの問題であり、コード・アンド・フィックスがプロセスの欠如を指すのに対して、カウボーイ・コーディングは個人の振る舞いの問題。

## なぜ起きるか

### 構造的な要因

- 初期フェーズの小規模チームで「とにかく速く動かす」文化が続く
- レビューや設計の工程がプロセスとして定まっていない
- 特定の人物が「この領域はあの人しかわからない」と属人化している

### 心理的な要因

- 設計・相談・レビューが「遅い」「面倒」と感じる
- 自分が最も正しい実装を知っていると確信している
- 承認を待つより自分で判断した方が速いと学習している

## 何が問題か

### 属人化

コードの意図や文脈が書いた本人の頭の中にしかない。担当者が離脱した瞬間にブラックボックス化する。

### 品質の劣化

他者の目が入らないため、設計上の問題・セキュリティ上の問題が気づかれないまま本番に入る。

### 技術的負債の蓄積

「動けばよい」の判断を繰り返すことで、変更しにくいコードが積み重なる。

### チームの分断

特定のメンバーだけが特定の領域を「所有」する状態になり、レビューや協力が起きにくくなる。

## いつ許容されるか

カウボーイ・コーディング的なアプローチが合理的な場面は限られるが存在する。

| 場面 | 条件 |
|---|---|
| PoC・プロトタイプ | 本番に持ち込まない前提で、捨てることが決まっている |
| 個人プロジェクト | 影響範囲が自分だけ |
| 緊急の本番障害対応 | 仮対応として、後で正式な修正に置き換えることが明確 |

いずれも「後で正式なプロセスに戻す」が前提であり、恒常化すると問題になる。

## コード・アンド・フィックスとの関係

| | カウボーイ・コーディング | コード・アンド・フィックス |
|---|---|---|
| 視点 | 個人の行動スタイル | 開発プロセスの欠如 |
| 問題の所在 | チームワーク・レビューの欠如 | 設計・テスト・計画の欠如 |
| 発生する場所 | チーム開発の中 | 開発プロセス全体 |

両者は同時に起きることが多く、カウボーイ・コーディングを放置するとコード・アンド・フィックスの状態に陥る。
