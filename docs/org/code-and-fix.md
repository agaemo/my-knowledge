# コード・アンド・フィックス

## 何か

事前の設計や計画なしにコードを書き始め、問題が出たら修正を繰り返すだけの開発モデル。
「書いては直す」サイクルに名前をつけたもので、正式なプロセスモデルが存在しない状態を指す。

```
コードを書く → 動かす → 問題が出る → 直す → 動かす → 問題が出る → ...
```

「Hack and Fix」とも呼ばれる。

## 歴史的位置づけ

1970年代以前、ソフトウェア工学が確立される前の開発スタイル。
Winston Royce がウォーターフォールモデルを発表したのは 1970 年で、その論文では「設計なしのコーディング」の危険性を指摘している。

Barry Boehm は 1988 年の螺旋モデルの論文で、コード・アンド・フィックスを「最も原始的なプロセスモデル」と位置づけた。

| プロセスモデルの進化 | 年代 |
|---|---|
| コード・アンド・フィックス（プロセスなし） | 〜1970年代 |
| ウォーターフォール（フェーズ分割） | 1970年代〜 |
| 螺旋モデル（リスク駆動） | 1980年代〜 |
| アジャイル（反復・適応） | 2000年代〜 |

## 何が問題か

### 要件の不明確さが解消されない

コードを書き始める段階で何を作るべきかが定まっていないため、方向性が定まらないまま実装が積み重なる。

### 修正コストが増大する

後から問題が発覚するほど修正コストが指数的に増える。Barry Boehm の研究では、テストフェーズでのバグ修正は要件段階の 100 倍のコストがかかるとされる。

### 品質保証の手段がない

テスト計画がないため、「動く」と「正しい」の区別がつかない。

### 技術的負債が蓄積する

「動けばよい」で積み重なったコードは内部品質が低く、変更コストが上がり続ける。

## なぜ今も起きるか

正式なプロセスモデルを採用していても、以下の状況でコード・アンド・フィックスに陥りやすい。

- 締め切りのプレッシャーで設計をスキップする
- 要件定義の代わりに「とりあえず動くものを見せてから決める」で始める
- PoC が本番コードになる（PoC の目的が失われる）

## アジャイルとの混同

「短い反復でコードを書いて改善する」という外形が似ているため混同されやすいが、本質的に異なる。

| | コード・アンド・フィックス | アジャイル |
|---|---|---|
| 計画 | なし | スプリント計画がある |
| テスト | 後付けか省略 | TDD・自動テストが前提 |
| 反省・改善 | なし | レトロスペクティブ |
| 完了の定義 | 動けば終わり | Definition of Done がある |

反復することとプロセスがあることは別である。
