# サイロ化

## 何か

組織内の部門・チームが孤立し、情報・知識・リソースを互いに共有しない状態。  
名前の由来は穀物倉庫（サイロ）の形状 — 縦長で、隣と繋がっていない。

## なぜ起きるか

| 要因 | 説明 |
|---|---|
| **専門化** | フロントエンド・バックエンド・インフラなど専門分野で分かれると、自然と壁ができる |
| **インセンティブの不一致** | 部門ごとのKPIが異なると、横断協力より自部門の最適化を優先する |
| **組織の成長** | 小さなチームが分割・拡張されるにつれ、コミュニケーションコストが増大する |
| **地理・時間的分断** | リモートワーク・タイムゾーン差がさらに壁を強化する |

サイロ化は意図して作られるものではなく、**組織が成長する過程で自然発生する**。

## 何が問題か

**重複コスト**  
複数チームが同じ機能をそれぞれ実装する。共通基盤が存在せず、変更も二重に発生する。

**エンドツーエンドのオーナーシップ欠如**  
「この機能は誰が責任者か」が不明確になる。ユーザー体験をまたぐ機能で一貫性が失われる。

**意思決定の遅さ**  
チームをまたぐ変更には多くの承認が必要になり、リードタイムが伸びる。

**コンウェイの法則による影響**  
[コンウェイの法則](./conways-law.md)の通り、組織のサイロはそのままシステムの分断として現れる。  
フロントエンド・バックエンド・インフラが別サイロなら、API の境界がそこに固定化される。

## どう解消するか

**クロスファンクショナルチーム**（機能横断チーム）  
フロントエンド・バックエンド・デザイン・QA を1チームに集める。  
エンドツーエンドのオーナーシップが生まれ、サイロを構造的に排除できる。

**逆コンウェイ戦略**  
目標とするアーキテクチャに合わせてチームを組み直す。  
チーム構造を変えずにアーキテクチャだけ変えようとすると、サイロが残り失敗しやすい。

**Platform team の整備**  
共通基盤（CI/CD・認証・監視など）を Platform team が提供することで、各 Stream-aligned team が重複実装せず自律的に動ける。→ [Team Topologies](./team-topologies.md)

## サイロ化のサイン

- 「この機能は○○チームの担当か△△チームの担当か」の議論が頻発する
- 似た機能が複数箇所に別々に実装されている
- ユーザー向けの変更に複数チームの承認が必要になる
- インシデント対応でオーナーが見つからない
- チーム間の会議が多いのに意思決定が遅い
