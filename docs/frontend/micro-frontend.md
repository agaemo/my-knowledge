# マイクロフロントエンド

フロントエンドアプリケーションを機能単位の独立したピースに分割し、それぞれが独立して開発・デプロイできる設計アーキテクチャ。マイクロサービスの思想をフロントエンドに適用したもの。

## なぜ存在するか

バックエンドをマイクロサービス化しても、フロントエンドが「モノリシックな SPA」のままだとボトルネックになる。

```
バックエンド: 注文サービス / 商品サービス / ユーザーサービス（独立）
フロントエンド: [全機能を含む1つの大きな SPA]
  → 全チームが同じコードベースを触る
  → 1つの変更が全体の再デプロイを必要とする
```

チームが増えるにつれてコンフリクト・リリース調整・認知コストが増大する。

## 実現方式

**ビルド時統合：** 各チームがパッケージとして公開し、シェルアプリが依存として取り込む。変更のたびにシェルの再ビルドが必要。

**iframe：** 各マイクロフロントエンドを iframe で埋め込む。独立性が高いが、UX・パフォーマンスの課題がある。

**Web Components：** カスタム要素として実装し、フレームワーク非依存で埋め込む。

**モジュールフェデレーション（Module Federation）：** Webpack 5 の機能。実行時に別アプリのコードを動的にロードする。現在最も主流の手法。

```
シェルアプリ（Shell）
  ├── ナビゲーション（チームA がデプロイ）
  ├── 商品一覧（チームB がデプロイ）
  └── カート（チームC がデプロイ）
```

## メリット

- **独立したデプロイ：** チームごとに自分のタイミングでリリースできる
- **技術の自由度：** チームごとに異なるフレームワークを選べる
- **チームの自律性：** 境界づけられた機能を1チームが所有する

## デメリット

- **初期コストの高さ：** シェルとマイクロフロントエンドの調整インフラが複雑
- **UX の一貫性：** デザインシステムを共有しないと見た目がバラバラになる
- **重複バンドル：** React などの共通ライブラリが複数回ロードされる（Module Federation で緩和可能）
- **横断的関心事：** ルーティング・状態共有・認証など、チーム間の設計調整が必要

## いつ使うか

```
使うケース:
  ✓ 複数の独立したチームが同じフロントエンドを開発している
  ✓ 機能ごとに独立したリリースサイクルが必要
  ✓ バックエンドがすでにマイクロサービス化されている

使わないケース:
  ✗ チームが1〜2チーム（調整コストに見合わない）
  ✗ 小〜中規模アプリ（モノリシック SPA で十分）
  ✗ UX の一貫性が最重要で、共通化の恩恵が小さい
```

マイクロフロントエンドはコンウェイの法則の表れでもある。[チームトポロジー](/org/team-topologies) でストリームアラインドチームを複数設ける場合に自然に発生する。
