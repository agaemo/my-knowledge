# フロントエンド JS フレームワーク

フロントエンド開発で使われる主要な JavaScript/TypeScript フレームワーク・ライブラリの概観。バージョン固有の機能よりも「設計思想」「どんな文脈で選ばれるか」を整理する。

## React

Meta（旧 Facebook）が開発・OSS 公開した UI ライブラリ。「ライブラリ」であり、ルーティングや状態管理は自分で選ぶ設計。エコシステムが最大規模で、求人数・ライブラリ数ともに最多。

コンポーネントという単位で UI を組み立てる考え方を広め、現代のフロントエンド開発の基盤概念になった。

**Next.js**（Vercel 製）が React の事実上の標準メタフレームワークとして定着しており、SSR・SSG・ファイルベースルーティングを提供する。

```
向いている用途: SPA・大規模プロダクト・エコシステムの豊富さが必要な場合
採用される背景: エンジニアの採用しやすさ・ライブラリの選択肢の広さ
```

## Vue

Evan You（元 Google エンジニア）が個人で開発した後 OSS 化したフレームワーク。React より学習コストが低く、HTML・CSS・JS の知識をそのまま活かしやすい設計。日本・中国・東南アジアでの採用が多い。

単一ファイルコンポーネント（.vue ファイル）にテンプレート・スクリプト・スタイルをまとめる書き方が特徴的。

**Nuxt**（Vue のメタフレームワーク）が Next.js に相当するポジションを担う。

```
向いている用途: 中小規模 Web アプリ・フロントエンド経験が浅いチーム
採用される背景: 学習コストの低さ・日本市場での普及率
```

## Angular

Google が開発・メンテナンスする本格的なフレームワーク。「フレームワーク」として型システム・ルーティング・状態管理・テストが最初からセットになっている。設計の自由度は低い代わりに、大人数チームでコードが散らかりにくい。

TypeScript が前提。企業向けシステムやチームが大きいプロダクトで選ばれることが多い。

```
向いている用途: エンタープライズ・大規模チーム・長期メンテナンスが前提のシステム
採用される背景: 設計の統一・TypeScript 前提の強制力
```

## Svelte / SvelteKit

コンパイル型フレームワーク。React・Vue が実行時に仮想 DOM を使うのに対し、Svelte はビルド時にコンパイルして純粋な JS を出力するため、ランタイムのオーバーヘッドがない。コードが短く書けることでも知られる。

**SvelteKit** がメタフレームワーク。パフォーマンスを重視するプロジェクトや、小規模で軽量なアプリに向く。

```
向いている用途: パフォーマンス重視・小〜中規模アプリ・組み込み系 UI
採用される背景: バンドルサイズの小ささ・書き方のシンプルさ
```

## Solid.js

React に似た API を持ちながら、仮想 DOM を使わないフレームワーク。React の書き方に慣れたエンジニアが移行しやすい設計で、パフォーマンスが高い。採用事例はまだ少ないが注目度は高い。

```
向いている用途: React 経験者がパフォーマンスを追求したい場合
採用される背景: React 的な書き方のまま速度を得たい
```

## フレームワーク選択の判断軸

| 判断軸 | 考慮すること |
|---|---|
| チームの既存スキル | 知っているフレームワークを使うのが最も安全 |
| 採用市場 | React > Vue > Angular の順に求人が多い傾向 |
| プロジェクト規模 | 小規模なら Vue・Svelte / 大規模なら React・Angular |
| エコシステム依存度 | 豊富なライブラリが必要なら React が有利 |
| パフォーマンス要件 | Svelte・Solid が有利。React/Vue でも最適化次第 |

## メタフレームワークとの関係

フレームワーク単体（React・Vue 等）は UI の構築が中心で、SSR・ルーティング・ビルド設定は自前になる。メタフレームワークはこれらを統合して提供する。

| フレームワーク | メタフレームワーク |
|---|---|
| React | Next.js・Remix |
| Vue | Nuxt |
| Svelte | SvelteKit |
| Angular | Analog（小規模） |

新規プロジェクトではフレームワーク単体より**メタフレームワークから始める**のが現在の主流。[レンダリング戦略](/frontend/rendering-strategy)（SSR・SSG・CSR）を柔軟に選べるため。
