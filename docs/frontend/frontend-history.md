# フロントエンドの歴史

Web フロントエンドがどのように進化してきたかの流れ。現代のフレームワーク・ツールがなぜ存在するかを理解する背景として。

## 1990年代 — 静的 HTML とサーバーサイドレンダリング

Web は「文書を配信するシステム」として始まった。HTML を書いてサーバーに置けば誰でも見られる。動的なページが必要な場合は **CGI**（Common Gateway Interface）で Perl・C のスクリプトがサーバー側で HTML を生成して返す方式が主流だった。

JavaScript は 1995 年に Netscape が「10日間で作った」言語として登場。当初はフォームのバリデーション程度の簡単な用途を想定していた。

## 1996〜2004 — ブラウザ戦争と Flash の全盛

Internet Explorer と Netscape がシェアを争った「ブラウザ戦争」の時代。各ブラウザが独自の機能を実装し、同じ JavaScript・CSS が別のブラウザで動かない問題が慢性化した。

**Flash**（旧 Macromedia、のち Adobe）がこの時代のリッチな Web 体験を担った。アニメーション・動画・インタラクティブコンテンツはほぼ Flash で実現されており、プラグインのインストールが前提だった。

## 2005 — AJAX の「発見」

**AJAX（Asynchronous JavaScript and XML）** という言葉を作ったのは Jesse James Garrett（2005年）。ただし技術自体（`XMLHttpRequest`）は 1999 年から存在していた。

Google Maps と Gmail が「ページ全体を再読み込みせずにデータを取得・更新する」体験を一般ユーザーに見せたことで、その可能性が広く認識された。

```
従来: リンクをクリック → ページ全体を再読み込み
AJAX: ボタンを押す → 差分データだけ取得 → ページの一部だけ更新
```

これが「Web アプリケーション」という概念が広まるきっかけになった。

## 2006〜2012 — jQuery の時代

**jQuery**（2006 年）はブラウザ差異を吸収する薄い抽象レイヤー。`$('セレクタ')` で要素を操作する書き方は、複雑なブラウザ対応を隠蔽し、誰でも JavaScript を書けるようにした。

10 年以上にわたって「Web の jQuery 前提」が常識となり、WordPress プラグインを含めあらゆる場所で使われた。現在も多くのレガシーシステムに残っている。

## 2009〜2012 — Node.js と SPA の黎明

**Node.js**（2009 年）は JavaScript をブラウザの外—サーバーサイド—で動かすランタイム。「フロントもバックも JS で書ける」という可能性が開いた。

SPA（Single Page Application）の概念も生まれた。**Backbone.js・Knockout.js・Angular 1・Ember** などが登場し、「HTML を丸ごとブラウザ側で組み立てる」アーキテクチャが試みられた。ただしこの頃のツールは複雑で学習コストが高かった。

## 2013〜2015 — React・Vue の登場と仮想 DOM

**React**（2013 年、Meta）が「仮想 DOM」という概念を持ち込んだ。DOM 操作を直接書かず、「UI の状態はこうあるべき」という宣言的な記述からフレームワークが差分を計算して更新する。この考え方が以後のフロントエンドの主流になった。

**Vue**（2014 年、Evan You 個人）は React より学習コストが低く、既存の HTML ファイルにも部分的に導入できる設計で急速に普及した。

**ES6（2015 年）** で JavaScript 自体が大幅に近代化され（`let/const`・アロー関数・モジュール・Promise など）、jQuery に頼らなくても書けるようになった。

## 2015〜2018 — Flash の終焉と HTML5 の完成

Apple が iPhone で Flash を拒否した判断（2010 年）が徐々に効いてきた。スマートフォンの普及でモバイルブラウザのシェアが急増し、Flash なしでも HTML5・CSS3・JS で動画・アニメーションが実現できるようになっていった。

Adobe は 2017 年に Flash の 2020 年終了を宣言。長らくリッチ表現の代名詞だった Flash は 25 年の歴史を終えた。

## 2018〜現在 — SSR の復権とビルドツールの進化

SPA が普及する一方、**SEO・初期表示速度・Core Web Vitals** の問題が表面化した。ページを全部ブラウザで組み立てると、最初に何も表示されない時間が長くなる。

**Next.js・Nuxt** などのメタフレームワークが SSR（サーバーサイドレンダリング）と SPA の長所を組み合わせる形で台頭し、「SSR の復権」と呼ばれた。

ビルドツールも進化し、**Webpack** を置き換える形で **Vite**（2020 年）が登場。ネイティブ ESM を活用した高速な開発体験が標準になった。

**TypeScript** も 2018 年前後から一気に普及し、大規模フロントエンド開発では事実上の標準になった。

## フロントエンドの複雑化

この 30 年でフロントエンドは「飾りとしての HTML」から「UI ロジックの主戦場」へ変わった。ビルド・バンドル・型システム・テスト・パフォーマンス最適化が必要になり、専業エンジニアが存在する理由になった。

一方で「シンプルな HTML・CSS・JS で十分なサイト」まで重いフレームワークを使う過剰設計も問題視され、Astro・Eleventy のような「必要な部分だけ JS を使う」方向への揺り戻しも起きている。
