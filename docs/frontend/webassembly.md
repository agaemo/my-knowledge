# WebAssembly（Wasm）

ブラウザ上でネイティブに近い速度でコードを実行するためのバイナリ形式の命令セット。2017 年に W3C が標準化。

## なぜ生まれたか

JavaScript はブラウザの「共通言語」として広まったが、動的型付け・インタープリタ実行という性質上、計算負荷の高い処理には限界があった。

```
JS の限界が顕在化したユースケース:
  - 3D グラフィックス・ゲームエンジン
  - 動画・音声のリアルタイム編集
  - CAD・画像処理
  - 暗号処理・科学計算
```

Mozilla が 2013 年に **asm.js** で「JS のサブセットをネイティブコンパイルで高速化する」実験を行い、その発展形として Google・Mozilla・Microsoft・Apple が共同で WebAssembly を設計した。

## 仕組み

Wasm はバイナリ形式で配信され、ブラウザの仮想マシンで実行される。JS より高速に動作する理由は主に2つ。

**型情報が確定している：** JS は実行時まで型が不明なため JIT コンパイラが推測しながら最適化する。Wasm はコンパイル済みで型が固定されているため、そのままネイティブ命令に変換できる。

**解析コストが低い：** テキスト形式の JS をパースする処理が不要。バイナリを即座にデコードして実行できる。

```
ソース（C++・Rust・Go など）
   ↓ コンパイル
WebAssembly バイナリ（.wasm）
   ↓ ブラウザの Wasm エンジン
ネイティブ命令として実行
```

## JavaScript との関係

Wasm は JS の置き換えではなく、**重い処理を担当するモジュールとして JS と連携する**設計。

- DOM 操作・ネットワーク・UI イベントは引き続き JS が担う
- 計算負荷の高い処理（画像処理・物理演算など）を Wasm に任せる
- JS から Wasm の関数を呼び出す形が基本

## 代表的な採用例

**Figma：** デザインツールのレンダリングエンジンを C++ で書き、Wasm にコンパイルして提供。ブラウザ上で複雑なベクター描画を高速に動かす。

**Adobe Photoshop Web：** デスクトップ版の C++ コードを Wasm にコンパイルしてブラウザ版を実現。数十年分のコードをゼロから書き直さずに Web 対応できた。

**Unity WebGL：** ゲームエンジン Unity のコードを Wasm にコンパイルしてブラウザで動かす。

**AutoCAD Web：** 30 年以上の C++ コードベースをそのまま Wasm 経由でブラウザで動かしている。

## ブラウザの外への展開

当初はブラウザ専用だったが、**WASI**（WebAssembly System Interface） の標準化でブラウザ外でも動くようになった。

**Cloudflare Workers・Fastly Compute** などのエッジランタイムで Wasm を実行できる。「一度コンパイルすれば、ブラウザでも Edge でもサーバーでも動く」という新しいクロスプラットフォームの形が生まれつつある。

Docker の創設者 Solomon Hykes は「WASI が 2008 年にあれば Docker を作る必要はなかった」とコメントしており、コンテナに代わるポータビリティの仕組みとして注目されている。

## 使われる言語

Wasm のバイナリを出力できる主な言語：

| 言語 | 特徴 |
|---|---|
| **Rust** | Wasm との親和性が最も高い。ランタイムが小さい |
| **C / C++** | Emscripten でコンパイル。既存資産の移植に使う |
| **Go** | 公式サポート。バイナリサイズが大きめ |
| **AssemblyScript** | TypeScript 風の構文で Wasm を直接書ける |

## 現状の制約

- DOM へ直接アクセスできない（JS を経由する必要がある）
- デバッグが JS より難しい
- バイナリサイズが大きくなりやすい（ランタイム込みでの配信になるため）
- すべての処理が速くなるわけではない。小さな計算なら JS の方が速いケースもある
