# レンダリング戦略

Webアプリのページをどこで・いつHTMLを生成するかのアプローチ。パフォーマンス・SEO・開発体験のトレードオフで選択する。

## なぜ複数の戦略が存在するか

初期のWebはサーバーがHTMLを生成して返す（SSR）のみだった。JavaScriptの進化でクライアントサイドレンダリング（CSR）が登場し、SPAが普及した。しかしSPAはSEO・初期表示速度に課題があり、それを解決するためにSSG・ISRなどが生まれた。

## CSR（Client-Side Rendering）

ブラウザがJavaScriptを実行してHTMLを生成する。サーバーは最小限のHTMLとJSバンドルだけを返す。

```
1. サーバー → 空のHTML + JS を返す
2. ブラウザ → JSを実行してDOMを構築
3. ブラウザ → APIからデータ取得してレンダリング
```

**メリット:**
- 初回以降のページ遷移が速い（SPA）
- サーバー負荷が低い
- インタラクティブなUIを構築しやすい

**デメリット:**
- 初期表示が遅い（JSのダウンロード・実行後にコンテンツが表示される）
- SEOが弱い（クローラーがJS実行前のHTMLを見る場合がある）
- JavaScriptが無効な環境では動作しない

**向いている場面:** 管理画面・ダッシュボード・ログイン後のアプリ（SEO不要・インタラクション重視）

## SSR（Server-Side Rendering）

リクエストのたびにサーバーがHTMLを生成して返す。

```
1. ユーザー → リクエスト
2. サーバー → データ取得 + HTML生成
3. ブラウザ → 受け取ったHTMLを即表示
4. ブラウザ → JSをハイドレーションしてインタラクティブに
```

**メリット:**
- 初期表示が速い（HTMLが完成した状態で届く）
- SEOに強い
- 常に最新データを表示できる

**デメリット:**
- リクエストのたびにサーバー処理が走るためサーバー負荷が高い
- TTFB（最初のバイトが届くまでの時間）がデータ取得に依存する

**向いている場面:** ECサイト・ニュースサイト・ユーザーごとにコンテンツが変わるページ

## SSG（Static Site Generation）

ビルド時に全ページのHTMLを生成して静的ファイルとして配信する。

```
ビルド時:
  データ取得 + HTML生成 → 静的ファイル群

リクエスト時:
  CDN → 生成済みHTMLをそのまま返す（サーバー処理なし）
```

**メリット:**
- 最速の表示速度（CDNから静的ファイルを返すだけ）
- サーバーレスで運用できる
- セキュリティリスクが低い

**デメリット:**
- コンテンツが更新されるたびに再ビルドが必要
- ページ数が多いとビルド時間が長くなる
- リアルタイム性のあるコンテンツには不向き

**向いている場面:** ドキュメントサイト・ブログ・更新頻度が低いLP

## ISR（Incremental Static Regeneration）

SSGの派生。ページを静的生成しつつ、一定時間ごとまたはオンデマンドでバックグラウンド再生成する。

```
初回リクエスト → 生成済みHTMLを返す
（バックグラウンドで再生成）
次のリクエスト → 新しいHTMLを返す
```

```ts
// Next.js の例
export async function generateStaticParams() { ... }

export const revalidate = 60 // 60秒ごとに再生成
```

**メリット:**
- SSGの速度を保ちつつコンテンツを更新できる
- ビルド時間がページ数に比例しない

**デメリット:**
- 再生成のタイミングによって古いコンテンツが一時的に表示される
- Next.jsなど特定のフレームワークへの依存がある

**向いている場面:** ECサイトの商品ページ・更新頻度が中程度のコンテンツ

## 比較

| 戦略 | HTMLの生成タイミング | SEO | 初期表示 | リアルタイム性 |
|------|-------------------|-----|---------|-------------|
| CSR | ブラウザ（実行時） | △ | 遅い | ◎ |
| SSR | サーバー（リクエスト時） | ◎ | 速い | ◎ |
| SSG | ビルド時 | ◎ | 最速 | △ |
| ISR | ビルド時 + 定期再生成 | ◎ | 速い | ○ |

## ハイブリッド戦略

Next.js・Nuxt・SvelteKitなどのモダンフレームワークは、ページ単位で戦略を混在させられる。

```
/ （トップページ）       → SSG
/blog/[slug]           → ISR（1時間ごとに再生成）
/dashboard             → CSR（認証後のため）
/products/[id]         → SSR（在庫・価格をリアルタイム表示）
```

## いつ使うか

- **CSR**: 管理画面・SaaS・ログイン後の画面（SEO不要）
- **SSR**: ユーザーごとにコンテンツが変わる・リアルタイム性が必要
- **SSG**: ドキュメント・ブログ・更新頻度が低いページ
- **ISR**: SSGとSSRの中間。数分〜数時間の更新頻度で許容できるページ
