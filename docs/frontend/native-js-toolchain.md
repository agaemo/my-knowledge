# ネイティブ速度 JS ツールチェーン

## なぜ生まれたか

Babel・webpack・ESLint は JavaScript で書かれており、プロジェクト規模が大きくなるにつれビルド時間が数十秒〜数分に膨らんだ。V8 の JIT 最適化では補えない根本的なシングルスレッド制限と、AST 変換の多段パイプラインがボトルネックになっていた。

2020 年代に入り、Rust・Go で書かれたツールが登場し、同じ処理を 10〜100 倍速で実現した。これが「ネイティブ速度 JS ツールチェーン」の潮流である。

## 主要ツール

### esbuild（Go）

Go 製のバンドラー・トランスパイラー。ネイティブ速度ツールの先駆け。

- **用途**: バンドル・TS トランスパイル
- **採用文脈**: Vite の内部エンジン、ライブラリのプロダクションビルド
- **制約**: Tree-shaking は webpack ほど高度でなく、コード分割が限定的

### SWC（Rust）

Rust 製の JS/TS コンパイラ。Babel の後継として設計された。

- **用途**: トランスパイル・最小化
- **採用文脈**: Next.js 12+ のデフォルトコンパイラ、Parcel のコアエンジン
- **Babel との違い**: プラグインエコシステムは Babel に劣るが速度は大幅に上回る

### Turbopack（Rust）

Vercel が開発する Rust 製バンドラー。webpack の後継として Next.js に統合される方向で開発中。

- **用途**: バンドル・HMR（Hot Module Replacement）
- **採用文脈**: Next.js 13+ の開発サーバー（`--turbo` フラグ）
- **現状**: 2024 時点でプロダクションビルドは段階的導入中

### Biome（Rust）

Rust 製のフォーマッター＋リンター。Prettier と ESLint を単一ツールで代替する。

- **用途**: コードフォーマット・静的解析
- **採用文脈**: 新規プロジェクトでのリントルール整備、Prettier + ESLint 構成の簡素化
- **制約**: ESLint のプラグインエコシステムは未カバー。既存 ESLint ルールの移行コストがある

### Oxc（Rust）

Rust 製の JS/TS ツールチェーンスイート。パーサー・リンター・トランスフォーマー・ミニファイアーを含む。

- **用途**: パース・リント・変換（個別コンポーネントとして利用可）
- **採用文脈**: Rolldown（Vite の次世代バンドラー）の内部エンジンとして採用
- **位置づけ**: ツールを作るための基盤ライブラリとしての性質が強く、エンドユーザーは Biome や Vite 経由で間接的に恩恵を受ける

## ツール選択の整理

| 目的 | 従来 | ネイティブ代替 |
|------|------|--------------|
| トランスパイル | Babel | SWC |
| バンドル | webpack | Turbopack / esbuild / Rolldown |
| フォーマット | Prettier | Biome |
| リント | ESLint | Biome（部分代替） |

## いつ切り替えるか

- **新規プロジェクト**: フレームワークのデフォルト（Next.js → SWC、Vite → esbuild）をそのまま使う
- **Biome 導入**: Prettier + ESLint の設定が煩雑になったタイミング。ただし ESLint プラグインに依存するルールがある場合は並行運用が必要
- **Babel からの移行**: カスタムプラグインに依存していなければ SWC への移行は低コスト。Babel プラグインに依存する場合は移行先の対応状況を確認する

## 注意点

ネイティブツールは速いが、JS 製ツールより拡張性が低い傾向がある。Babel のプラグインエコシステムや ESLint の豊富なルールに依存している場合、速度を取るためにカスタマイズ性を犠牲にするトレードオフが生じる。
