# JavaScript ランタイム

JavaScript がブラウザの外（サーバー・エッジ）でも動く仕組みと、その上に成り立つ実行環境。

## なぜ JavaScript はサーバーで動くか

JavaScript はもともとブラウザのための言語だった。ブラウザ内部には JS を実行する**エンジン**が組み込まれており、Google Chrome には **V8** が使われている。

2009年、Ryan Dahl が V8 をブラウザから取り出し、ファイル I/O やネットワーク機能を加えて独立したプログラムとして動かせるようにした。これが **Node.js** の誕生。

```
ブラウザ: [HTML/CSS エンジン] + [V8（JS エンジン）] + [Web API]
                                      ↓ V8 だけ取り出す
Node.js:                         [V8（JS エンジン）] + [ファイル・ネットワーク API]
```

## V8 とは

Google が開発した JavaScript エンジン。JS のコードを機械語にコンパイルして実行する。

**JIT（Just-In-Time）コンパイル** が特徴。実行しながらホットなコードを最適化することで、インタープリタより高速に動作する。

V8 は Node.js・Deno・Bun・Cloudflare Workers などのランタイムの基盤として使われている（Bun は JavaScriptCore を使用）。

## イベントループ

Node.js が「シングルスレッドなのに多数の接続を捌ける」理由がイベントループ。

### 通常のサーバー（スレッドモデル）

```
リクエスト1 → スレッド1 が DB 待ち（ブロック）
リクエスト2 → スレッド2 が DB 待ち（ブロック）
リクエスト3 → スレッド3 が DB 待ち（ブロック）
→ 接続数だけスレッドが必要。スレッドはメモリを消費する
```

### Node.js（イベントループモデル）

```
リクエスト1 → DB に問い合わせ開始 → 待たずに次へ
リクエスト2 → DB に問い合わせ開始 → 待たずに次へ
リクエスト3 → DB に問い合わせ開始 → 待たずに次へ
      ↓
DB から応答 → コールバック/Promise で処理
```

1本のスレッドで多数の I/O を捌く。I/O を待っている間に別の処理を進める。

### イベントループの構造

```
コールスタック（同期処理）
      ↓ 空になったら
マイクロタスクキュー（Promise の .then・queueMicrotask）← 優先度高
      ↓ 空になったら
タスクキュー（setTimeout・setInterval・I/O コールバック）
      ↓ 繰り返す
```

```javascript
console.log('1');                           // 同期 → すぐ実行

setTimeout(() => console.log('2'), 0);      // タスクキューへ

Promise.resolve().then(() => console.log('3')); // マイクロタスクキューへ

console.log('4');                           // 同期 → すぐ実行

// 出力順: 1 → 4 → 3 → 2
// 理由: 同期 → マイクロタスク（Promise）→ タスク（setTimeout）の順
```

## Node.js の特性と向き・不向き

**向いている:**
- I/O が多い処理（API サーバー・チャット・ストリーミング）
- 多数の同時接続を低コストで捌く

**向いていない:**
- CPU を長時間使う処理（動画エンコード・機械学習・重い計算）
- メインスレッドを長時間ブロックすると他の接続がすべて止まる

## 主要なランタイムの比較

| ランタイム | ベース | 特徴 |
|---|---|---|
| **Node.js** | V8 | 最も実績がある。npm エコシステムが豊富 |
| **Deno** | V8 | Ryan Dahl が Node.js の反省から作った。TypeScript をネイティブサポート・セキュリティモデルが強固 |
| **Bun** | JavaScriptCore | 高速起動・高いスループット。Node.js 互換を重視。テスト・バンドラーも内蔵 |

## エッジランタイム

Cloudflare Workers・Vercel Edge Functions などは V8 を使いながら、Node.js の全 API ではなく限定的なサブセット（Web API 準拠）で動く。

```
通常の Node.js:    世界1〜数か所のサーバーで実行
エッジランタイム:  世界200か所以上の PoP で実行 → ユーザーに近い場所で処理
```

ただし使えるモジュールが制限される（ファイルシステムなし・Node.js 固有の API が使えないことがある）。
