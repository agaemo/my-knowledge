# i18n（国際化）

アプリケーションを複数の言語・地域に対応させるための設計・実装アプローチ。**i18n** は "internationalization"（i と n の間に18文字）の略。実際の翻訳作業（l10n: localization）とは区別される。

## なぜ必要か

UIに文字列をハードコードすると、言語を追加するたびにコードを変更しなければならない。i18nはテキストをコードから分離し、翻訳データとして外部管理することで、コードを変更せずに言語を追加できる構造を作る。

## 主要な概念

### 翻訳キーとメッセージカタログ

UIテキストを文字列ではなくキーで参照し、言語ごとのカタログファイルにマッピングする。

```ts
// コード上はキーで参照
t('button.submit')

// en.json
{ "button": { "submit": "Submit" } }

// ja.json
{ "button": { "submit": "送信" } }
```

### 動的な値の埋め込み（補間）

翻訳文字列の中に変数を埋め込む。文字列を分割して結合しないことが重要（言語によって語順が変わるため）。

```ts
// 悪い例（語順が固定される）
t('welcome') + username + t('suffix')

// 良い例（言語側で語順をコントロールできる）
t('welcome', { name: username })
// en: "Welcome, {name}!"
// ja: "{name}さん、ようこそ！"
```

### 複数形（Pluralization）

英語は単数・複数の2形式だが、言語によって形式数が異なる（アラビア語は6形式）。ライブラリが言語ごとのルールを内包している。

```ts
t('item_count', { count: n })
// count=1 → "1 item"
// count=3 → "3 items"
```

### 日付・数値・通貨のフォーマット

翻訳だけでなく、表記形式もロケールによって異なる。ブラウザ標準の `Intl` API を使うことで対応できる。

```ts
// 日付
new Intl.DateTimeFormat('ja-JP').format(date) // "2026/4/29"
new Intl.DateTimeFormat('en-US').format(date) // "4/29/2026"

// 通貨
new Intl.NumberFormat('ja-JP', { style: 'currency', currency: 'JPY' }).format(1500) // "￥1,500"
new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' }).format(15.0) // "$15.00"
```

## 翻訳ファイルの管理

### ファイル形式

- **JSON**: フロントエンドで最も一般的。ネスト構造で階層整理できる
- **YAML**: 可読性が高く、バックエンドや設定ファイルで多い
- **PO/POT**: GNU gettextの形式。翻訳ツールとの相性が良い

### ディレクトリ構成の例

```
locales/
├── en/
│   ├── common.json     # 共通テキスト（ボタン・エラーなど）
│   └── checkout.json   # 機能単位
└── ja/
    ├── common.json
    └── checkout.json
```

機能が増えてすべてを1ファイルに入れると管理が困難になるため、名前空間（namespace）で分割するのが一般的。

## 言語検出と切り替え

1. `Accept-Language` ヘッダー（ブラウザの設定言語）
2. URLパスやサブドメイン（`/en/`・`en.example.com`）
3. ユーザーが明示的に選択した設定

URLに言語を含める方式はSEOに有利で、シェアしたURLが受け取り手の言語で表示されるという利点がある。

## いつ使うか

- 複数言語対応が要件にある、または将来的に対応する可能性があるアプリ
- 翻訳者（非エンジニア）がテキストを管理する場合
- 日付・通貨・単位など地域ごとのフォーマット差異がある場合

単一言語で完結し今後も拡張予定がないなら導入コストに見合わないことが多い。
