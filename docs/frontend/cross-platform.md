# クロスプラットフォーム

「1つのコードベースで複数の OS・環境向けアプリを動かす」という考え方。ソフトウェア開発の歴史の中で繰り返し求められてきたテーマ。

## なぜ繰り返し求められるか

OS やデバイスが増えるたびに「同じものをまた作る」コストが問題になる。Windows・macOS・Linux・iOS・Android それぞれにネイティブ実装を持つのは、多くのチームにとって現実的でない。

```
プラットフォームの分断:
  デスクトップ: Windows / macOS / Linux
  モバイル: iOS / Android
  Web: ブラウザ（Chrome・Safari・Firefox）
  組み込み: 各種 OS
```

クロスプラットフォームの試みは常にトレードオフを伴う。「1つで済む」便利さの代わりに、各プラットフォームのネイティブな体験・パフォーマンス・最新機能の追随で妥協が生まれる。

## 歴史的な流れ

### Java の "Write Once, Run Anywhere"（1995〜）

Sun Microsystems が Java で掲げたスローガン。JVM（Java Virtual Machine）を各 OS に用意することで、同じバイトコードをどこでも動かす。エンタープライズのサーバーサイドでは成功したが、デスクトップアプリは「重い・遅い・見た目が独特」として普及しなかった。

実際には「Write Once, Debug Everywhere（一度書いて、どこでもデバッグする）」と揶揄されるほど、プラットフォームごとの差異は残った。

### Web をプラットフォームに（2000 年代〜）

ブラウザが最も普及した「共通ランタイム」になった。HTML・CSS・JS で書けばどの OS のブラウザでも動く。クロスプラットフォーム問題の一つの答えが「Web アプリ化」だった。

### Electron — Web 技術でデスクトップへ（2013〜）

GitHub が開発した **Electron** は、Chromium（ブラウザエンジン）と Node.js をパッケージして、Web 技術（HTML・CSS・JS）でデスクトップアプリを作れるようにした。

```
採用しているアプリ:
  VS Code・Slack・Discord・Figma（初期）・Notion・Twitch
```

Web エンジニアがデスクトップアプリを作れるようになり、多くのツールが Electron で作られた。一方で「Chromium を丸ごと同梱するためアプリが重い」という批判も多い（VS Code が 200MB 超になるのはこのため）。

### Tauri — より軽量な代替（2021〜）

Electron の反省を踏まえて登場した **Tauri** は、Chromium を同梱せずに OS 標準の WebView を使う。バックエンドは Rust で書く。バイナリサイズが Electron の 10 分の 1 以下になることもある。

| | Electron | Tauri |
|---|---|---|
| バックエンド | Node.js | Rust |
| レンダリング | 同梱 Chromium | OS 標準 WebView |
| バイナリサイズ | 大きい（100MB+） | 小さい（数 MB〜） |
| パフォーマンス | 重め | 軽量 |

### モバイルのクロスプラットフォーム

Flutter・React Native などモバイル向けの詳細は [アプリプラットフォーム](/frontend/app-platform) を参照。

## アプローチの分類

クロスプラットフォームの実現方法はいくつかのアプローチに分類できる。

**WebView ラップ型：** Web アプリを WebView（組み込みブラウザ）でラップしてアプリに見せる。Capacitor・Ionic・Cordova など。実装が楽な反省面パフォーマンスはネイティブに劣る。

**ネイティブブリッジ型：** JS や Dart でコードを書き、実行時にネイティブコンポーネントに変換する。React Native が代表。OS の UI パーツを使うためネイティブに近い見た目になる。

**独自レンダリング型：** OS の UI コンポーネントを使わず、独自のレンダリングエンジンで全プラットフォームで同じ見た目を出す。Flutter が代表。

**共通ランタイム型：** JVM・CLR（.NET）・Wasm など、共通の仮想マシンを各 OS に用意して同じバイナリを動かす。

## WebAssembly による次世代のクロスプラットフォーム

[WebAssembly](/frontend/webassembly) と WASI の登場で、「ブラウザ・Edge・サーバーで同じバイナリを動かす」新しい形のクロスプラットフォームが生まれつつある。コンテナ（Docker）に代わるポータビリティの仕組みとして注目されている。

## トレードオフの整理

```
ネイティブ:
  ✓ 最高のパフォーマンスと UX
  ✓ プラットフォームの最新機能に即座に対応
  ✗ 各プラットフォーム分の開発コスト

クロスプラットフォーム:
  ✓ 1つのコードベースで複数環境をカバー
  ✓ チームの技術スタックを統一できる
  ✗ プラットフォーム固有の最適化が難しい
  ✗ OS の新機能追随が遅れる
```

「クロスプラットフォームで十分か」はプロダクトの性質と対象ユーザーによる。消費者向けの高品質アプリと、社内ツールや B2B SaaS では求められる水準が異なる。
