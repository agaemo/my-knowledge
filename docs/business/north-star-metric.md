# North Star Metric（北極星指標）

会社・プロダクトが生み出す「コアバリュー」を最も端的に表す単一の指標。航海士が北極星を道しるべにするように、組織全体が向かう方向を示す。

Sean Ellis（Growth Hacking の提唱者）が広めた概念。

## なぜ単一の指標か

指標が複数あると、組織の注意が分散する。

```
「DAU を上げたい」「収益を上げたい」「継続率を上げたい」
→ チームごとに別の方向に動く
→ 施策の優先順位を統一できない

North Star Metric が1つ決まっていると:
→ 「この施策はNSMを動かすか？」が判断基準になる
→ 全チームが同じ方向を向ける
```

## 収益を North Star にしない理由

収益は「結果」であって「原因」ではない。顧客がプロダクトから価値を得たから、収益が生まれる。

North Star Metric は**顧客が価値を体験している状態**を表す指標を選ぶ。それが持続的に増えれば、収益はついてくる。

```
収益（Revenue）← 価値提供 ← North Star Metric
                              ↑ここを測る
```

## 有名な例

| 企業 | North Star Metric | 意味 |
|---|---|---|
| Airbnb | 予約泊数（Nights Booked） | ゲストとホスト双方が価値を体験した数 |
| Spotify | 月間リスニング時間 | 音楽を楽しんでいる量 |
| Slack | 週あたり送信メッセージ数 | チームが活発にコミュニケーションしている量 |
| Facebook | DAU（日次アクティブユーザー） | 毎日価値を感じている人数 |
| HubSpot | 週次アクティブチーム数 | 実際に使っているチームの数 |
| Notion | ドキュメント作成数 | 知識が蓄積されている量 |

## 良い North Star Metric の条件

1. **顧客価値を表す** — 顧客が「使ってよかった」と感じる瞬間と連動している
2. **成長を反映する** — 増えれば会社が成長していると分かる
3. **測定可能** — 毎日/毎週数値として把握できる
4. **行動に落とせる** — チームが「この施策でNSMが動くか？」を考えられる

**良くない例：** 売上・登録ユーザー数（価値体験を表していない）、NPS（動きが遅く日常の意思決定に使いにくい）

## Input Metric との組み合わせ

North Star Metric（遅行指標）だけでは「何をすれば動くか」が分かりにくい。そのため、NSM を動かす**Input Metric**（先行指標）をセットで定義する。

```
North Star Metric: 予約泊数（Airbnb）
  Input Metric:
    - 検索から閲覧に進んだ率
    - 閲覧から問い合わせに進んだ率
    - ホストの応答速度
    - 新規ホスト登録数
```

Input Metric を改善する施策が NSM を動かす、という因果関係を仮説として持つ。

## エンジニアとの接点

North Star Metric が定義されている組織では、機能の優先順位判断が「この機能は NSM を動かすか？」で統一される。逆に NSM が定義されていないと、各チームが別々の指標を追いかけて整合が取れなくなる。
