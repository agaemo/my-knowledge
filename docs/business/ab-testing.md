# A/Bテスト

2つ以上のバリアント（A案・B案）を実際のユーザーに同時に提示し、どちらがより目標指標を達成するかをデータで検証する手法。スプリットテストとも呼ぶ。

## なぜ存在するか

プロダクトの意思決定は「どちらが良いか」という問いに溢れている。

```
「ボタンを緑にするか青にするか」
「テキストを A にするか B にするか」
「フローを1ステップ減らすべきか」
```

経験・直感・デザイナーの好みで決めると、実際のユーザー行動とズレることが多い。A/Bテストは**実際のユーザー行動というデータで決める**ための仕組み。

## 仕組み

```
ユーザーをランダムに2グループに分割
  Group A（コントロール）: 現在の UI・機能
  Group B（バリアント）  : 変更を加えた UI・機能

一定期間、両グループの指標を計測
  → 統計的に有意な差が出たら、良い方を採用
```

**ランダム分割が重要：** 同じユーザーが A と B の両方を見ると、結果が汚染される。ユーザーごとに割り当てを固定する。

## 統計的有意性

数十件のデータで「B の方がコンバージョン率が高い」と判断するのは危険。サンプル数が少ないと偶然の差を見誤る。

```
有意水準（p値）: 一般的に p < 0.05（5%未満の確率で偶然起きた差）を有意とする
検出力: 真の差があるときにそれを検出できる確率（通常 80% 以上を目標）
サンプルサイズ: 検出したい差の大きさと検出力から必要なサンプル数を計算する
```

実務では「何週間テストすれば十分か」を事前にサンプルサイズ計算ツールで見積もる。

## Feature Flags との関係

A/Bテストの実装には [フィーチャーフラグ](/deployment/feature-flags) を使うことが多い。

```
Feature Flag でユーザーを振り分け:
  user_id % 2 == 0 → variant_a
  user_id % 2 == 1 → variant_b

計測ツール（Amplitude・Mixpanel・自前）でコンバージョンを追跡
```

フィーチャーフラグがあれば、コードを変更せずにテスト開始・終了・ロールバックができる。

## 何をテストするか

**向いているもの:**
- UI・コピー・ボタンの色・配置
- オンボーディングフローのステップ数
- 価格ページのレイアウト
- メールの件名・送信タイミング

**向いていないもの:**
- 効果が出るまでに時間がかかる変更（長期的な学習機能など）
- サンプル数が少ないユーザー層（B2B エンタープライズなど）
- 倫理的に片方のグループを不利にする変更

## よくある失敗

**早期終了：** 「もうBが勝っている！」とデータが揃う前に終わらせる。偶然の変動を真の差と勘違いする。

**複数指標の多重検定：** 10個の指標を同時に見ていれば、偶然どれかが有意になる確率が上がる。事前に「主要指標」を1つ決めておく。

**ノベルティ効果：** 新しいUIへの一時的な興味でBが勝つが、数週間後には差がなくなる。十分な期間テストする。

**セグメントを分けすぎる：** 「スマホユーザーだけ」「新規ユーザーだけ」と分析していると、どこかで偶然有意な結果が出る。

## いつ使うか

- 変更の効果を定量的に示したいとき
- 複数の案から一つを選ぶ根拠が欲しいとき
- トラフィックが十分にある（最低でも週数百件以上のコンバージョン）とき

トラフィックが少ない段階では A/B テストより定性的なユーザーインタビューの方が学びが多い。
