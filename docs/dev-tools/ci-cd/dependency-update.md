# 依存パッケージの自動更新

ライブラリ・フレームワークの更新を人間が手動で追跡するのではなく、ボットが定期的にチェックして PR を自動作成する仕組み。

## なぜ存在するか

依存パッケージの更新には2つの問題がある。

**放置リスク：** 更新を手動で確認する運用は後回しにされやすく、気づけば数十バージョン分の差が生じる。差が大きいほどマイグレーションコストが増し、さらに後回しにされるという悪循環に陥る。

**セキュリティリスク：** 脆弱性の修正はパッチバージョンで配布されることが多い。更新が遅れるほど既知の脆弱性を抱えたまま運用する期間が伸びる。

自動更新ボットはこの問題を「更新の検知と PR 作成」を自動化することで解決する。マージの判断は人間が行うため、意図しない変更が混入するリスクを保ちながら更新コストを下げる。

## 仕組み

```
定期実行（週次など）
  ↓
依存ファイル（package.json / go.mod など）を解析
  ↓
レジストリの最新バージョンと比較
  ↓
更新があれば PR を自動作成
  ↓
CI でテストが通ればマージ（手動 or 自動）
```

## ツール比較

| | Dependabot | Renovate |
|---|---|---|
| 提供元 | GitHub 標準機能 | オープンソース（Mend） |
| 設定コスト | 低い（YAML 数行） | 高い（柔軟な分、設定項目が多い） |
| PR のグルーピング | 1パッケージ1PR（基本） | 関連パッケージをまとめられる |
| 対応エコシステム | npm / pip / Go / Docker など主要なもの | より広範囲 |
| 料金 | 無料 | 無料（self-host も可能） |

依存が少ないプロジェクトや GitHub 中心の運用であれば Dependabot で十分。Docusaurus 系のように複数パッケージを同時に上げる必要がある場合は Renovate のグルーピングが有効。

## Dependabot の設定例

`.github/dependabot.yml` を置くだけで有効になる。

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"   # daily / weekly / monthly
```

## いつ使うか

- 継続的にメンテナンスするプロジェクトすべてに導入を検討する
- セキュリティパッチを素早く取り込みたい場合は `interval: "daily"` にする
- モノレポで複数の `package.json` がある場合は `directory` を複数定義する
