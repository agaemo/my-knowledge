# プラットフォームエンジニアリング / IDP

開発チームが自律的・効率的に開発・デプロイできるよう、共通の内部プラットフォームを構築・提供する実践。  
IDP = Internal Developer Platform（内部開発者プラットフォーム）。

## なぜ存在するか

マイクロサービス・クラウドネイティブへの移行で、開発者が自分で管理すべきことが増えた。

```
各チームが個別に対応することになったもの:
  - CI/CD パイプラインの構築
  - Kubernetes のデプロイ設定
  - 監視・ロギングの設定
  - シークレット管理
  - 環境のプロビジョニング
  → チームごとに車輪を再発明 → 認知負荷が高い
```

プラットフォームエンジニアリングは、これらの共通インフラを「セルフサービスで使えるプラットフォーム」として提供し、**開発チームがプロダクト開発に集中できるようにする**。

## Team Topologies との関係

Team Topologies の **Platform team** がプラットフォームエンジニアリングを担うチーム。

```
Stream-aligned team（プロダクト開発チーム）
  ↓ プラットフォームを利用
Platform team（プラットフォームエンジニアリングチーム）
  ↓ プラットフォームを構築・運用
```

Platform team は他のチームを「顧客」として扱う。API・ドキュメント・サポートを提供し、開発チームの認知負荷を下げることがミッション。

## IDP の構成要素

IDP は複数の機能をセルフサービスで提供するポータル・ツール群。

| 機能 | 内容 |
|---|---|
| **サービスカタログ** | 社内のサービス・API・ライブラリの一覧と所有者 |
| **セルフサービスデプロイ** | 開発者が UI・CLI から環境・デプロイを自分で操作できる |
| **ゴールデンパス** | 推奨する技術スタック・テンプレートを用意し、すぐに始められる |
| **スコアカード** | サービスの成熟度・セキュリティ・ドキュメントの充実度を可視化 |
| **開発者ポータル** | 上記を統合したダッシュボード |

### ゴールデンパス（Golden Path / Paved Road）

「これを使えば正しく・速く始められる」という推奨の道。  
強制はしないが、推奨ルートを整備することで多くのチームが自然に標準的な方法を選ぶ。

```
例:
  新しいマイクロサービスを作るとき
  → テンプレートリポジトリ（CI/CD・監視・ログ設定済み）をボタン1つで生成できる
  → チームは独自設定に悩む時間ゼロでプロダクトコードから書ける
```

## 代表的なツール

**Backstage**（Spotify）  
Spotify が開発したオープンソースの開発者ポータル。サービスカタログ・ドキュメント・テンプレートを統合する。多くの企業が自社 IDP のベースとして採用。

**Port / Cortex / OpsLevel**  
SaaS 型の IDP ソリューション。Backstage より導入コストが低い。

## DevOps との違い

DevOps は「開発と運用の壁をなくす文化・プラクティス」。  
プラットフォームエンジニアリングは「その文化を実現するためのプラットフォームを作る専門チーム」。

```
DevOps の理想:「開発者が自分でデプロイ・監視・運用できる」
プラットフォームエンジニアリングの役割:「それができるための道具・環境を整える」
```

## いつ必要になるか

- チームが増え、インフラ・CI/CD の設定を各チームが個別に悩んでいるとき
- 「本番デプロイの手順書が各チームでバラバラ」「セキュリティ設定が統一されていない」が問題になってきたとき
- プラットフォームエンジニアリング専門チームを立てられる規模（目安: エンジニア 50名以上）になったとき

小規模では過剰。共通の痛みが可視化されてから投資するのが現実的。
