# Ansible

## 何か

Red Hat が開発するオープンソースの構成管理・自動化ツール。サーバーへのソフトウェアインストール・設定変更・ファイル配布などを YAML で記述し、複数台のサーバーに一括適用する。

→ [公式サイト](https://www.ansible.com/) / [ドキュメント](https://docs.ansible.com/)

**エージェントレス**が大きな特徴で、管理対象のサーバーに専用ソフトをインストールする必要がない。SSH 経由で接続して操作する。

::: info エージェントレスとは
Chef・Puppet など他の構成管理ツールは、管理したいサーバー側に「エージェント」と呼ばれる専用ソフトをあらかじめインストールしておく必要がある。Ansible は代わりに SSH を使って操作するため、管理対象のサーバーに何も入れなくてよい。

```
Chef・Puppet：管理サーバー → [エージェント経由] → 対象サーバー（事前インストールが必要）
Ansible    ：管理サーバー → [SSH]              → 対象サーバー（何も入れなくてよい）
```

管理対象を増やすときにエージェントのインストール作業が不要で、エージェント自体の管理・更新コストもかからない。

また、SSH の仕組みをそのまま使うため、**踏み台サーバー（Bastion Host）経由の接続にも対応できる**。対象サーバーに直接アクセスできないネットワーク構成でも、踏み台を経由する SSH 設定（ProxyJump）を渡すことで操作できる。

```
手元の PC → [SSH] → 踏み台サーバー → [SSH] → 対象サーバー
```

踏み台経由の設定はインベントリに `ansible_ssh_common_args` として記述する。

```ini
[web]
192.168.0.11
192.168.0.12

[web:vars]
# -J で踏み台を指定（ProxyJump）
ansible_ssh_common_args='-J user@bastion.example.com'
```

または `ansible.cfg` に全体設定として書くこともできる。

```ini
[defaults]
remote_user = ec2-user

[ssh_connection]
ssh_args = -o ProxyJump=user@bastion.example.com
```

認証は SSH 鍵認証が標準。`--ask-pass` オプションでパスワード認証も使えるが、複数サーバーへの自動化という Ansible の目的と相性が悪いため、実運用では鍵認証が一般的。
:::

## なぜ存在するか

複数台のサーバーを手動で設定すると、ミスや作業漏れで「このサーバーだけ設定が違う」という状態（**構成ドリフト**）が起きやすい。スクリプトで自動化しても、冪等性（何度実行しても同じ結果になる性質）の担保が難しい。

Ansible は冪等な操作を標準で提供し、サーバーの状態をコードとして管理・再現できるようにする。

## 主要概念

**インベントリ（Inventory）**  
管理対象のサーバー一覧。IPアドレスやホスト名を記述し、グループで分類できる（webサーバー群・DBサーバー群など）。

**プレイブック（Playbook）**  
「どのサーバーに」「何をするか」を定義する YAML ファイル。Ansible の中心的な設定ファイル。

**タスク（Task）**  
プレイブックの中の1つの操作単位。「Nginxをインストールする」「設定ファイルをコピーする」「サービスを起動する」など。

**モジュール（Module）**  
タスクで使う操作のライブラリ。パッケージ管理・ファイル操作・サービス制御・クラウドAPI操作など数千種類が標準で用意されている。

**ロール（Role）**  
関連するタスク・変数・ファイルをまとめた再利用単位。「Webサーバーの構築」「監視エージェントの導入」などをロールとして定義し、複数プロジェクトで使い回す。

## サンプル

インベントリでサーバーを定義し、プレイブックで操作を記述する。

**インベントリ（inventory.ini）**

```ini
# webグループに2台のサーバーを登録
[web]
192.168.0.11
192.168.0.12

# dbグループに1台のサーバーを登録
[db]
192.168.0.21
```

**プレイブック（site.yml）**

```yaml
- name: WebサーバーにNginxをセットアップする
  hosts: web          # インベントリの web グループが対象
  become: true        # sudo で実行する

  tasks:
    - name: Nginx をインストールする
      ansible.builtin.package:
        name: nginx
        state: present  # インストールされている状態にする（冪等）

    - name: 設定ファイルを配置する
      ansible.builtin.copy:
        src: nginx.conf
        dest: /etc/nginx/nginx.conf

    - name: Nginx を起動して自動起動を有効にする
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true
```

`state: present` のように「この状態にしてほしい」と宣言する書き方になっている。何度実行しても既にその状態なら何もしない（冪等性）。

## Terraform との違い

よく比較されるが、扱う領域が異なる。

| | Ansible | Terraform |
|---|---|---|
| 主な用途 | サーバーの設定・構成管理 | クラウドインフラのプロビジョニング |
| 操作スタイル | 手続き的（この順番でこれをする） | 宣言的（この状態にしてほしい） |
| 状態管理 | なし（毎回実行して確認） | state ファイルで現在状態を管理 |
| 対象 | OS・ミドルウェア・アプリの設定 | EC2・VPC・RDS などクラウドリソース |
| エージェント | 不要（SSH） | 不要 |

「EC2インスタンスを作る（Terraform）→ そのサーバーにNginxをインストールして設定する（Ansible）」のように組み合わせて使うことも多い。

## Terraform との組み合わせ

Terraform と Ansible は競合するツールではなく、担う層が異なる。

```
Terraform：クラウド上のリソースを「作る」
  ↓ EC2インスタンス・VPC・RDS が存在するようになる
Ansible ：そのサーバーの中を「設定する」
  ↓ Nginx・アプリ・設定ファイルが入り、サービスが起動する
```

Terraform がインフラの「外側」、Ansible がサーバーの「内側」を担うイメージで、組み合わせて使うことがある。

ただし**コンテナ中心の構成では Ansible は不要になることが多い**。アプリ環境を Dockerfile でコンテナに閉じ込め、ECS や Kubernetes で動かす場合、サーバーに直接入って設定する必要がないため Terraform だけで完結する。

## いつ使うか

- オンプレミスや VM 中心の環境でサーバーの設定を自動化したい
- 大量のサーバーに同じミドルウェア・設定を一括適用したい
- コンテナ化されていないレガシー構成を管理したい

コンテナ・クラウドネイティブ構成が前提の場合は、Ansible を使わず Terraform のみで完結する設計が一般的になっている。
