# ウォーターフォール開発

## 何か

要件定義 → 設計 → 実装 → テスト → リリース を順番に進める開発モデル。
前のフェーズが完了してから次に進む。前のフェーズには原則戻らない。

```
要件定義
   ↓（完了後に次へ）
設計
   ↓
実装
   ↓
テスト
   ↓
リリース
```

## なぜ今も使われるか

「アジャイルの方が良い」という風潮があるが、ウォーターフォールが合う場面は存在する。

### 向いている場面

| 場面 | 理由 |
|---|---|
| 規制・法令がある領域（医療・金融・航空） | 承認プロセスがフェーズ単位で必要 |
| 要件が完全に固定されている | 変更コストより計画遵守が重要 |
| 外部ベンダーへの発注 | 契約・見積もりが仕様確定を前提とする |
| ハードウェアと連動する開発 | ハードの仕様変更コストが高い |

### 向いていない場面

- ユーザーニーズが曖昧・変化しやすい
- 市場への早期フィードバックが必要
- チームが小さく素早く動ける

## アジャイルとの比較

| | ウォーターフォール | アジャイル |
|---|---|---|
| 要件変更への対応 | 難しい（コスト大） | 前提として受け入れる |
| 動くものが出るまで | 最後（数ヶ月〜年単位） | 早期（数週間） |
| リスクが顕在化するタイミング | テストフェーズ（遅い） | 各スプリント（早い） |
| 向いている規模 | 大規模・固定要件 | 中小規模・変化する要件 |
| ドキュメント | 重視 | 最小限 |

## ウォーターフォールの本質的な問題

**リスクの顕在化が遅い。**

要件が間違っていても、テストフェーズまで誰も気づかない。
6ヶ月かけて作ったものが「ユーザーが求めていたものと違う」という事態が起きる。

```
要件定義の誤りが発覚するタイミング:

ウォーターフォール → テストフェーズ（開発の80%終了後）
アジャイル        → 最初のスプリント（開発の10%程度）
```

## ハイブリッドアプローチ

完全なウォーターフォールでも完全なアジャイルでもない、中間的なアプローチが現実には多い。

### スクラムバン

スクラムのスプリントサイクル + カンバンのWIP制限を組み合わせる。

### フェーズゲート（Stage-Gate）

大きなフェーズ単位（設計完了・開発完了）でゲートレビューを行い、承認を得てから次へ進む。
規制業界で多い。

### ミニウォーターフォール

2週間スプリント内で「設計→実装→テスト」を完結させる。
アジャイルの形を取りながら、チーム文化はウォーターフォール的。

## 選択基準

```
要件は明確か？
├─ Yes → 変更される可能性は低いか？
│         ├─ Yes → 規制・外部発注がある？
│         │         ├─ Yes → ウォーターフォール or ハイブリッド
│         │         └─ No  → どちらでも可（アジャイル推奨）
│         └─ No  → アジャイル
└─ No  → アジャイル（まず動くものを作って検証する）
```
